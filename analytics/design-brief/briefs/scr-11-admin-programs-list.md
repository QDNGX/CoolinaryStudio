# SCR-11 — Каталог программ

| | |
|---|---|
| **Роль** | Администратор |
| **Платформа** | Веб (адаптив: десктоп + мобильный браузер) |
| **Приоритет** | Must |
| **Реестр** | [screen-registry-cooking.md](../screen-registry-cooking.md) |

## 1. Кто и зачем сюда приходит

Артём — владелец студии, а не оператор CRM. Сюда он попадает в двух настроениях: либо спокойно
«наводит порядок на витрине» (придумал новую программу, хочет обновить описание старой), либо
мимоходом — по пути к составлению расписания на неделю, чтобы убедиться, что нужная программа
уже есть в каталоге. Он не работает с этим списком часами: заглянул, нашёл, ушёл. Каталог для
него — «меню студии», и оно небольшое (единицы, максимум пара десятков программ).

## 2. Работа экрана

Когда я обновляю предложения студии или готовлюсь составлять расписание, я хочу увидеть все
программы одним обозримым списком и войти в нужную (или завести новую), чтобы каталог всегда
отражал то, что студия реально проводит.

## 3. Входы и выходы

- **Приходит из:** навигация раздела администратора (основной путь); возврат после сохранения
  или отмены на форме программы (SCR-12).
- **Уходит на:** форма создания программы — [SCR-12](scr-12-admin-program-form.md) (пустая);
  форма редактирования — тот же [SCR-12](scr-12-admin-program-form.md), открытый с данными
  выбранной программы.

## 4. Что пользователь видит

| Приоритет | Элемент | Что показывает | Заметки для дизайнера |
|---|---|---|---|
| 1 | Список программ | Каждая программа: название (главное), тип кухни, уровень сложности («новичок» / «продвинутый») | Название — точка входа в редактирование. Список короткий: не проектировать под сотни строк, таблица-«простыня» не нужна |
| 2 | Признак «сложное оборудование» | Заметная, но спокойная пометка у программ, где нужно сложное оборудование | Это не декорация: у таких программ группа меньше (8 вместо 12), и Артём должен видеть это ещё из списка, планируя расписание |
| 3 | Кнопка «Создать программу» | Вход в пустую форму SCR-12 | Одна, заметная, стабильное место; доступна и при пустом каталоге |
| 4 | Фото программы (превью) | Первое фото, если загружено | Помогает узнавать программы «на глаз»; при отсутствии фото — аккуратный плейсхолдер, не «сломанная картинка» |
| 5 | Счётчик/заголовок раздела | Сколько программ в каталоге, где я нахожусь | Служебное, не перетягивать внимание |

## 5. Что пользователь делает

- **Основное действие (одно):** создать программу — переход в пустую форму SCR-12.
- **Вторичные действия:** открыть существующую программу на редактирование (переход в SCR-12
  с данными).
- **Что здесь сделать нельзя:** удалить или заархивировать программу — таких операций в системе
  нет: в MVP список только растёт, это осознанное ограничение (решение Р-15, см.
  [журнал решений](../design-decisions-cooking.md)), и не нужно рисовать кнопку-обещание;
  редактировать поля программы прямо в списке — правки только через форму, чтобы важное правило
  про оборудование и лимит группы всегда показывалось в одном месте (см. SCR-12).

## 6. Состояния экрана

| Состояние | Когда возникает | Что видит пользователь |
|---|---|---|
| По умолчанию | В каталоге есть программы | Список программ + кнопка создания |
| Загрузка | Данные ещё едут | Скелетон/плейсхолдер списка; кнопка создания может появиться сразу |
| Пусто | Каталог ещё не наполнялся (старт студии в системе) | Дружелюбный empty state: объяснение, что программы — основа расписания, и та же кнопка «Создать программу» как единственный акцент |
| Ошибка | Список не загрузился (сеть/сервер) | Спокойное сообщение «не получилось загрузить каталог» + повторить попытку; без технических кодов |
| Программа со сложным оборудованием | У программы включён признак оборудования | Пометка у строки/карточки; желательно с намёком на следствие («группа до 8») — коротко, без лекции |

## 7. Правила, которые нельзя нарушать

- **ФТ-01** — программу можно создать и отредактировать только администратору; этот экран —
  единственная точка входа в обе операции.
- **ФТ-02** — признак «сложное оборудование» снижает лимит группы с 12 до 8. В списке это
  проявляется как обязательная видимость признака: Артём принимает решения о расписании,
  глядя на каталог, и не должен вспоминать лимит по памяти.
- **UC-07** — ведение каталога — часть единого административного цикла «каталог → расписание»;
  переходы не должны терять контекст (после сохранения программы Артём возвращается сюда).

## 8. Тон и настроение

«Полка с меню студии», а не реестр записей: тепло, наглядно, с фото и человеческими названиями.
Избегать ощущения энтерпрайз-админки — плотных таблиц, серых гридов, массовых операций.
Экран должен подсказывать: «всё под контролем, добавить новую программу — минутное дело».

## 9. Свобода дизайнера и открытые вопросы

- **Решает дизайнер:** карточки или строки; порядок сортировки по умолчанию; как именно выглядит
  пометка сложного оборудования (иконка, бейдж, текст); поведение на мобильном; микрокопирайт
  empty state и ошибок; нужны ли поиск/фильтр по каталогу — требованиями не зафиксированы, и при
  ожидаемом размере каталога (единицы–десятки программ), вероятно, избыточны.
- **Открытые вопросы к команде:** открытых вопросов нет — все закрыты, см.
  [журнал решений](../design-decisions-cooking.md). Вопрос об архивации/удалении программ закрыт:
  в MVP таких операций нет, список только растёт (решение Р-15) — см. раздел 5.
