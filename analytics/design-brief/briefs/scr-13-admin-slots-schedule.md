# SCR-13 — Расписание слотов (админ)

| | |
|---|---|
| **Роль** | Администратор |
| **Платформа** | Веб (адаптив: десктоп + мобильный браузер) |
| **Приоритет** | Must |
| **Реестр** | [screen-registry-cooking.md](../screen-registry-cooking.md) |

## 1. Кто и зачем сюда приходит

Это рабочий стол Артёма — экран, куда он заходит чаще всего. Реальный ритм студии: расписание
собирается «на неделю вперёд», поэтому типовой визит — сесть раз в несколько дней и добить сетку
ближайшей недели: где какая программа, какой шеф, во сколько. Второй сценарий — тревожный:
что-то случилось (шеф заболел, форс-мажор), и нужно быстро найти конкретный класс и отменить его.
Третий — просто глянуть, как заполняются классы: где аншлаг, где пусто. Артём здесь хозяин
своего календаря, а не диспетчер за пультом.

## 2. Работа экрана

Когда я планирую неделю студии или реагирую на изменение обстоятельств, я хочу видеть все классы
сеткой по датам — с программой, шефом, заполненностью и статусом — и отсюда же создавать новые
слоты или отменять существующие, чтобы расписание студии всегда соответствовало реальности.

## 3. Входы и выходы

- **Приходит из:** навигация раздела администратора (основной путь); возврат после создания
  слота (SCR-14) или после отмены слота (SCR-15).
- **Уходит на:** создание слота — [SCR-14](scr-14-admin-slot-form.md); отмена конкретного слота —
  диалог [SCR-15](scr-15-admin-slot-cancel.md), открытый из этого же расписания; список броней
  завершённого класса — открывается отсюда же, чтобы снять ошибочную отметку неявки (решение
  Р-02, см. [журнал решений](../design-decisions-cooking.md)).

## 4. Что пользователь видит

| Приоритет | Элемент | Что показывает | Заметки для дизайнера |
|---|---|---|---|
| 1 | Сетка слотов по датам | Классы ближайшего горизонта, сгруппированные по дням; на карточке слота: программа, шеф, время начала | Горизонт планирования — неделя вперёд: текущая+следующая неделя должны быть на расстоянии одного взгляда/жеста. Каждый класс длится 3 часа — в день их немного, сетка не бывает плотной |
| 2 | Занятость слота | Брони против вместимости (например, «7 из 12») | Артёму важно с одного взгляда отличать «пустой», «заполняется», «полный». Достойна визуального кода, не только цифр |
| 3 | Статус слота | Запланирован / отменён студией / завершён | Отменённые не прятать: Артём должен видеть, что уже отменено, чтобы не искать. Но и не давать им спорить за внимание с живыми слотами |
| 4 | Кнопка «Создать слот» | Вход в SCR-14 | Главный акцент экрана; желательно уметь создавать «в контексте дня» (см. раздел 9) |
| 5 | Прокат на слоте | Сколько прокатных наборов заложено (и почём) | Вторичная справка при обзоре; на карточке — кратко, без детализации |
| 6 | Навигация по датам | Переход между неделями/периодами, включая прошлые | Служебное: быстро «на следующую неделю» важнее всего, но история доступна вся, без ограничения глубины (решение Р-16 п.4, см. [журнал решений](../design-decisions-cooking.md)) — прошлые периоды нужны в том числе для снятия ошибочной отметки неявки |

## 5. Что пользователь делает

- **Основное действие (одно):** создать слот — переход в форму SCR-14.
- **Вторичные действия:** открыть существующий слот и перейти к его отмене (диалог SCR-15);
  листать недели; смотреть заполненность и детали слота; с завершённого класса — открыть список
  броней и снять ошибочную отметку неявки, поставленную шефом (счётчик нарушений и блокировка
  гостя при этом пересчитываются) — это единственное место в админке для такого исправления
  (решение Р-02, см. [журнал решений](../design-decisions-cooking.md)).
- **Что здесь сделать нельзя:** отменить слот «в один клик» прямо из сетки — отмена требует
  обязательной причины и уведомляет клиентов, поэтому она всегда идёт через отдельный диалог
  (SCR-15), случайное касание не должно ничего рушить; редактировать состав слота после создания
  (перенос времени, замена шефа) — такой операции в системе нет: единственный инструмент
  исправления — отмена и создание заново (решение Р-03, см.
  [журнал решений](../design-decisions-cooking.md)); завершить класс вручную — кнопки «завершить»
  не существует, слот завершается сам по времени (решение Р-10, см. журнал решений); отмечать
  неявку — это делает шеф, админу доступно только снятие ошибочной отметки на завершённом классе.

## 6. Состояния экрана

| Состояние | Когда возникает | Что видит пользователь |
|---|---|---|
| По умолчанию | Есть слоты в отображаемом периоде | Сетка по дням со статусами и заполненностью, кнопка создания |
| Загрузка | Данные периода ещё едут | Скелетон сетки; навигация по датам не скачет |
| Пусто | В периоде нет ни одного слота (типично — следующая неделя ещё не спланирована) | Не «ошибка», а приглашение к работе: «на эту неделю классов пока нет» + кнопка создать слот прямо из пустого состояния |
| Ошибка | Расписание не загрузилось | Спокойное сообщение + «повторить»; без технических деталей |
| Слот отменён студией | В периоде есть отменённые слоты | Карточка остаётся на месте, явно приглушена/маркирована «отменён»; отмена из неё больше недоступна |
| Слот завершён | Класс завершился автоматически по времени (начало + 3 часа) — кнопки «завершить» нет (решение Р-10, см. [журнал решений](../design-decisions-cooking.md)) | Спокойная «архивная» подача; единственное доступное действие — открыть список броней, чтобы при необходимости снять ошибочную отметку неявки (Р-02) |
| Снятие отметки неявки | Админ открыл брони завершённого класса и снимает ошибочную отметку, поставленную шефом | Список гостей класса с отметками; снятие отметки — осознанное действие с подтверждением, после которого счётчик нарушений и блокировка гостя пересчитываются; интерфейс подтверждает результат («отметка снята») |
| Слот полон | Броней столько же, сколько мест | Заполненность читается как «полный» с одного взгляда — для Артёма это радость, а не тревога |
| Смешанный день | В одном дне живые, полные и отменённые классы | Статусы различимы между собой и не сливаются; проверить на дальтонизм — не только цветом |

## 7. Правила, которые нельзя нарушать

- **ФТ-03** — слот создаётся только через форму с полным набором параметров; сетка — точка входа,
  а не место инлайн-создания «наполовину заполненных» слотов.
- **ФТ-04 / UC-03** — отмена слота требует обязательной причины и имеет последствия (уведомления
  клиентам, отмена их броней), поэтому из расписания она доступна только как переход в отдельный
  диалог SCR-15 — никаких быстрых «крестиков» на карточке.
- **ФТ-19** — слот живёт в трёх состояниях (запланирован / отменён студией / завершён); все три
  должны быть представимы в сетке и визуально различимы. В «завершён» слот переходит
  автоматически по времени (начало + 3 часа): у администратора нет кнопки «завершить», и
  интерфейс не должен её изображать (решение Р-10, см.
  [журнал решений](../design-decisions-cooking.md)).
- **ФТ-05** — отменённый слот не возвращается в продажу; интерфейс не должен предлагать для него
  никаких действий, намекающих на «восстановить».
- **Слот не редактируется после создания** (решение Р-03, см.
  [журнал решений](../design-decisions-cooking.md)) — перенос времени или замена шефа делаются
  только через отмену (с письмами гостям) и создание заново; сетка не должна предлагать
  «карандашей» на карточке слота.
- **Снятие ошибочной отметки неявки** (решение Р-02, см.
  [журнал решений](../design-decisions-cooking.md)) — администратор может снять отметку неявки,
  поставленную шефом, из списка броней завершённого класса; счётчик нарушений и блокировка гостя
  пересчитываются автоматически. Это единственное место в админке для этого действия; ставить
  неявку админ не может — только снимать.
- **US-09** — расписание существует ради клиентов: слот, созданный здесь, сразу становится
  доступен для бронирования — сетка должна давать Артёму уверенность, что «опубликовано именно то,
  что я вижу».

## 8. Тон и настроение

Календарь хозяина студии, а не диспетчерская: спокойные дни, живые карточки классов, радость от
заполняющихся групп. Плотность информации — в пользу беглого взгляда: Артём «сканирует» неделю,
а не читает таблицу. Избегать эстетики enterprise-планировщиков — конфигураторов представлений,
многоуровневых фильтров, массовых операций.

## 9. Свобода дизайнера и открытые вопросы

- **Решает дизайнер:** форма сетки (календарная неделя, лента по дням, список); визуальный код
  заполненности и статусов; создание слота «из дня» (клик по дню предзаполняет дату в SCR-14) —
  приветствуется, но необязательно; поведение на мобильном; глубина деталей на карточке слота;
  как подать список броней завершённого класса и действие снятия отметки неявки (панель,
  отдельный экран — лишь бы действие оставалось осознанным, с подтверждением).
- **Открытые вопросы к команде:** открытых вопросов нет — все закрыты, см.
  [журнал решений](../design-decisions-cooking.md): слот завершается автоматически по времени,
  без кнопки «завершить» (Р-10); слот не редактируется — только отмена и создание заново (Р-03);
  архив прошлых периодов показывается весь, без обрезания (Р-16 п.4) — навигация по датам должна
  позволять уйти в прошлое на любую глубину, пусть и без акцента на этом.
