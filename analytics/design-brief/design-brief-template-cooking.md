# Шаблон постановки на дизайн — «Шеф-стол»

> Это **не** шаблон ТЗ. Постановка на дизайн отвечает не на вопрос «что система должна делать»
> (это уже описано в [требованиях](../requirements/requirements-overview-cooking.md)), а на вопрос
> «какую работу этот экран выполняет для человека и в каких состояниях он живёт». Дизайнер по этому
> документу должен суметь спроектировать экран, не читая всю аналитику, — но каждая постановка
> даёт ссылки на первоисточники для проверки.

## Принципы заполнения

- **Одна постановка = один экран** из [реестра экранов](screen-registry-cooking.md).
- Пишем для дизайнера, а не для разработчика: без имён полей БД в тексте разделов 1–6
  (техдетали живут только в разделе 7 «Правила» и в разделе 10 «Связанные требования»).
- Раздел 5: **основное действие всегда одно.** Если хочется два — это два экрана или неверно
  выбран акцент.
- Раздел 6 обязателен: экран без описания пустого и ошибочного состояний считается
  не поставленным.
- Раздел 9 — не формальность: явно перечислить, что дизайнер решает сам. Постановка фиксирует
  «что и почему», дизайнер владеет «как».

---

## Структура постановки

```markdown
# <SCR-XX> — <Название экрана>

| | |
|---|---|
| **Роль** | Клиент / Шеф / Администратор / Гость |
| **Платформа** | Веб (адаптив: десктоп + мобильный браузер) |
| **Приоритет** | Must / Should / Could — по требованиям, которые экран закрывает |
| **Реестр** | [screen-registry-cooking.md](../screen-registry-cooking.md) |

## 1. Кто и зачем сюда приходит
Портрет пользователя в момент попадания на экран: что он только что делал, чего хочет,
в каком настроении (спешит? насторожен? раздражён отменой?). 2–4 предложения.

## 2. Работа экрана
Одна формула: «Когда <ситуация>, я хочу <действие>, чтобы <результат>».
Если формул получается несколько — экран перегружен, вернуться к реестру.

## 3. Входы и выходы
- **Приходит из:** откуда пользователь попадает сюда (экраны, письмо, прямая ссылка).
- **Уходит на:** куда ведёт успех, куда — отказ/отмена.

## 4. Что пользователь видит
Таблица содержимого по убыванию приоритета внимания:

| Приоритет | Элемент | Что показывает | Заметки для дизайнера |
|---|---|---|---|
| 1 | … | … | … |

Приоритет 1 — то, ради чего пользователь пришёл; последним — служебное.

## 5. Что пользователь делает
- **Основное действие (одно):** …
- **Вторичные действия:** …
- **Что здесь сделать нельзя** (и почему это важно показать/спрятать): …

## 6. Состояния экрана
| Состояние | Когда возникает | Что видит пользователь |
|---|---|---|
| По умолчанию | … | … |
| Загрузка | … | … |
| Пусто | … | … |
| Ошибка | … | … |
| <краевые случаи экрана> | … | … |

## 7. Правила, которые нельзя нарушать
Доменные инварианты и ограничения, влияющие на этот экран, — с ID (ФТ-…, D-…, НФТ-…)
и человеческим пояснением, как правило проявляется в интерфейсе.

## 8. Тон и настроение
Пара предложений: какое впечатление экран должен оставлять, чего избегать.
(Общий тон продукта: тёплая гастрономия, «студия, куда хочется прийти», — не «система бронирования».)

## 9. Свобода дизайнера и открытые вопросы
- **Решает дизайнер:** компоновка, визуальные приёмы, микрокопирайт вне фиксированных формулировок…
- **Открытые вопросы к команде:** если есть.
```
