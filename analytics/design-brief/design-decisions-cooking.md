## Пакет мелких дефолтов (Р-16, приняты разом)

1. **Email клиента в MVP не редактируется** — это идентификатор входа и канал уведомлений; смена — вручную через администратора.
2. **Гостевой доступ**: витрина и карточка класса видны без входа; вход требуется на шаге брони; после входа клиент возвращается к прерванной броне.
3. **Лимиты текстов**: причина отмены слота ≤ 300 символов; комментарий отзыва ≤ 1000; заметка об аллергии ≤ 500; фото программы ≤ 10.
4. **История** броней клиента и классов шефа показывается вся, без обрезания.
5. **Отчёт о недоставленных письмах** после отмены слота — вне MVP (best-effort без обратной связи, НФТ-03).
6. **Отправленный отзыв** клиент видит read-only из «Моих броней»; редактирование не предусмотрено (D-05: один отзыв на бронь).

### Оставлены на усмотрение дизайнера/разработки (не требуют решения заказчика)

- Тип кухни в форме программы: свободный текст с подсказками из уже использованных значений — рекомендация, не требование.
- Шаг сетки времени начала слота, нижняя граница вместимости, ограничение прокатного фонда вместимостью — уточнит разработка при реализации формы SCR-14.
- Показ контакта студии рядом с плашкой блокировки (SCR-08), обязательность полей настроек студии (SCR-17) — на усмотрение дизайнера.
- Кейс «email нового шефа уже занят клиентом» (смена роли) — оставлен открытым в SCR-16; в MVP админ решает вручную (другой email).
