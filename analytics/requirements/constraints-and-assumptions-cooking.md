## 1. Бизнес-правила (инварианты)

Единственный раздел, который в доменной модели формально называется «Бизнес-правила
(инварианты)» — [`domain-model-cooking.md`, строки 192–228](../domain/domain-model-cooking.md#L192-L228).
Попадание сюда по методологии автоматически даёт приоритет **Must** (см.
[requirements-overview-cooking.md](requirements-overview-cooking.md#методология)).

| ID | Правило | Суть | Источник |
|---|---|---|---|
| D-01 | Порог бесплатной отмены | Бесплатна, если до начала слота ≥ 6 часов; иначе — поздняя отмена (`isLateCancellation`) | [строки 194–195](../domain/domain-model-cooking.md#L194-L195) |
| D-02 | Лимит поздних отмен, с периодическим сбросом | 3 поздние отмены/неявки подряд → блок новых броней на 7 дней (`ClientProfile.blockedUntil`), после чего блок и счётчик сбрасываются автоматически | [строки 196–199](../domain/domain-model-cooking.md#L196-L199) |
| D-03 | Запрет двойной брони | Оптимистичная блокировка (`@Version` на `Slot`) + пересчёт `freeSpots` в той же транзакции, что и создание `Booking` | [строки 200–204](../domain/domain-model-cooking.md#L200-L204) |
| D-04 | Запрет повторной записи на отменённый слот | Слот со статусом `CANCELLED_BY_STUDIO` недоступен для новой брони | [строки 205–206](../domain/domain-model-cooking.md#L205-L206) |
| D-05 | Момент доступности отзыва | `Review` создаётся только когда `slot.status = COMPLETED` и бронь не отменена; один отзыв на бронь | [строки 207–208](../domain/domain-model-cooking.md#L207-L208) |
| D-06 | Снапшот цены проката | `Booking.rentalPriceSnapshot` фиксируется при создании брони, задним числом не пересчитывается | [строки 209–210](../domain/domain-model-cooking.md#L209-L210) |
| D-07 | Уведомления по email | Отмена студией — email; напоминание — два письма, за 24ч и за 2ч до `slot.startAt`; сбой отправки — best-effort, не блокирует транзакцию и не влияет на другое напоминание | [строки 211–216](../domain/domain-model-cooking.md#L211-L216) |
| D-08 | Заведение аккаунтов Chef | Только через Admin по email; самостоятельная регистрация шефа не предусмотрена | [строки 217–219](../domain/domain-model-cooking.md#L217-L219) |
| D-09 | Автозавершение слота | По наступлении `startAt + durationMinutes` слот и его подтверждённые брони автоматически переходят в `COMPLETED`; ручного «завершить» у Admin нет | [строки 220–222](../domain/domain-model-cooking.md#L220-L222) |
| D-10 | Запрет пересечения слотов шефа | Нельзя создать слот, пересекающийся по времени с другим слотом того же шефа в статусе `SCHEDULED` | [строки 223–225](../domain/domain-model-cooking.md#L223-L225) |
| D-11 | Снятие ошибочной неявки | Admin может снять `NO_SHOW` (бронь → `COMPLETED`), счётчик и блокировка клиента пересчитываются; у Chef обратного действия нет | [строки 226–228](../domain/domain-model-cooking.md#L226-L228) |

## 2. Проектные ограничения

Отдельного раздела под них в доменной модели нет — собраны из разных мест документа (и в одном
случае — напрямую из брифа, без прохождения через доменную модель).

| Ограничение | Формулировка | Источник |
|---|---|---|
| Платформа | Веб-приложение (фронт + бэкенд на Spring Boot), без отдельного мобильного приложения | [строки 5–10](../domain/domain-model-cooking.md#L5-L10) (НФТ-01) |
| Вместимость класса | 12 человек по умолчанию, 8 — если класс требует сложного оборудования (`requiresComplexEquipment`) | [строка 138](../domain/domain-model-cooking.md#L138) (ФТ-02) |
| Длительность класса | Фиксированно 180 минут, не настраивается на уровне слота | [строка 150](../domain/domain-model-cooking.md#L150) |
| Механизм аутентификации | Только email + одноразовый шестизначный код для всех трёх ролей; без SMS-шлюза и без паролей | [строки 86–92](../domain/domain-model-cooking.md#L86-L92) (НФТ-06) |
| Вне скоупа MVP | Реальная онлайн-оплата; программа лояльности / отметки постоянных клиентов; web push / мобильные уведомления | [«Вне скоупа MVP», строки 230–234](../domain/domain-model-cooking.md#L230-L234) (БТ-08, приоритет Won't) |
| Регистрация клиента | Отдельного flow регистрации нет: аккаунт клиента создаётся молчаливо при первом успешном входе по коду, имя запрашивается одним полем после входа | [«Вход и первая регистрация клиента», строки 257–264](../domain/domain-model-cooking.md#L257-L264) (ФТ-21) |
| Срок и бюджет *(из брифа, не из доменной модели)* | Старт к началу нового сезона (~2 месяца на момент письма); бюджет ограниченный — «начинаем с самого нужного, навороты потом» | [`brief-cooking.md`, строка 37](../customer-brief/brief-cooking.md#L37) |

Последняя строка — единственное в этом файле ограничение, процитированное напрямую из брифа, а
не выведенное из доменной модели. Формально оно и есть содержательное обоснование того, зачем в
проекте вообще есть MoSCoW-приоритизация и список «Вне скоупа MVP»: именно эти два ограничения
(срок и бюджет) — причина, по которой часть функциональности сознательно откладывается.

## 3. Допущения и открытые вопросы

Все прежние допущения и открытые вопросы закрыты (см. «Журнал решений» в
[requirements-overview-cooking.md](requirements-overview-cooking.md#журнал-решений)) — на момент
ревью в файлах не осталось ни одной активной пометки «Предположение» или «Открытый вопрос».
Последним пакетом закрыты 16 вопросов, собранных при постановке экранов на дизайн (журнал
решений, п. 8; полный перечень — [design-decisions-cooking.md](../design-brief/design-decisions-cooking.md)). →
[«Открытые вопросы», строки 236–241](../domain/domain-model-cooking.md#L236-L241).
Два мелких вопроса уровня интерфейса (длительность сессии; email нового шефа занят клиентом)
остаются в постановках на дизайн и на требования не влияют.

**Логические выводы** *(не то же самое, что допущения — это факт, выведенный из модели, а не
предположенный)*:

- Что связанные подтверждённые брони при отмене слота студией переходят в статус
  `CANCELLED_BY_STUDIO` — явного описания этого шага в тексте домена нет, но статус
  `Booking.CANCELLED_BY_STUDIO` не имеет другого описанного триггера (см. UC-03, шаг 2).
- Приоритет Should для просмотра клиентом среднего рейтинга шефа (US-06) выведен из самого
  наличия поля `ChefProfile.averageRating`, а не из прямой формулировки в тексте.
