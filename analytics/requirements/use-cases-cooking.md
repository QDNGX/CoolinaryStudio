## UC-01 — Клиент бронирует слот

- **Актёр:** Клиент
- **Предусловия:** пользователь аутентифицирован как `CLIENT` (см. ФТ-13); `ClientProfile.blockedUntil` пуст или уже в прошлом; существует `Slot` со статусом `SCHEDULED` и `freeSpots > 0`
- **Основной поток:**
  1. Клиент выбирает слот (программа, время, шеф) из расписания.
  2. Клиент указывает экипировку: `OWN` или `RENTAL`.
  3. Если `RENTAL` — система фиксирует текущую `Slot.rentalPricePerSet` в `Booking.rentalPriceSnapshot` (D-06).
  4. В одной транзакции: проверка версии `Slot`, уменьшение `freeSpots`, создание `Booking` со статусом `CONFIRMED` (D-03).
- **Альтернативные потоки:**
  - A1 — активная блокировка (`now() < blockedUntil`): система отказывает в брони (D-02).
  - A2 — слот в статусе `CANCELLED_BY_STUDIO`: бронирование запрещено (D-04).
  - A3 — конфликт версии/нет мест на момент фиксации: retry или явный отказ (D-03).
- **Постусловие:** `Booking` создан (`CONFIRMED`), `freeSpots` уменьшено; при `RENTAL` цена зафиксирована неизменно.

## UC-02 — Клиент отменяет бронь

- **Актёр:** Клиент
- **Предусловие:** у клиента есть `Booking` со статусом `CONFIRMED`
- **Основной поток:**
  1. Клиент инициирует отмену брони.
  2. Система фиксирует `cancelledAt`.
  3. Система вычисляет `isLateCancellation = slot.startAt − cancelledAt < 6ч` (D-01).
  4. `Booking.status → CANCELLED_BY_CLIENT`.
  5. Если `isLateCancellation = true` — `lateCancelCount += 1`; если после инкремента `lateCancelCount ≥ 3` — `ClientProfile.blockedUntil = now() + 7 дней` (D-02).
- **Постусловие:** бронь отменена; при поздней отмене — счётчик клиента обновлён, при достижении лимита клиент временно (~7 дней) не может создавать новые брони, после чего блок и счётчик сбрасываются автоматически (см. UC-01/A1).

## UC-03 — Администратор отменяет слот (форс-мажор/решение студии)

- **Актёр:** Администратор
- **Предусловие:** слот в статусе `SCHEDULED`
- **Основной поток:**
  1. Администратор переводит `Slot.status → CANCELLED_BY_STUDIO` и указывает `cancellationReason`.
  2. Связанные подтверждённые брони переходят в статус `CANCELLED_BY_STUDIO` — *вывод*: явного описания этого шага в тексте нет, но статус `Booking.CANCELLED_BY_STUDIO` (строка 169) не имеет другого описанного триггера.
  3. Затронутым клиентам отправляется email с причиной отмены (D-07, best-effort — сбой отправки не блокирует операцию).
  4. Слот недоступен для новых броней (D-04).
- **Постусловие:** слот и брони отражают отмену; клиенты уведомлены (best-effort); повторная запись невозможна.

## UC-04 — Клиент оставляет отзыв

- **Актёр:** Клиент
- **Предусловие:** `slot.status = COMPLETED`, бронь не отменена, отзыв на эту бронь ещё не существует (D-05)
- **Основной поток:**
  1. Клиент открывает завершённую бронь.
  2. Указывает `chefRating` (1–5), `programRating` (1–5), опционально `comment`.
  3. Система создаёт `Review`, привязанный к `Booking`.
  4. `ChefProfile.averageRating`/`reviewsCount` пересчитываются.
- **Альтернативные потоки:**
  - A1 — слот не `COMPLETED` или бронь отменена: отклонено (D-05).
  - A2 — отзыв на эту бронь уже есть: отклонено (D-05).
- **Постусловие:** отзыв сохранён; рейтинг шефа обновлён.

## UC-05 — Аутентификация пользователя (passwordless)

- **Актёр:** Клиент / Шеф / Администратор
- **Предусловие:** существует `User` с указанным `email`
- **Основной поток:**
  1. Пользователь вводит email.
  2. Система отправляет одноразовый шестизначный код на email.
  3. Пользователь вводит код.
  4. Система проверяет код (соответствие, срок действия, число попыток).
  5. Успех — пользователь аутентифицирован под своей ролью.
- **Альтернативный поток:** A1 — код просрочен или превышено число попыток: отказ.
- **Постусловие:** сессия установлена; пароль не используется ни для одной роли.

## UC-06 — Проставление статуса «неявка» (`NO_SHOW`)

- **Актёр:** Шеф
- **Предусловие:** бронь существует
- **Основной поток:**
  1. Шеф отмечает `Booking` статусом `NO_SHOW`.
  2. `lateCancelCount` клиента увеличивается на 1 (D-02).
  3. При достижении 3 — `ClientProfile.blockedUntil = now() + 7 дней`, новые брони клиента блокируются до этой даты.
- **Постусловие:** статус брони — `NO_SHOW`; счётчик клиента обновлён.
- **Примечание:** ранее вопрос, кто вправе проставлять `NO_SHOW` (Chef, Admin или оба), был открытым — решено оставить только Chef, так как он лично ведёт класс и видит фактическую явку.

## UC-07 — Администратор ведёт каталог и расписание

- **Актёр:** Администратор
- **Основной поток:**
  1. Создание/редактирование `Program` (название, описание, кухня, сложность, признак сложного оборудования, блюда, фото).
  2. Создание `Slot` с привязкой `Program` и `ChefProfile`, временем начала, вместимостью (см. ФТ-02), прокатным фондом и ценой.
- **Постусловие:** программа в каталоге; слот доступен клиентам при статусе `SCHEDULED`.

## UC-08 — Администратор заводит аккаунт шефа

- **Актёр:** Администратор
- **Основной поток:**
  1. Администратор создаёт `User(role=CHEF)` по email нового/приглашённого шефа.
  2. Отдельный пароль не создаётся — первый вход через тот же passwordless-код (UC-05).
- **Постусловие:** у шефа есть учётная запись.
- **Примечание:** «шеф не регистрируется самостоятельно» подтверждено как решение (было допущением в предыдущей версии домена).

## UC-09 — Система отправляет email-напоминания о записи

- **Актёр:** Система (фоновый/плановый процесс)
- **Предусловие:** существует `Booking` со статусом `CONFIRMED`, для которой наступил один из двух моментов напоминания — за 24 часа или за 2 часа до `slot.startAt`
- **Основной поток:**
  1. Система находит подтверждённые брони, для которых наступил момент напоминания (за 24ч или за 2ч до начала слота).
  2. Система отправляет клиенту email-напоминание на `User.email`.
- **Альтернативный поток:** A1 — сбой отправки одного из двух напоминаний: не блокирует и не откатывает никакую транзакцию брони и не влияет на отправку другого напоминания (best-effort, D-07).
- **Постусловие:** клиент получил напоминание (best-effort, не гарантированно при сбое почтового провайдера).
- **Примечание:** момент отправки раньше был открытым вопросом в домене; закрыт решением
  «24ч и 2ч до `slot.startAt`» (см. журнал решений в requirements-overview-cooking.md, п. 7).
