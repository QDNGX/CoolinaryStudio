## 1. Назначение и охват

Модуль реализует публичную витрину студии: ленту предстоящих классов и карточку класса, достаточную для решения «иду / не иду».

| Экран | Название | Роли |
|---|---|---|
| SCR-03 | Расписание классов | Клиент / Гость (просмотр без входа) |
| SCR-04 | Карточка класса (слота) | Клиент / Гость (просмотр без входа) |

**Гостевой доступ (Р-16 п. 2):** витрина и карточка класса видны без входа; аутентификация (ФТ-13) требуется только на шаге создания брони, не раньше; после входа клиент возвращается к прерванной броне, а не на общее расписание. В API это отражено `security: []` у операций `GET /slots`, `GET /slots/{slotId}`, `GET /programs`, `GET /programs/{programId}`, `GET /studio-settings`.

**Платформа:** адаптивный веб — десктоп + мобильный браузер (НФТ-01, брифы SCR-03/SCR-04).

Само оформление брони (SCR-05), вход (SCR-01/02) и «Мои брони» (SCR-06) — вне этого модуля; витрина лишь ведёт к ним.

## 2. Пользовательские сценарии

1. **Просмотр расписания по умолчанию.** Гость или клиент открывает SCR-03 и видит классы на ближайшие 7 дней от текущего момента, сгруппированные по дням, от ближайшего (ФТ-20, US-15, бриф SCR-03 §4).
2. **Смена периода фильтром по датам.** Пользователь выбирает период дальше ближайших 7 дней. Фильтр по датам — **единственный** фильтр витрины: фильтров по кухне, сложности или шефу в MVP нет (Р-11, ФТ-20). Есть быстрый возврат к периоду «ближайшие 7 дней» (бриф SCR-03 §5).
3. **Переход в карточку класса.** Основное (единственное основное) действие ленты — открыть карточку выбранного класса, SCR-04 (бриф SCR-03 §5). Бронирование из ленты невозможно — решение принимается в карточке.
4. **Переход к оформлению брони.** Основное действие карточки — перейти к бронированию (SCR-05). Гость без входа по этому действию сначала попадает на вход (SCR-01/02) и затем возвращается к оформлению брони этого же слота (ФТ-13, UC-01, Р-16 п. 2, бриф SCR-04 §3, §5).
5. **Вход в карточку по прямой ссылке.** Карточку могут открыть по пересланной ссылке (в т.ч. на уже отменённый или прошедший класс) — карточка обязана корректно показать эти состояния (бриф SCR-04 §3, §6).

## 3. Интерфейс: экраны и состояния

### 3.1 SCR-03 — Расписание классов

Элементы (приоритеты — из брифа SCR-03 §4):

| # | Элемент | Содержимое | Источник данных |
|---|---|---|---|
| 1 | Лента классов на 7 дней | Карточки предстоящих классов, сгруппированы по дням, от ближайшего; день без классов можно не показывать или показывать свёрнуто | `GET /slots` (список `SlotSummary`) |
| 2 | Карточка класса в ленте | Название программы, дата и время начала, шеф (имя + средний рейтинг), число свободных мест | `SlotSummary`: `programTitle`, `startAt`, `chef.name`, `chef.averageRating` (+ `chef.reviewsCount`), `freeSpots` |
| 3 | Фильтр по датам | Выбор периода дальше ближайших 7 дней; визуально вторичен, не конкурирует с лентой | Параметры `dateFrom`/`dateTo` |
| 4 | Отметка «класс отменён студией» | Отменённый класс либо скрыт, либо явно помечен как недоступный — решает дизайнер; запись с него невозможна в любом варианте (D-04, ФТ-05) | `SlotSummary.status = CANCELLED_BY_STUDIO` |
| 5 | Служебное | Навигация сайта, вход/профиль; для гостя — ненавязчивое приглашение войти | — |

Состояния (бриф SCR-03 §6):

| Состояние | Когда | Что видит пользователь |
|---|---|---|
| По умолчанию | Есть классы на ближайшие 7 дней | Афиша недели: классы по дням; у каждого — программа, время, шеф с рейтингом, свободные места |
| Загрузка | Расписание запрашивается | Скелетон/плейсхолдеры карточек; сетка не «прыгает» при появлении данных |
| Пусто | На ближайшие 7 дней нет ни одного доступного класса | Empty state с текстом **«Пока нет доступных классов»** (формулировка фиксирована — ФТ-20) + заметное предложение посмотреть более поздние даты через фильтр |
| Пусто (по фильтру) | В выбранном пользователем периоде классов нет | Тот же дружелюбный тон + быстрый сброс фильтра к «ближайшим 7 дням» |
| Ошибка | Расписание не удалось загрузить | Человечное сообщение и действие «попробовать снова»; без технических деталей |
| Класс без мест | `freeSpots = 0` | Класс виден (интерес к студии не прячем), состояние «мест нет» явно читается; карточку открыть можно |
| Класс отменён студией | `status = CANCELLED_BY_STUDIO` | Класс недоступен для записи; скрыть или пометить — решает дизайнер (бриф SCR-03 §9); пути к бронированию нет (D-04, ФТ-05) |

### 3.2 SCR-04 — Карточка класса (слота)

Элементы (приоритеты — из брифа SCR-04 §4):

| # | Элемент | Содержимое | Источник данных |
|---|---|---|---|
| 1 | Программа вечера | Название, фото блюд, список блюд, описание, тип кухни, уровень сложности (для новичков / продвинутый) — каталог целиком (ФТ-01) | `SlotDetails.program`: `title`, `photos` (до 10 — Р-16), `dishes`, `description`, `cuisineType`, `difficultyLevel` |
| 2 | Когда и сколько | Дата, время начала, длительность 3 часа; длительность фиксирована — подать как «вечер на три часа», а не параметр (ФТ-03) | `startAt`, `durationMinutes` (всегда 180) |
| 3 | Шеф | Фото, имя, био, средний рейтинг **вместе с** числом отзывов; рейтинг без числа отзывов не показывать (ФТ-12, US-06); тексты отзывов не показываются (Р-07) | `chef` (`ChefPublic`): `photo`, `name`, `bio`, `averageRating`, `reviewsCount` |
| 4 | Места | Свободные места и общая вместимость («осталось 3 из 12»); вместимость 12, у программ со сложным оборудованием — 8 (ФТ-02) | `freeSpots`, `capacityTotal` |
| 5 | Экипировка | Своя или прокатный набор; цена проката за набор — справочно, выбор делается на следующем шаге; оплата проката на месте, не онлайн (БТ-08) | `rentalSetsAvailable`, `rentalPricePerSet` |
| 6 | Служебное | Возврат к расписанию, навигация | — |

Состояния (бриф SCR-04 §6):

| Состояние | Когда | Что видит пользователь |
|---|---|---|
| По умолчанию | `status = SCHEDULED`, `freeSpots > 0` | Полная карточка; действие бронирования активно |
| Загрузка | Данные запрашиваются | Скелетон карточки; фото подгружаются без скачков макета |
| Пусто / не найдено | Ссылка на несуществующий класс (API 404) | Дружелюбное «такого класса нет» + переход к актуальному расписанию |
| Ошибка | Карточку не удалось загрузить | Человечное сообщение и «попробовать снова» |
| Мест нет | `freeSpots = 0` | Карточка читается; действие бронирования выключено с пояснением «места закончились»; предложение вернуться к расписанию |
| Отменён студией | `status = CANCELLED_BY_STUDIO` | Явная плашка «класс отменён студией»; бронирование недоступно (D-04, ФТ-05); приглашение посмотреть другие классы |
| Класс уже прошёл | `status = COMPLETED` (время в прошлом) | Карточка читается как архивная, бронирование недоступно |
| Прокат закончился | `rentalSetsAvailable = 0`, места есть | Бронирование доступно; в блоке экипировки честно видно: только со своей экипировкой |

Формулировки состояний «мест нет» / «класс отменён» — смысл фиксирован, слова на усмотрение дизайнера (бриф SCR-04 §9). Отдельной публичной страницы шефа нет — био, фото и рейтинг показываются прямо в карточке (бриф SCR-04 §9).

## 4. Контракт API

Все операции модуля — публичные (`security: []`), токен не требуется (Р-16 п. 2). Схема ошибки везде — `Error { code, message, details? }`.

### 4.1 GET /slots — лента расписания (SCR-03)

Параметры:

| Параметр | Тип | Дефолт | Правило |
|---|---|---|---|
| `dateFrom` | `date` | текущий момент | Начало периода |
| `dateTo` | `date` | `dateFrom + 7 дней` | Конец периода (ФТ-20 — горизонт 7 дней по умолчанию) |
| `status` | `SlotStatus` | — | Гостю и клиенту по умолчанию отдаются только `SCHEDULED`; параметр предназначен для админской сетки SCR-13 и витриной **не используется** |

Ответ `200` — массив `SlotSummary`, отсортирован по времени начала. Маппинг полей на UI:

| Поле `SlotSummary` | Где отображается |
|---|---|
| `id` | Ссылка на карточку SCR-04 |
| `programId` | Технический идентификатор (для перехода/предзагрузки; прямого отображения нет) |
| `programTitle` | Название класса в карточке ленты |
| `chef.name` | Имя шефа |
| `chef.averageRating` + `chef.reviewsCount` | Компактный рейтинг шефа (агрегат; `averageRating` = null, пока отзывов нет — ФТ-12) |
| `chef.photo`, `chef.bio` | В ленте не обязательны (бриф SCR-03 требует «имя + рейтинг»); использование — на усмотрение дизайна |
| `startAt` | Дата и время начала; группировка по дням |
| `durationMinutes` | Фиксированно 180; в ленте не акцентируется |
| `capacityTotal` | В ленте бриф требует только число свободных мест; показ «из N» — на усмотрение дизайна |
| `freeSpots` | «Осталось N мест»; при 0 — состояние «мест нет» |
| `status` | `SCHEDULED` — обычная карточка; `CANCELLED_BY_STUDIO` — скрыть/пометить (см. §3.1) |
| `rentalSetsAvailable`, `rentalPricePerSet` | В ленте не отображаются (прокат — информация карточки, бриф SCR-03 §4) |

Ошибки и реакция UI:

| Код | Когда | Реакция UI |
|---|---|---|
| `400 ValidationError` | Некорректные параметры дат | Состояние «Ошибка» + сброс фильтра к дефолтным 7 дням |
| Сетевая/5xx | Недоступен бэкенд | Состояние «Ошибка»: человечное сообщение + «попробовать снова» |

Пустой массив `200 []` — не ошибка: empty state «Пока нет доступных классов» (ФТ-20).

### 4.2 GET /slots/{slotId} — карточка слота (SCR-04)

Ответ `200` — `SlotDetails` = `SlotSummary` + `program` (объект `Program` целиком) + `cancellationReason`. Маппинг:

| Поле `SlotDetails` | Где отображается |
|---|---|
| `program.title`, `program.description`, `program.cuisineType`, `program.difficultyLevel`, `program.dishes`, `program.photos` | Блок «Программа вечера» (ФТ-01); фото — до 10 (Р-16) |
| `startAt`, `durationMinutes` (180) | Блок «Когда и сколько» (ФТ-03) |
| `chef.photo`, `chef.name`, `chef.bio`, `chef.averageRating`, `chef.reviewsCount` | Блок «Шеф» (ФТ-12, Р-07 — см. §4.5) |
| `freeSpots`, `capacityTotal` | Блок «Места»: «осталось X из Y» (ФТ-02) |
| `rentalSetsAvailable`, `rentalPricePerSet` | Блок «Экипировка» — справочно (БТ-08: без намёков на чекаут) |
| `status` | Управляет состояниями карточки (§3.2) |
| `cancellationReason` | Заполнена только при `CANCELLED_BY_STUDIO` (ФТ-04); показ причины на карточке брифом не зафиксирован — **на усмотрение разработки/дизайна** (бриф требует только плашку «класс отменён студией») |

Ошибки:

| Код | Когда | Реакция UI |
|---|---|---|
| `404 NotFound` | Слот не существует | Состояние «Пусто / не найдено»: «такого класса нет» + переход к расписанию |
| Сетевая/5xx | Недоступен бэкенд | Состояние «Ошибка» + «попробовать снова» |

### 4.3 GET /programs, GET /programs/{programId} — каталог программ

Чтение доступно всем ролям и гостю (Р-16); архивации нет — каталог только растёт (Р-15). Ответы: `200` — массив `Program` / объект `Program`; у `GET /programs/{programId}` также `404 NotFound`.

Для витрины эти операции **вспомогательные**: карточка слота уже содержит `program` внутри `SlotDetails`, поэтому обязательного обращения к `/programs` из SCR-03/SCR-04 нет. Использование (например, предзагрузка/кэширование каталога) — на усмотрение разработки.

### 4.4 GET /studio-settings — адрес и контакты студии

Синглтон (ФТ-17), читается без входа — адрес показывает, где проходит класс. Ответ `200` — `StudioSettings { address, contactPhone, contactEmail }`. Место и способ показа адреса на витрине брифами SCR-03/SCR-04 не зафиксированы — **на усмотрение разработки/дизайна**.

### 4.5 Рейтинг шефа: только агрегат (Р-07)

Явно и безусловно: на витрине (и в ленте, и в карточке) рейтинг шефа — **только агрегат** `averageRating` + `reviewsCount` из `ChefPublic` (ФТ-12, US-06). Рейтинг показывается **вместе** с числом отзывов; «голая» звезда без числа отзывов не допускается (бриф SCR-04 §7). **Тексты отзывов не показываются никому, кроме администратора** (Р-07): API отдаёт их только через `GET /reviews` (роль ADMIN); в `ChefPublic`, `SlotSummary`, `SlotDetails` текстов отзывов нет, и ленту отзывов на карточке проектировать не нужно. `averageRating = null`, пока отзывов нет — UI в этом случае не показывает рейтинг (подача «нет отзывов» — на усмотрение дизайна).

## 5. Бизнес-правила и валидации

| ID | Правило | Применение в модуле |
|---|---|---|
| ФТ-20 / US-15 | По умолчанию показываются слоты на ближайшие 7 дней от текущего момента; более длинный период — только через фильтр по датам; empty state — «Пока нет доступных классов» (формулировка фиксирована) | Дефолты `dateFrom`/`dateTo` в `GET /slots`; empty state SCR-03 |
| Р-11 | Единственный фильтр витрины — по датам; фильтров по кухне/сложности/шефу в MVP нет | SCR-03: никаких других контролов фильтрации |
| ФТ-02 | Вместимость класса: 12 по умолчанию, 8 при `requiresComplexEquipment = true`; зафиксирована при создании слота (Р-04) | Карточка показывает `freeSpots` и `capacityTotal`; витрина ничего не вычисляет — берёт готовые значения |
| ФТ-03 | Длительность фиксирована: 180 минут | `durationMinutes` всегда 180; на карточке — «вечер на три часа», не настройка |
| D-03 / ФТ-07 | `freeSpots` — кэш, пересчитывается системой в транзакции брони; read-only, меняет только система (матрица доступа ER-модели) | Витрина **только читает** `freeSpots`; не декрементирует, не пересчитывает и не «резервирует» на фронте |
| ФТ-05 / D-04 | Новая бронь на слот `CANCELLED_BY_STUDIO` запрещена навсегда | Из ленты и карточки нет пути к бронированию отменённого слота |
| Правило выдачи | Гостю и клиенту `GET /slots` отдаёт только слоты `SCHEDULED` (описание параметра `status` в openapi.yaml); параметр `status` — для админа | Витрина не передаёт `status` и не полагается на присутствие иных статусов в ленте; карточка по прямой ссылке при этом обязана обрабатывать `CANCELLED_BY_STUDIO` и `COMPLETED` (§3.2) |
| ФТ-12 / US-06 | Средний рейтинг и число отзывов считает система (агрегат по `Review.chefRating`); поля read-only | UI показывает пару «рейтинг + число отзывов» как есть |
| Р-07 | Тексты комментариев отзывов — только администратору | На витрине текстов отзывов нет (§4.5) |
| ФТ-13 / UC-01 / Р-16 п. 2 | Бронирует только вошедший клиент; просмотр — свободный; после входа — возврат к прерванной броне | Барьер входа — на действии «перейти к бронированию» (SCR-04 → SCR-01/02 → SCR-05), с сохранением контекста слота |
| БТ-08 | Онлайн-оплат нет | Цена проката — справочная; никаких элементов чекаута |
| Р-16 п. 3 | Фото программы — до 10 | Галерея карточки рассчитана максимум на 10 фото |
| НФТ-01 | Адаптивный веб | Оба экрана полноценно работают в мобильном браузере |

Валидации ввода в модуле минимальны: единственный ввод пользователя — период дат фильтра. Некорректный период отклоняется API как `400 ValidationError`; клиентские правила виджета дат (например, запрет `dateTo < dateFrom`) — **на усмотрение разработки** (в источниках не зафиксированы).

## 6. Краевые случаи

| Случай | Поведение | Источник / статус |
|---|---|---|
| Пустой период (нет слотов на 7 дней) | Empty state «Пока нет доступных классов» + предложение посмотреть более поздние даты | ФТ-20, бриф SCR-03 §6 |
| Пусто по пользовательскому фильтру | Тот же тон + быстрый сброс к «ближайшим 7 дням» | Бриф SCR-03 §6 |
| `freeSpots = 0` в ленте | Класс **показывается** (не скрывается), «мест нет» читается сразу, карточка открывается; бронь недоступна | Бриф SCR-03 §6 — зафиксировано, показывать |
| `freeSpots = 0` в карточке | Карточка читается, действие брони выключено с пояснением «места закончились», предложение вернуться к расписанию | Бриф SCR-04 §6 |
| Слот отменён студией — в ленте | Скрыть или пометить недоступным — **на усмотрение дизайна** (оба варианта допустимы); запись невозможна в любом случае | Бриф SCR-03 §4, §9; D-04 |
| Слот отменён студией — по прямой ссылке в карточку | Плашка «класс отменён студией», бронь недоступна, приглашение к другим классам; показывать ли текст `cancellationReason` — **на усмотрение разработки/дизайна** (в брифе не зафиксировано) | Бриф SCR-04 §6; ФТ-04 |
| Класс уже прошёл (`COMPLETED`) | Карточка архивная, бронь недоступна | Бриф SCR-04 §6 |
| Прокат закончился (`rentalSetsAvailable = 0`), места есть | Бронь доступна; в блоке экипировки честно видно: только со своей | Бриф SCR-04 §6 |
| Длинный список фото (до 10 — Р-16) | Галерея (слайдер/сетка — на усмотрение дизайнера, бриф SCR-04 §9); фото подгружаются без скачков макета | Р-16 п. 3; бриф SCR-04 §6, §9 |
| `averageRating = null` (отзывов ещё нет) | Рейтинг не показывается как «0»; подача «нет отзывов» — **на усмотрение дизайна** | openapi.yaml (`ChefPublic.averageRating` nullable); прямого правила в брифах нет |
| Гость дошёл до брони | Переход на вход, после входа — возврат к прерванной броне этого слота | Р-16 п. 2, бриф SCR-04 §3 |
| Несуществующий `slotId` (404) | «Такого класса нет» + переход к актуальному расписанию | Бриф SCR-04 §6 |
| День без классов внутри периода | Можно не показывать вовсе или показывать свёрнуто — на усмотрение дизайна | Бриф SCR-03 §4 |

## 7. Вне скоупа

- **Фильтры, кроме дат** — по кухне, сложности, шефу: осознанно нет в MVP (Р-11).
- **Отдельная публичная страница шефа** — не требуется ни одним требованием; всё о шефе — в карточке класса (бриф SCR-04 §9). Предложение такой страницы — расширение скоупа, требует решения команды.
- **Оплата** — онлайн-оплат в продукте нет (БТ-08); цена проката справочная, оплата на месте.
- **Бронирование из ленты** — решение принимается только в карточке (бриф SCR-03 §5).
- **Выбор экипировки и подтверждение брони на карточке** — это шаг оформления SCR-05 (бриф SCR-04 §5).
- **Тексты отзывов на витрине** — только агрегат; тексты видит один администратор (Р-07).
- **Сами экраны входа (SCR-01/02), оформления (SCR-05), «Мои брони» (SCR-06)** — отдельные модули; здесь — только переходы к ним.

## 8. Критерии приёмки

**КП-1. Расписание по умолчанию (ФТ-20)**
Дано: гость без токена; на ближайшие 7 дней есть слоты `SCHEDULED`.
Когда: он открывает SCR-03 (запрос `GET /slots` без параметров).
Тогда: показаны только слоты со `startAt` в интервале «сейчас … сейчас + 7 дней», статус только `SCHEDULED`, сгруппированы по дням от ближайшего; у каждого — название программы, дата/время, шеф с рейтингом, число свободных мест.

**КП-2. Empty state (ФТ-20)**
Дано: на ближайшие 7 дней нет ни одного слота `SCHEDULED`.
Когда: пользователь открывает SCR-03.
Тогда: показан empty state с текстом ровно «Пока нет доступных классов» и предложением посмотреть более поздние даты через фильтр.

**КП-3. Фильтр по датам — единственный (Р-11, ФТ-20)**
Дано: открыт SCR-03.
Когда: пользователь выбирает период за пределами 7 дней.
Тогда: выполняется `GET /slots?dateFrom=…&dateTo=…` и лента показывает слоты выбранного периода; на экране нет других фильтров (кухня/сложность/шеф); доступен возврат к «ближайшим 7 дням».

**КП-4. Гостевой доступ (Р-16 п. 2)**
Дано: пользователь не аутентифицирован.
Когда: он открывает SCR-03 и SCR-04 (в т.ч. по прямой ссылке).
Тогда: оба экрана открываются полностью без запроса входа; запросы `GET /slots`, `GET /slots/{slotId}` уходят без токена и возвращают 200.

**КП-5. Вход на шаге брони с возвратом (ФТ-13, UC-01, Р-16 п. 2)**
Дано: гость на карточке слота `SCHEDULED` с `freeSpots > 0`.
Когда: он нажимает «перейти к бронированию».
Тогда: он направляется на вход (SCR-01/02), а после успешного входа возвращается к оформлению брони **этого же** слота, не на общее расписание.

**КП-6. Карточка целиком (ФТ-01, ФТ-02, ФТ-03, ФТ-12)**
Дано: существует слот `SCHEDULED` со свободными местами.
Когда: пользователь открывает его карточку (`GET /slots/{slotId}` → 200 `SlotDetails`).
Тогда: показаны — программа (название, описание, тип кухни, сложность, блюда, фото до 10), дата/время и длительность «3 часа», шеф (фото, имя, био, средний рейтинг вместе с числом отзывов), места «осталось X из Y», условия экипировки с ценой проката; действие бронирования активно.

**КП-7. Только агрегат рейтинга (Р-07, ФТ-12)**
Дано: у шефа есть отзывы с текстовыми комментариями.
Когда: гость или клиент просматривает SCR-03 и SCR-04.
Тогда: виден только `averageRating` + `reviewsCount`; ни одного текста отзыва нет ни в UI, ни в ответах используемых операций; рейтинг никогда не показан без числа отзывов.

**КП-8. Мест нет (бриф SCR-03/SCR-04 §6)**
Дано: слот `SCHEDULED` с `freeSpots = 0`.
Когда: пользователь видит его в ленте и открывает карточку.
Тогда: в ленте класс показан с явным состоянием «мест нет» и карточка открывается; на карточке действие бронирования выключено с пояснением «места закончились» и предложением вернуться к расписанию.

**КП-9. Слот отменён студией (ФТ-05, D-04)**
Дано: слот в статусе `CANCELLED_BY_STUDIO`; у пользователя прямая ссылка на его карточку.
Когда: он открывает карточку.
Тогда: показана явная плашка «класс отменён студией», действие бронирования недоступно, предложены другие классы; из ленты SCR-03 путь к бронированию этого слота также отсутствует (скрыт или помечен недоступным).

**КП-10. Несуществующий слот**
Дано: ссылка на `slotId`, которого нет (API вернёт 404).
Когда: пользователь открывает её.
Тогда: показано дружелюбное «такого класса нет» с переходом к актуальному расписанию; технические детали ошибки не показываются.

**КП-11. Ошибка загрузки**
Дано: `GET /slots` (или `GET /slots/{slotId}`) завершился сетевой ошибкой/5xx.
Когда: пользователь находится на SCR-03 (или SCR-04).
Тогда: показано человечное сообщение об ошибке с действием «попробовать снова»; повторное нажатие повторяет запрос.

**КП-12. Прокат закончился (бриф SCR-04 §6)**
Дано: слот `SCHEDULED`, `freeSpots > 0`, `rentalSetsAvailable = 0`.
Когда: пользователь открывает карточку.
Тогда: бронирование доступно; в блоке экипировки видно, что прийти можно только со своей экипировкой.

**КП-13. freeSpots read-only (D-03)**
Дано: открыта лента или карточка.
Когда: UI отображает свободные места.
Тогда: значение берётся строго из ответа API (`freeSpots`); фронт нигде его не вычисляет и не изменяет.
