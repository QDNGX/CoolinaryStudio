# ТЗ-05 — Отзыв о классе

> Модуль веб-приложения «Шеф-стол» (Spring Boot + веб-фронт; роли `CLIENT`/`CHEF`/`ADMIN`).
> Документ ничего не добавляет к аналитике — каждое утверждение снабжено ID источника.
> Источники: [бриф SCR-07](../design-brief/briefs/scr-07-client-review-form.md),
> [openapi.yaml](../api/openapi.yaml),
> [UC-04](../requirements/use-cases-cooking.md),
> [ФТ-11, ФТ-12](../requirements/functional-requirements-cooking.md),
> [D-05, D-09](../requirements/constraints-and-assumptions-cooking.md),
> [Р-07, Р-10, Р-16](../design-brief/design-decisions-cooking.md).

## 1. Назначение и охват

Модуль реализует отзыв клиента о завершённом классе — экран **SCR-07** и связанные операции API.

Охват:

- **создание отзыва клиентом** по своей завершённой брони: две обязательные оценки (шефу и
  программе, шкала 1–5) и необязательный комментарий (UC-04, ФТ-11);
- **чтение клиентом собственного отзыва** в режиме read-only — редактирование и удаление
  не предусмотрены (Р-16, п. 6; D-05 — один отзыв на бронь);
- **чтение всех отзывов администратором** с текстами комментариев — тексты видит только
  администратор, шеф текстов не видит, публичен лишь агрегат «средний рейтинг + число отзывов»
  (Р-07);
- **автоматический пересчёт агрегатов шефа** `averageRating`/`reviewsCount` при создании отзыва
  (ФТ-12).

Бизнес-цель — **БТ-05**: дать клиентам возможность оценивать шефа и программу после класса как
сигнал качества работы шефов (см.
[business-requirements-cooking.md](../requirements/business-requirements-cooking.md)).
Комментарии — внутренний сигнал качества для студии (Р-07), оценка шефу — вход в публичный
рейтинг (ФТ-12).

Вне охвата модуля: редактирование/удаление отзыва (Р-16, п. 6 — не предусмотрены в MVP),
модерация комментариев (в источниках отсутствует), отображение агрегата рейтинга в карточке
класса (ФТ-12 — сторона витрины, отдельный модуль; здесь — только пересчёт).

Приоритет: **Should** (ФТ-11, ФТ-12, UC-04).

## 2. Пользовательские сценарии

### 2.1. Оставить отзыв по завершённой брони (UC-04, основной поток)

1. Слот завершается **автоматически по времени** — `startAt` + 180 мин (Р-10, D-09);
   подтверждённые брони переходят в `COMPLETED`, ручной кнопки «завершить» у администратора нет.
   Форма отзыва открывается **сразу после автозавершения** (Р-10) — никакого дополнительного
   ожидания или ручного действия студии.
2. Клиент из списка «Мои брони» (SCR-06) по действию «Оставить отзыв» на завершённой записи без
   отзыва попадает на SCR-07 (бриф SCR-07, разд. 3). Возможен вход по прямой ссылке — тогда экран
   сам проверяет доступность отзыва при открытии (бриф SCR-07, разд. 3 и 7).
3. Клиент выставляет `chefRating` (1–5) и `programRating` (1–5), опционально пишет комментарий
   до 1000 символов (UC-04, шаг 2; Р-16, п. 3) и отправляет.
4. Система создаёт `Review`, привязанный к `Booking`, и пересчитывает
   `ChefProfile.averageRating`/`reviewsCount` (UC-04, шаги 3–4; ФТ-12).
5. Клиент видит короткую благодарность и возвращается в «Мои брони», где у записи вместо кнопки —
   отметка «отзыв оставлен» (бриф SCR-07, разд. 3; в API это поле `hasReview` брони).

### 2.2. Попытка повторного отзыва (UC-04/A2)

Один отзыв на одну бронь (D-05). Повторный вход на SCR-07 по уже оценённой записи **не показывает
пустую форму** — экран отображает отправленный отзыв в режиме просмотра (бриф SCR-07, разд. 5–6).
Если запрос на создание всё же дошёл до API (гонка, прямой вызов) — сервер отвечает `409`
(UC-04/A2, openapi `createReview`).

### 2.3. Просмотр отправленного отзыва (Р-16, п. 6)

Клиент из «Моих броней» открывает запись с отметкой «отзыв оставлен» и видит свой отзыв
**read-only**: обе оценки и комментарий, без возможности изменить, с путём назад в «Мои брони»
(бриф SCR-07, состояние «Отзыв уже оставлен»; API `getBookingReview`). Редактирование не
предусмотрено (Р-16, п. 6); экран не должен намекать на обратное — никаких «сохранить черновик»,
«изменить позже» (бриф SCR-07, разд. 5).

### 2.4. Отказ от заполнения

Клиент уходит с экрана без отправки — назад в «Мои брони», без каких-либо последствий
(бриф SCR-07, разд. 3).

## 3. Интерфейс: экран и состояния (SCR-07)

Источник — [бриф SCR-07](../design-brief/briefs/scr-07-client-review-form.md).
Роль — клиент; платформа — адаптивный веб (десктоп + мобильный браузер, НФТ-01).

### 3.1. Элементы экрана (бриф SCR-07, разд. 4)

| # | Элемент | Требование |
|---|---|---|
| 1 | Оценка шефу | Шкала 1–5, **обязательна**; рядом — имя и фото шефа. Без преднажатого значения по умолчанию (бриф SCR-07, разд. 7) |
| 2 | Оценка программе | Шкала 1–5, **обязательна**; рядом — название программы. Две шкалы читаются как два разных вопроса. Без значения по умолчанию |
| 3 | Контекст класса | Название программы, дата и время прошедшего класса, шеф — шапка-напоминание «о чём речь» |
| 4 | Комментарий | Свободный текст, **необязателен**, ≤ 1000 символов (Р-16, п. 3); необязательность помечена явно; лимит виден до упора в него, а не обрубает текст ошибкой при отправке (бриф SCR-07, разд. 7) |
| 5 | Кнопка отправки | Неактивна, пока не выставлены обе оценки, — с понятной подсказкой почему (бриф SCR-07, разд. 5) |
| 6 | Пояснение о публичности | Оценка шефу влияет на публичный рейтинг; текст комментария публично не показывается — его читает только администратор студии (Р-07) |

Форма шкалы (звёзды/цифры), порядок блоков, формулировки подсказок, вид счётчика символов и
оформление read-only-режима — на усмотрение дизайнера (бриф SCR-07, разд. 9).

### 3.2. Состояния экрана (бриф SCR-07, разд. 6)

| Состояние | Когда возникает | Что видит пользователь |
|---|---|---|
| По умолчанию | Отзыв по записи доступен, форма пуста | Контекст класса, две пустые шкалы, пустой комментарий, отправка неактивна |
| Загрузка | Проверяется доступность отзыва и грузится контекст класса | Плейсхолдер формы без прыжков вёрстки |
| Частично заполнено | Выставлена одна оценка из двух | Отправка неактивна; ненавязчивая подсказка, чего не хватает («осталось оценить программу») |
| Отправка | Клиент нажал отправить | Кнопка в состоянии ожидания, повторное нажатие невозможно (защита от дублей) |
| Ошибка отправки | Технический сбой | Введённое не теряется; «не получилось отправить» + повторить |
| Успех | Отзыв сохранён (201) | Короткая тёплая благодарность, возврат в «Мои брони» |
| Отзыв уже оставлен | Повторный вход по оценённой записи | Отправленный отзыв read-only: обе оценки и комментарий, без изменения, путь назад (Р-16, п. 6) |
| Отзыв недоступен | Прямая ссылка: класс не завершён, запись отменена / неявка | Не пустая форма и не голая ошибка — объяснение конкретной причины человеческим языком и путь назад в «Мои брони» |

Ключевое правило UX: экран **проверяет условия доступности при открытии, а не при отправке** —
клиент не должен заполнить форму и лишь потом узнать, что отзыв невозможен (бриф SCR-07, разд. 7,
D-05/ФТ-11).

## 4. Контракт API

Источник — [../api/openapi.yaml](../api/openapi.yaml). Все операции требуют bearer-токен
(passwordless-вход, ФТ-13); гостевого доступа к отзывам нет.

### 4.1. `POST /bookings/{bookingId}/review` — создать отзыв (CLIENT)

`operationId: createReview`. Тело запроса — `ReviewCreateRequest`:

| Поле | Тип | Ограничения | Обязательность |
|---|---|---|---|
| `chefRating` | integer | 1–5 | да |
| `programRating` | integer | 1–5 | да |
| `comment` | string, nullable | ≤ 1000 символов (Р-16, п. 3); текст видит только ADMIN (Р-07) | нет |

Ответы и реакция UI:

| Код | Смысл (по openapi) | Реакция UI (по брифу SCR-07) |
|---|---|---|
| `201` | Отзыв сохранён, рейтинг шефа обновлён (тело — `Review`); агрегаты `averageRating`/`reviewsCount` пересчитаны системой (ФТ-12) | Состояние «Успех»: благодарность, возврат в «Мои брони», у записи отметка «отзыв оставлен» |
| `400` | Ошибка валидации (оценка вне 1–5, комментарий > 1000) | При соблюдении разд. 3 на клиенте недостижим штатно; при получении — состояние «Ошибка отправки», введённое не теряется |
| `401` | Не аутентифицирован | Переход на вход (passwordless, UC-05); после входа клиент возвращается к прерванному действию |
| `403` | Бронь принадлежит другому клиенту | Состояние «Отзыв недоступен»: объяснение и путь назад в «Мои брони» |
| `404` | Бронь не найдена | Состояние «Отзыв недоступен»: объяснение и путь назад |
| `409` | Отзыв на эту бронь уже существует (D-05, UC-04/A2; код ошибки `REVIEW_ALREADY_EXISTS`) | Состояние «Отзыв уже оставлен»: показать отправленный отзыв read-only (через `GET .../review`), не пустую форму |
| `422` | Класс не завершён или бронь отменена (D-05, UC-04/A1) | Состояние «Отзыв недоступен»: конкретная причина человеческим языком, путь назад |

Тело ошибки — схема `Error` (`code`, `message`, `details`); `message` — человекочитаемый текст
для UI.

Защита от двойного сабмита — на клиенте: кнопка блокируется на время отправки (бриф SCR-07,
состояние «Отправка»); на сервере повтор гасится правилом «один отзыв на бронь» → `409` (D-05).

### 4.2. `GET /bookings/{bookingId}/review` — свой отзыв read-only (CLIENT)

`operationId: getBookingReview`. Р-16, п. 6 — отправленный отзыв клиент видит read-only из
«Моих броней»; редактирование не предусмотрено.

| Код | Смысл | Реакция UI |
|---|---|---|
| `200` | Тело — `Review` (`id`, `bookingId`, `chefRating`, `programRating`, `comment`, `createdAt`) | Состояние «Отзыв уже оставлен»: обе оценки и комментарий без возможности изменить |
| `401` | Не аутентифицирован | Переход на вход |
| `403` | Бронь принадлежит другому клиенту | «Отзыв недоступен» + путь назад |
| `404` | Бронь не найдена или отзыва по ней нет | Если бронь доступна для отзыва — показать пустую форму; иначе «Отзыв недоступен» |

В ответе для клиента `comment` — его собственный комментарий; полный доступ к текстам — только
ADMIN (Р-07, описание схемы `Review`).

### 4.3. `GET /reviews?chefId&programId` — все отзывы с текстами (ADMIN)

`operationId: listReviews`. Р-07: текстовые комментарии — внутренний сигнал качества (БТ-05),
их читает **только администратор**. Публичен только агрегат (средний рейтинг + число отзывов в
карточке класса). **Шеф тексты не видит** — эндпоинта чтения текстов для роли `CHEF` в API нет,
и он не должен появиться.

Параметры (оба необязательны, комбинируются):

| Параметр | Тип | Смысл |
|---|---|---|
| `chefId` | uuid | Фильтр по шефу (путь Review → Booking → Slot → Chef) |
| `programId` | uuid | Фильтр по программе |

| Код | Смысл |
|---|---|
| `200` | Массив `Review`, от новых к старым |
| `401` | Не аутентифицирован |
| `403` | Роль не ADMIN (в т.ч. CHEF и CLIENT) |

## 5. Бизнес-правила

| ID | Правило | Что это значит для модуля |
|---|---|---|
| D-05 | `Review` создаётся только когда `slot.status = COMPLETED` и бронь не отменена; **один отзыв на бронь** | Сервер проверяет оба условия при создании: нарушение условий доступности → `422`, дубль → `409`. Уникальность отзыва по `bookingId` обеспечивается на уровне данных (один-к-одному Review–Booking). Редактирование/удаление не предусмотрены (Р-16, п. 6) |
| ФТ-11 | Отзыв = оценка шефу 1–5 + оценка программе 1–5 + необязательный комментарий, с проверкой условий доступности | Обе оценки обязательны, значений «по умолчанию» нет (бриф SCR-07, разд. 7); комментарий ≤ 1000 символов (Р-16, п. 3) |
| ФТ-12 | `ChefProfile.averageRating`/`reviewsCount` — агрегат по `Review.chefRating`, рассчитывает система | Пересчёт выполняется при создании отзыва (UC-04, шаг 4); поля производные — ни одна роль не меняет их напрямую (openapi, «Производные поля»). Пока отзывов нет — `averageRating = null` (openapi, схема ChefProfile) |
| Р-07 | Тексты комментариев видит только ADMIN; публичен только агрегат; шеф тексты не видит | `GET /reviews` — только ADMIN (`403` остальным); клиент видит лишь свой собственный комментарий через `GET /bookings/{id}/review`; в интерфейсах шефа комментарии не показываются нигде |
| Р-10 / D-09 | Слот завершается автоматически по `startAt + 180 мин`; брони `CONFIRMED → COMPLETED` | Отзыв становится доступен сразу после автозавершения, без ручных действий студии |
| Р-16 (п. 3, 6) | Комментарий ≤ 1000 символов; отправленный отзыв — read-only | Лимит валидируется и на клиенте (видимый лимит до упора), и на сервере (`400`); эндпоинтов PUT/PATCH/DELETE для отзыва нет |
| UC-04 (финальность) | После успешной отправки отзыв создан окончательно | Повторный вход на экран не предлагает форму заново (бриф SCR-07, разд. 7) |

## 6. Краевые случаи

1. **Бронь `NO_SHOW`** — отзыв **недоступен**: бриф SCR-07 прямо требует этого дважды
   (разд. 7: «класс завершён и запись не отменена **(и не является неявкой)**»; разд. 6,
   состояние «Отзыв недоступен»: «запись отменена / **неявка**»).
   ⚠️ **Открытый вопрос (расхождение источников):** формулировка D-05 — «бронь не отменена» —
   формально NO_SHOW не покрывает: неявка не является отменой (статусы `CANCELLED_BY_CLIENT` /
   `CANCELLED_BY_STUDIO` отличны от `NO_SHOW`, ФТ-19). Описание `422` в openapi («класс не
   завершён или бронь отменена») случай NO_SHOW тоже не называет.
   **Рекомендация:** реализовать по брифу SCR-07 — отзыв доступен только по брони в статусе
   `COMPLETED` (это автоматически исключает и отмены, и `NO_SHOW`); ответ API для NO_SHOW —
   `422`; уточнить формулировку D-05 и описание `422` в openapi у аналитика.
2. **Двойной сабмит.** Клиент дважды нажимает «отправить» (медленная сеть). Первая линия защиты —
   UI: в состоянии «Отправка» повторное нажатие невозможно (бриф SCR-07, разд. 6). Вторая —
   сервер: правило «один отзыв на бронь» (D-05) — второй запрос получает `409`
   (`REVIEW_ALREADY_EXISTS`); UI на `409` показывает отправленный отзыв read-only, а не ошибку.
3. **Комментарий из одних пробелов.** Комментарий необязателен (ФТ-11); в источниках правила
   нормализации нет — ⚠️ **открытый вопрос**. **Рекомендация:** трактовать строку из одних
   пробельных символов как отсутствие комментария (сохранять `null` после trim), не отклонять
   запрос — это согласуется с необязательностью поля и приглашающим тоном экрана
   (бриф SCR-07, разд. 8). Согласовать с аналитиком.
4. **Прямая ссылка на SCR-07** (закладка): экран сам проверяет доступность отзыва при открытии
   и показывает соответствующее состояние — форму, read-only-отзыв или «отзыв недоступен» с
   причиной (бриф SCR-07, разд. 3 и 6).
5. **Технический сбой при отправке**: введённые оценки и текст не теряются, доступен повтор
   (бриф SCR-07, состояние «Ошибка отправки»).
6. **Снятие NO_SHOW администратором после класса** (D-11, ФТ-22): бронь переходит в `COMPLETED` —
   с этого момента отзыв по ней становится доступен на общих основаниях (D-05). Специальной
   логики в модуле не требуется: проверка выполняется по текущему статусу брони на момент
   открытия экрана/запроса.

## 7. Критерии приёмки

Формат: Дано / Когда / Тогда.

**КП-1. Успешное создание с пересчётом рейтинга (UC-04, ФТ-11, ФТ-12)**
- **Дано:** клиент аутентифицирован; его бронь в статусе `COMPLETED` (слот автозавершён — D-09/Р-10); отзыва по брони нет; у шефа `reviewsCount = N`, `averageRating = A`.
- **Когда:** клиент отправляет `POST /bookings/{id}/review` с `chefRating = 5`, `programRating = 4`, `comment` ≤ 1000 символов.
- **Тогда:** ответ `201` с телом `Review`; у шефа `reviewsCount = N + 1`, `averageRating` пересчитан как среднее по всем `Review.chefRating` (ФТ-12); у брони `hasReview = true`; UI показывает благодарность и возвращает в «Мои брони».

**КП-2. 409 при повторном отзыве (D-05, UC-04/A2)**
- **Дано:** по брони клиента уже существует отзыв.
- **Когда:** клиент повторно отправляет `POST /bookings/{id}/review` с валидным телом.
- **Тогда:** ответ `409` (код `REVIEW_ALREADY_EXISTS`); второй `Review` не создан; агрегаты шефа не изменились; UI показывает отправленный отзыв read-only, а не пустую форму и не голую ошибку.

**КП-3. 422 до завершения класса (D-05, UC-04/A1)**
- **Дано:** бронь клиента в статусе `CONFIRMED`, слот в статусе `SCHEDULED` (момент `startAt + 180 мин` не наступил — D-09).
- **Когда:** клиент отправляет `POST /bookings/{id}/review` (например, по прямой ссылке в обход UI).
- **Тогда:** ответ `422`; `Review` не создан; агрегаты шефа не изменились; при открытии SCR-07 по такой брони — состояние «Отзыв недоступен» с объяснением причины и путём назад.

**КП-4. 422 по отменённой брони (D-05, UC-04/A1)**
- **Дано:** бронь клиента в статусе `CANCELLED_BY_CLIENT` или `CANCELLED_BY_STUDIO`; слот при этом может быть `COMPLETED`.
- **Когда:** клиент отправляет `POST /bookings/{id}/review`.
- **Тогда:** ответ `422`; `Review` не создан; UI — состояние «Отзыв недоступен» с конкретной причиной.

**КП-5. Комментарий виден админу и не виден шефу (Р-07)**
- **Дано:** существует отзыв с непустым `comment`.
- **Когда:** администратор запрашивает `GET /reviews` (в т.ч. с фильтрами `chefId`/`programId`); шеф запрашивает `GET /reviews`.
- **Тогда:** администратор получает `200` со списком `Review`, включая тексты `comment`, от новых к старым; шеф получает `403`; ни в одном экране шефа текст комментария не отображается; публично доступен только агрегат `averageRating`/`reviewsCount` (ФТ-12).

**КП-6. Обязательность обеих оценок (ФТ-11, бриф SCR-07)**
- **Дано:** клиент на SCR-07, выставлена только одна из двух оценок.
- **Когда:** клиент пытается отправить отзыв.
- **Тогда:** кнопка отправки неактивна с подсказкой, чего не хватает (ошибки после нажатия нет); запрос без одной из оценок, отправленный в обход UI, получает `400` (обязательные поля схемы `ReviewCreateRequest`).
