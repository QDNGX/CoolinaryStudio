## 1. Назначение и охват

Модуль реализует единый passwordless-вход по email + одноразовому шестизначному коду для
всех трёх ролей — CLIENT / CHEF / ADMIN (ФТ-13, UC-05). Для нового клиента этот же модуль —
вся его «регистрация»: отдельной анкеты не существует (ФТ-21, Р-01).

**Экраны:**

| Экран | Назначение | Бриф |
|---|---|---|
| SCR-01 | Вход: ввод email | [scr-01-auth-email.md](../analytics/design-brief/briefs/scr-01-auth-email.md) |
| SCR-02 | Вход: ввод одноразового кода (+ шаг «Как к вам обращаться?» для нового клиента) | [scr-02-auth-code.md](../analytics/design-brief/briefs/scr-02-auth-code.md) |

**Эндпоинты** (контракт — [openapi.yaml](../api/openapi.yaml)):

| Операция | operationId | Назначение |
|---|---|---|
| `POST /auth/code/request` | `requestAuthCode` | Запросить одноразовый код на email (шаги 1–2 UC-05) |
| `POST /auth/code/verify` | `verifyAuthCode` | Проверить код и открыть сессию (шаги 3–5 UC-05) |
| `POST /auth/logout` | `logout` | Завершить сессию (инвалидация токена) |
| `PATCH /me` | `updateMe` | Сохранить имя нового клиента одним полем (ФТ-21, Р-12) |

**Закрываемые требования:** ФТ-13, ФТ-14, ФТ-21; UC-05, UC-08 (в части первого входа шефа);
НФТ-04, НФТ-05, НФТ-06; решения Р-01, Р-09, Р-12, Р-16 (п. 1–2); D-08. Полная трассировка —
раздел 9.

**Не входит в модуль:** заведение аккаунта шефа администратором (`POST /chefs`, UC-08/ФТ-14) —
это модуль админки; здесь учитывается только его следствие: шеф/админ существуют в системе
до первого входа (D-08).

## 2. Пользовательские сценарии

Все потоки — конкретизация UC-05 ([use-cases-cooking.md](../analytics/requirements/use-cases-cooking.md))
и брифов SCR-01/SCR-02.

### 2.1. Основной поток — вход существующего пользователя (UC-05, шаги 1–5)

1. Пользователь на SCR-01 вводит email и нажимает «Получить код».
2. Фронт вызывает `POST /auth/code/request`; при `202` — переход на SCR-02
   (email показан на экране, таймер повторной отправки 60 сек запущен).
3. Система отправляет шестизначный код письмом (EML-01; единственный канал — email, НФТ-06).
4. Пользователь вводит код (ручной ввод или вставка из буфера — равноценны, бриф SCR-02).
5. Фронт вызывает `POST /auth/code/verify`; при `200` получает bearer-токен и `CurrentUser`.
6. Пользователь попадает в интерфейс своей роли автоматически, без экрана выбора
   (постусловие UC-05, бриф SCR-02): CLIENT → SCR-03 (расписание), CHEF → SCR-09
   («Мои классы»), ADMIN → SCR-13 (сетка слотов). Роль определяет система по email —
   никаких переключателей роли в UI (бриф SCR-01).

### 2.2. Первый вход нового клиента (ФТ-21, Р-01, Р-12)

1. Шаги 1–5 идентичны п. 2.1 — неизвестный email **не ошибка** и внешне ничем не отличается:
   ответ `202` не раскрывает, существует ли аккаунт (описание `requestAuthCode` в openapi, UC-05).
2. При `200` на `verify` бэкенд молчаливо создаёт `User(role=CLIENT)` (Р-01, ФТ-21)
   и возвращает `isNewUser: true`, `user.name = null`.
3. Фронт показывает ровно один дополнительный шаг — единственное поле
   «Как к вам обращаться?» (Р-12); ответ сохраняется через `PATCH /me { "name": ... }`.
4. После ответа клиент попадает туда же, куда шёл (бриф SCR-02): к прерванной броне,
   если вход начался с неё (Р-16 п. 2), иначе на SCR-03.
5. Никакого явного шага «зарегистрироваться» и слова «регистрация» в UI нет (Р-01, бриф SCR-01).

### 2.3. Просроченный код (UC-05/A1)

1. Пользователь вводит код позже, чем через 10 минут после отправки (НФТ-04/Р-09).
2. `verify` возвращает `400` с `code: CODE_EXPIRED`.
3. SCR-02 переходит в состояние «Код просрочен»: поле ввода недоступно (чтобы не тратить
   попытки на мёртвый код), единственное очевидное действие — «Отправить новый код»
   (бриф SCR-02, состояние «Код просрочен»).

### 2.4. Исчерпание попыток (UC-05/A1, НФТ-04/Р-09)

1. Пользователь 5 раз вводит неверный код. После каждой ошибки (`400 INVALID_CODE`)
   UI показывает мягкое сообщение и остаток попыток; до первой ошибки счётчик скрыт
   (бриф SCR-02, элемент 5).
2. На исчерпании `verify` возвращает `400` с `code: ATTEMPTS_EXCEEDED`.
3. SCR-02 переходит в состояние «Попытки исчерпаны»: ввод закрыт, пути дальше —
   запросить новый код (с учётом паузы 60 сек) или изменить email. Тупика быть не должно (Р-09).

### 2.5. Повторная отправка кода

1. На SCR-02 действие «Отправить новый код» доступно не чаще 1 раза в 60 секунд
   (НФТ-04/Р-09); до истечения паузы кнопка видима, но неактивна, с обратным отсчётом.
2. Повторная отправка = повторный `POST /auth/code/request` с тем же email
   (отдельного эндпоинта в контракте нет).
3. При `202` — подтверждение «новый код уже летит», таймер перезапущен; прежний код
   объявляется недействительным, чтобы не путаться между двумя письмами (бриф SCR-02,
   состояние «Повторная отправка»).
4. Слишком ранний запрос → `429` + `Retry-After`; UI показывает «код уже в пути»
   с обратным отсчётом (бриф SCR-01, состояние «Слишком частые запросы»).

### 2.6. Гостевой сценарий: вход на шаге брони (Р-16 п. 2)

1. Гость смотрит расписание и карточку класса без входа (Р-16 п. 2 — гостевой доступ).
2. При попытке забронировать — переход на SCR-01; контекст прерванной брони сохраняется.
3. После успешного входа (и — для нового клиента — шага имени) пользователь возвращается
   к прерванному бронированию, а не на расписание (Р-16 п. 2, бриф SCR-02 «Уходит на»).
4. Отказ от входа («назад»/закрытие) возвращает туда, откуда пришёл; гостевой просмотр
   расписания остаётся доступен (бриф SCR-01).

### 2.7. Выход

1. Авторизованный пользователь инициирует выход; фронт вызывает `POST /auth/logout`.
2. `204` — токен инвалидирован, фронт удаляет его и возвращает пользователя в гостевой режим.
   На каком экране расположено действие «выйти» — вне охвата этого ТЗ (экраны профиля).

## 3. Интерфейс: экраны и состояния

### 3.1. SCR-01 — Вход: ввод email

Полный бриф: [scr-01-auth-email.md](../analytics/design-brief/briefs/scr-01-auth-email.md).

**Элементы** (по брифу, раздел 4):

| № | Элемент | Требования к реализации |
|---|---|---|
| 1 | Поле ввода email | Единственное поле формы; автофокус; `type="email"`/email-клавиатура на мобильном; инлайн-валидация формата без агрессии |
| 2 | Кнопка «Получить код» | Единственный путь дальше; текст — про код, не про «вход/регистрацию» |
| 3 | Объяснение механики | «Пришлём шестизначный код на почту — пароль не нужен», 1–2 строки |
| 4 | Приветствие студии | Не конкурирует с полем ввода; визуал — свобода дизайнера |
| 5 | Подсказка для команды студии | Ненавязчивая строка внизу: шефы и администратор входят здесь же; отдельной «служебной» двери нет (ФТ-14/D-08). **Важно:** строка статична — сервер не раскрывает, известен ли email (см. БП-08), поэтому UI не может показывать её «только неузнанным» |

Чего на экране **нет** (бриф, раздел 5): поля пароля и ссылки «забыли пароль?» (ФТ-13),
выбора роли, отдельной кнопки «зарегистрироваться» (Р-01).

**Состояния** (по брифу, раздел 6):

| Состояние | Триггер | Поведение |
|---|---|---|
| По умолчанию | Открытие экрана | Пустое поле с автофокусом, кнопка, объяснение механики |
| Загрузка | Отправлен `POST /auth/code/request`, ответа нет | Кнопка в ожидании, повторный клик невозможен, поле заблокировано |
| Невалидный формат | Строка не похожа на email | Мягкая инлайн-подсказка; запрос не отправляется |
| Email не найден / роль не заведена | Email неизвестен студии | Не отдельное визуальное состояние: код отправляется как обычно (`202`), аккаунт клиента появится молча после verify (Р-01); статичная строка-подсказка для незаведённого шефа — «напишите студии» (ФТ-14/D-08, UC-05 примечание) |
| Ошибка отправки | Сеть/сервис недоступны (запрос не дошёл или 5xx) | Человеческое сообщение без технических кодов + «Попробовать ещё раз»; введённый email сохранён |
| Слишком частые запросы | Ответ `429` (повторный запрос раньше 60 сек) | «Код уже в пути» + обратный отсчёт по `Retry-After` (НФТ-04/Р-09) |

### 3.2. SCR-02 — Вход: ввод одноразового кода

Полный бриф: [scr-02-auth-code.md](../analytics/design-brief/briefs/scr-02-auth-code.md).

**Элементы** (по брифу, раздел 4):

| № | Элемент | Требования к реализации |
|---|---|---|
| 1 | Поле ввода кода | Ровно 6 разрядов (ФТ-13); автофокус; цифровая клавиатура на мобильном; вставка всего кода из буфера |
| 2 | Email получателя | «Код отправлен на …» + действие «изменить email» → возврат на SCR-01 с сохранённым адресом |
| 3 | «Отправить новый код» | С обратным отсчётом 60 сек; до истечения паузы видима, но неактивна (НФТ-04/Р-09) |
| 4 | Подсказка про ожидание | «Письмо обычно приходит за минуту, загляните в спам» — мелко, под полем |
| 5 | Счётчик оставшихся попыток | Появляется только после первой ошибки; всего попыток 5 (НФТ-04/Р-09) |
| 6 | Шаг знакомства (только `isNewUser: true`) | Единственное поле «Как к вам обращаться?» после успешного verify (Р-12); сохраняется `PATCH /me` |

Чего на экране **нет** (бриф, раздел 5): выбора/смены роли, обхода кода
(вход по ссылке, «запомнить без кода»), безлимитного перебора.

**Состояния** (по брифу, раздел 6):

| Состояние | Триггер | Поведение |
|---|---|---|
| По умолчанию | Переход с SCR-01 после `202` | Поле кода с автофокусом, адрес получателя, таймер 60 сек, подсказка про спам |
| Загрузка | Отправлен `POST /auth/code/verify` | Ожидание на поле/кнопке; повторный ввод заблокирован, чтобы не жечь попытки случайными кликами |
| Неверный код | `400 INVALID_CODE` | Мягкое сообщение + остаток попыток (из 5); поле очищено/выделено для повторного ввода |
| Код просрочен | `400 CODE_EXPIRED` | Объяснение + единственное действие «Отправить новый код»; поле ввода недоступно |
| Попытки исчерпаны | `400 ATTEMPTS_EXCEEDED` | Ввод закрыт; пути: новый код (с паузой 60 сек) или изменить email; без тона наказания (Р-09) |
| Повторная отправка | Повторный `202` на `request` | «Новый код уже летит», таймер перезапущен, прежний код недействителен |
| Первый вход | `200` + `isNewUser: true` | Шаг «Как к вам обращаться?» (Р-12), затем — туда, куда шёл пользователь |
| Ошибка сети/сервиса | Запрос не выполнен технически (сеть, 5xx) | Введённый код не потерян, действие «попробовать ещё раз»; **не** выглядит как «неверный код» |

**Гостевой сценарий на этом экране (Р-16 п. 2):** если вход начат с шага брони, после
успеха (и шага имени у нового клиента) — возврат именно к прерванному бронированию.
Механизм сохранения контекста (state роутера, параметр `returnTo` и т. п.) — на усмотрение
разработки; требование только к результату.

**Мобильный сценарий (НФТ-01, бриф SCR-02):** переключение в почтовое приложение и обратно
не должно сбрасывать состояние экрана (введённые цифры, таймер, счётчик попыток).

**Свобода дизайнера** (бриф SCR-02, раздел 9): форма поля (6 ячеек или одно поле),
автосабмит после шестой цифры или явная кнопка (каждая проверка расходует попытку),
подача таймера и счётчика, формулировки сообщений.

## 4. Контракт API

Источник истины — [openapi.yaml](../api/openapi.yaml). Все ошибки — схема `Error`
(`code`, `message`, `details?`). Аутентификация ответов verify — bearer JWT
(`securitySchemes.bearerAuth`); паролей нет ни для одной роли (ФТ-13, БТ-06).

### 4.1. `POST /auth/code/request` — запросить код

- **Назначение:** шаги 1–2 UC-05 — отправить шестизначный код на email (EML-01).
- **Security:** без токена (`security: []`).
- **Тело запроса** (`AuthCodeRequest`): `{ "email": "client@example.com" }` — обязательное,
  `format: email`.
- **Успех:** `202` без тела — «код отправлен (или будет отправлен)». Ответ одинаков для
  известного и неизвестного email — существование аккаунта не раскрывается (UC-05, Р-01).

| Код | Условие | Тело | Реакция UI (SCR-01 / SCR-02) |
|---|---|---|---|
| `202` | Запрос принят | — | SCR-01: переход на SCR-02, запуск таймера 60 сек. SCR-02 (повторная отправка): «новый код уже летит», перезапуск таймера |
| `400` | Невалидный запрос (формат email) | `Error` | Инлайн-подсказка у поля email (состояние «Невалидный формат»). В норме недостижимо — фронт валидирует формат до отправки. Конкретное значение `code` в спецификации не задано — на усмотрение разработки |
| `429` | Повторный запрос раньше 60 сек (НФТ-04/Р-09) | `Error` + заголовок `Retry-After` (секунд до возможности повтора) | Состояние «Слишком частые запросы»: «код уже в пути» + обратный отсчёт, длительность отсчёта — из `Retry-After`; кнопка отправки неактивна до нуля |

### 4.2. `POST /auth/code/verify` — подтвердить код, открыть сессию

- **Назначение:** шаги 3–5 UC-05 — проверка соответствия кода, срока действия (10 минут)
  и числа попыток (≤ 5, НФТ-04).
- **Security:** без токена (`security: []`).
- **Тело запроса** (`AuthCodeVerify`): `{ "email": ..., "code": "482913" }`;
  `code` — строка, `pattern: ^[0-9]{6}$`.
- **Успех:** `200`, тело `AuthResult`:
  - `accessToken` — bearer-токен для последующих запросов;
  - `tokenType` — `Bearer`;
  - `isNewUser` — `true`, если аккаунт клиента только что создан молчаливо (ФТ-21, Р-01);
  - `user` — `CurrentUser` (`id`, `role`, `name` — `null` у нового клиента, `email`,
    `enabled`, `createdAt`, профиль роли).

| Код | `code` в теле | Условие | Реакция UI (SCR-02) |
|---|---|---|---|
| `200` | — | Код верен, сессия установлена | Сохранить токен. `isNewUser: false` → в интерфейс роли (CLIENT → SCR-03 / возврат к прерванной броне по Р-16 п. 2; CHEF → SCR-09; ADMIN → SCR-13). `isNewUser: true` → сначала шаг «Как к вам обращаться?» (Р-12), затем туда же |
| `400` | `INVALID_CODE` | Код не совпал; попытка учтена | Состояние «Неверный код»: мягкое сообщение + остаток попыток из 5; поле готово к повторному вводу |
| `400` | `CODE_EXPIRED` | Истекли 10 минут | Состояние «Код просрочен»: ввод недоступен, действие одно — «Отправить новый код» (UC-05/A1) |
| `400` | `ATTEMPTS_EXCEEDED` | Исчерпаны 5 попыток | Состояние «Попытки исчерпаны»: ввод закрыт; новый код (с паузой 60 сек) или смена email (UC-05/A1, Р-09) |

UI обязан различать три причины `400` по полю `code`, а не по HTTP-статусу: у состояний
разные выходы (бриф SCR-02, раздел 7).

### 4.3. `POST /auth/logout` — завершить сессию

- **Назначение:** инвалидация текущего bearer-токена.
- **Security:** требуется токен.

| Код | Условие | Реакция UI |
|---|---|---|
| `204` | Сессия завершена | Удалить токен на клиенте, перейти в гостевой режим (расписание доступно без входа, Р-16 п. 2) |
| `401` | Токен отсутствует/недействителен (`Unauthorized`) | Токен и так мёртв: удалить локально и считать выход состоявшимся (реакция в источниках не описана — на усмотрение разработки, предложенное поведение не противоречит контракту) |

Длительность сессии на контракт не влияет (описание `logout` в openapi) — см. раздел 7.

### 4.4. `PATCH /me` — имя нового клиента (в охвате модуля только этот кейс)

- **Назначение (в рамках модуля):** сохранить ответ на «Как к вам обращаться?» после
  первого входа (ФТ-21, Р-12).
- **Security:** требуется токен (только что полученный из verify).
- **Тело запроса** (`UpdateMeRequest`): `{ "name": "..." }`; `maxLength: 100`.
  Поле `allergyNote` — вне охвата модуля (профиль, ФТ-18). Email через этот метод
  не меняется — email не редактируется никем (Р-16 п. 1).

| Код | Условие | Реакция UI (шаг знакомства на SCR-02) |
|---|---|---|
| `200` | Профиль обновлён, тело `CurrentUser` | Завершить вход: переход в интерфейс роли / к прерванной броне (Р-16 п. 2) |
| `400` | Валидация (`name` длиннее 100 символов и т. п.) | Инлайн-подсказка у поля, значение сохранено для правки |
| `401` | Токен недействителен | Возврат на SCR-01 (сессии нет); поведение в источниках не описано — на усмотрение разработки |

## 5. Бизнес-правила и валидации

| ID | Правило | Источник |
|---|---|---|
| БП-01 | Код — ровно 6 десятичных цифр (`^[0-9]{6}$`); единая схема для всех ролей, паролей не существует | ФТ-13, UC-05, схема `AuthCodeVerify` |
| БП-02 | Срок действия кода — 10 минут с момента отправки; по истечении verify отвечает `CODE_EXPIRED` | НФТ-04, Р-09 |
| БП-03 | Не более 5 попыток ввода кода; неверный ввод учитывается (`INVALID_CODE`), на 6-й — `ATTEMPTS_EXCEEDED`; после исчерпания единственный путь — запросить новый код | НФТ-04, Р-09, UC-05/A1 |
| БП-04 | Повторная отправка кода — не чаще 1 раза в 60 секунд на email; нарушение → `429` + `Retry-After` | НФТ-04, Р-09 |
| БП-05 | Повторная отправка делает прежний код недействительным (одновременно валиден только последний код) | бриф SCR-02, состояние «Повторная отправка» |
| БП-06 | Код хранится как недолговечная запись с TTL, а не постоянная доменная сущность; механизм (таблица с очисткой по расписанию или кэш) — на выбор реализации | НФТ-05 |
| БП-07 | Неизвестный email — не ошибка: обрабатывается как вход нового клиента; аккаунт `User(role=CLIENT)` создаётся молчаливо после успешного verify, ответ содержит `isNewUser: true` | ФТ-21, Р-01, UC-05 |
| БП-08 | Ответ `request` не раскрывает существование аккаунта: `202` и текст одинаковы для известного и неизвестного email | UC-05, описание `requestAuthCode` в openapi |
| БП-09 | Сразу после первого входа система запрашивает имя одним полем «Как к вам обращаться?»; сохраняется `PATCH /me { name }`, `maxLength: 100`; до ответа `user.name = null` | ФТ-21, Р-12, схемы `CurrentUser`/`UpdateMeRequest` |
| БП-10 | Аккаунты CHEF и ADMIN заводятся заранее (шефа — администратор через `POST /chefs`); самостоятельной регистрации шефа нет, экран входа не предлагает «стать шефом» | ФТ-14, D-08, UC-08 |
| БП-11 | Роль определяется системой по email; после verify пользователь попадает в интерфейс своей роли без промежуточного выбора | UC-05 (постусловие), брифы SCR-01/SCR-02 |
| БП-12 | Email не редактируется никем; это идентификатор входа и канал уведомлений; смена — вручную через администратора (вне MVP-интерфейса) | Р-16 п. 1 |
| БП-13 | Единственный канал доставки кода — email; без SMS, соцсетей и прочих внешних каналов | НФТ-06 |
| БП-14 | Гостевой доступ: расписание, карточка слота, каталог и настройки студии читаются без входа; вход требуется на шаге брони, после входа — возврат к прерванной броне | Р-16 п. 2 |
| БП-15 | Валидация формата email на фронте — до отправки запроса (инлайн, без агрессии); бэкенд дублирует проверку (`format: email`, ответ `400`) | бриф SCR-01, схема `AuthCodeRequest` |
| БП-16 | Счётчик попыток относится к действующему коду: запрос нового кода после `ATTEMPTS_EXCEEDED` даёт новые попытки (иначе «запросить новый код» не был бы выходом из тупика, Р-09). Точная гранулярность счётчика в источниках не формализована — детали на усмотрение разработки при сохранении инварианта «≤ 5 попыток на код» | НФТ-04, Р-09 (производное) |

## 7. Вне скоупа

- **Пароли** — не существуют ни для одной роли: ни полей, ни ссылок «забыли пароль?»,
  ни хранения (ФТ-13, БТ-06).
- **SMS и внешние каналы** — только email; без платного SMS-шлюза и регистрации
  отправителя (НФТ-06).
- **Вход через соцсети / magic-link / «запомнить без кода»** — альтернатив коду
  не проектировать (бриф SCR-02, раздел 5).
- **Самостоятельная регистрация шефа** — только администратор через `POST /chefs`
  (ФТ-14, D-08, UC-08); кейс «email нового шефа уже занят клиентом» в MVP решается
  вручную другим email (Р-16, открытый пункт SCR-16) — вне этого модуля.
- **Редактирование email** — не предусмотрено ни для кого (Р-16 п. 1).
- **Длительность сессии / срок жизни токена** — открытый вопрос уровня интерфейса
  (бриф SCR-02, раздел 9; описание `logout` в openapi: на контракт не влияет).
  **Фиксируется как решение разработки:** значение выбирает и документирует команда
  разработки при реализации; UI до этого решения не даёт обещаний вида «входить нужно
  будет редко».

## 8. Критерии приёмки

Формат: Дано / Когда / Тогда. Критерии покрывают все потоки раздела 2 и все коды ошибок
раздела 4.

**КП-01 — вход существующего пользователя (UC-05, ФТ-13)**
Дано: пользователь с ролью CLIENT существует; открыт SCR-01.
Когда: он вводит свой email, получает `202`, вводит присланный шестизначный код в течение 10 минут.
Тогда: `verify` возвращает `200` с `accessToken`, `tokenType: Bearer`, `isNewUser: false`;
пользователь оказывается на SCR-03 без промежуточных экранов выбора роли.

**КП-02 — роль определяется по email (UC-05 постусловие)**
Дано: заведённые пользователи CHEF и ADMIN.
Когда: каждый проходит вход по своему email.
Тогда: шеф попадает на SCR-09, администратор — на SCR-13; переключателей роли в UI нет.

**КП-03 — первый вход нового клиента (ФТ-21, Р-01, Р-12)**
Дано: email отсутствует в системе; открыт SCR-01.
Когда: пользователь запрашивает код (`202` — ответ неотличим от ответа для известного email)
и подтверждает его.
Тогда: `verify` возвращает `200` с `isNewUser: true`, `user.role: CLIENT`, `user.name: null`;
UI показывает единственное поле «Как к вам обращаться?»; после `PATCH /me { name }` → `200`
пользователь попадает туда, куда шёл; слово «регистрация» нигде не показано.

**КП-04 — неразглашение существования аккаунта (UC-05, БП-08)**
Дано: один известный и один неизвестный системе email.
Когда: для каждого выполняется `POST /auth/code/request`.
Тогда: оба ответа — `202` без тела; по ответу и UI нельзя определить, есть ли аккаунт.

**КП-05 — неверный код и счётчик попыток (НФТ-04, Р-09)**
Дано: код отправлен; открыт SCR-02, ошибок ещё не было (счётчик попыток скрыт).
Когда: пользователь вводит неверный код.
Тогда: ответ `400` с `code: INVALID_CODE`; UI показывает мягкое сообщение и остаток
попыток (4 из 5); поле готово к повторному вводу.

**КП-06 — исчерпание попыток (НФТ-04, Р-09, UC-05/A1)**
Дано: по действующему коду уже сделано 4 неверных попытки.
Когда: пользователь вводит неверный код в 5-й раз (и любой код после).
Тогда: ответ `400` с `code: ATTEMPTS_EXCEEDED`; ввод кода закрыт; доступны «Отправить
новый код» (с учётом паузы 60 сек) и «изменить email»; после получения нового кода
верный ввод завершается `200`.

**КП-07 — просроченный код (НФТ-04, UC-05/A1)**
Дано: код отправлен более 10 минут назад.
Когда: пользователь вводит этот код (даже верный).
Тогда: ответ `400` с `code: CODE_EXPIRED`; поле ввода недоступно; единственное действие —
«Отправить новый код».

**КП-08 — троттлинг повторной отправки (НФТ-04, Р-09)**
Дано: код отправлен менее 60 секунд назад.
Когда: выполняется повторный `POST /auth/code/request` с тем же email.
Тогда: ответ `429` с заголовком `Retry-After`; UI показывает «код уже в пути» и обратный
отсчёт; по истечении отсчёта повторный запрос возвращает `202`.

**КП-09 — повторная отправка гасит старый код (БП-05)**
Дано: получено два письма с кодами (после повторной отправки).
Когда: пользователь вводит код из первого письма.
Тогда: вход не выполняется (ответ — ошибка `400`); ввод кода из второго письма
завершается `200`.

**КП-10 — TTL-хранение кода (НФТ-05)**
Дано: код создан.
Когда: истёк его срок действия (плюс период очистки механизма хранения).
Тогда: запись кода не существует как постоянная доменная сущность — она удалена
по TTL/расписанию (проверяется на уровне реализации хранилища).

**КП-11 — возврат к прерванной броне (Р-16 п. 2)**
Дано: гость без сессии начинает бронирование слота и переадресован на SCR-01.
Когда: он успешно проходит вход (для нового клиента — включая шаг имени).
Тогда: он возвращается к прерванному бронированию того же слота, а не на общий экран
расписания.

**КП-12 — гостевой доступ не требует входа (Р-16 п. 2)**
Дано: пользователь без сессии.
Когда: он открывает расписание и карточку слота.
Тогда: контент доступен без SCR-01; экран входа появляется только на шаге брони или по
явному действию «Войти».

**КП-13 — двойной сабмит (брифы, состояния «Загрузка»)**
Дано: запрос `request` или `verify` отправлен, ответ не получен.
Когда: пользователь повторно нажимает кнопку/вводит символы.
Тогда: повторный запрос не уходит; на SCR-02 попытки не расходуются случайными кликами.

**КП-14 — сетевая ошибка не выглядит как неверный код (бриф SCR-02)**
Дано: открыт SCR-02, введён код.
Когда: запрос `verify` завершается сетевой ошибкой или 5xx.
Тогда: UI показывает техническое (но человеческое) сообщение с действием «попробовать
ещё раз»; введённый код сохранён; сообщение «неверный код» и списание попытки в UI
не показываются.

**КП-15 — незаведённый шеф (ФТ-14, D-08, UC-05 примечание)**
Дано: шеф, которого администратор не завёл, открывает SCR-01.
Когда: он читает экран.
Тогда: он видит подсказку, что команда студии входит здесь же, и строку «Ждёте доступ
шефа, а вас не узнали? Напишите студии»; экран не предлагает «стать шефом» и не содержит
отдельного служебного входа.

**КП-16 — валидация email на фронте (бриф SCR-01)**
Дано: открыт SCR-01.
Когда: введена строка, не похожая на email, и нажата кнопка.
Тогда: запрос не отправляется; у поля — мягкая инлайн-подсказка.

**КП-17 — выход (контракт `logout`)**
Дано: активная сессия.
Когда: пользователь выходит.
Тогда: `POST /auth/logout` возвращает `204`; токен удалён на клиенте; повторный запрос
защищённого ресурса возвращает `401`; гостевые экраны остаются доступны.

**КП-18 — лимит имени (`PATCH /me`)**
Дано: шаг «Как к вам обращаться?» после первого входа.
Когда: введено имя длиннее 100 символов.
Тогда: ответ `400` (ValidationError); UI показывает подсказку, введённое значение
сохранено для правки.
