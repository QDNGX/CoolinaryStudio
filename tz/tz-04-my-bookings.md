# ТЗ-04 — Мои брони и отмена брони

> Модуль клиента веб-приложения «Шеф-стол». Экран **SCR-06** «Мои брони».
> Источники: [бриф SCR-06](../analytics/design-brief/briefs/scr-06-client-my-bookings.md),
> [openapi.yaml](../api/openapi.yaml) (операции `listMyBookings`, `cancelBooking`, `getBookingReview`),
> [UC-02](../analytics/requirements/use-cases-cooking.md#uc-02--клиент-отменяет-бронь),
> [ФТ-08, ФТ-09, ФТ-19](../analytics/requirements/functional-requirements-cooking.md),
> [D-01, D-02](../analytics/requirements/constraints-and-assumptions-cooking.md),
> [Р-06, Р-14, Р-16](../analytics/design-brief/design-decisions-cooking.md).
> ТЗ ничего не добавляет к источникам; всё, что источниками не зафиксировано, помечено
> «на усмотрение разработки».

## 1. Назначение и охват

Модуль реализует экран **SCR-06 «Мои брони»** (роль CLIENT, адаптивный веб — десктоп и мобильный
браузер, приоритет Must):

- список всех броней клиента — **вся история без обрезания по давности** (Р-16, п. 4);
  предстоящие сверху (ближайшая — акцентно), прошедшие визуально приглушены;
- **отмена подтверждённой брони** с диалогом подтверждения, который различает обычную и «позднюю»
  отмену **до** финального нажатия (D-01, Р-06);
- отображение всех пяти статусов брони по жизненному циклу ФТ-19;
- переход к форме отзыва (SCR-07) с завершённой брони без отзыва и **просмотр уже отправленного
  отзыва read-only** (Р-16, п. 6).

**Вне охвата модуля:** создание брони (SCR-05, ТЗ бронирования), форма отзыва SCR-07 (отдельный
модуль; здесь — только переходы), перенос брони (в продукте отсутствует — бриф SCR-06, раздел 5:
не показывать ничего похожего на «изменить»).

## 2. Пользовательские сценарии

### 2.1. Просмотр списка броней

Клиент открывает «Мои брони» из навигации, после оформления брони (SCR-05), из письма-напоминания
или после отправки отзыва (SCR-07). Видит предстоящие записи (ближайшая — акцентно: дата и время
крупно, читается с одного взгляда на телефоне) и историю целиком (Р-16, п. 4). Состав данных
каждой записи — раздел 3.

### 2.2. Отмена заранее (бесплатная)

До начала класса **6 часов и более** (D-01). Клиент нажимает «Отменить» на предстоящей записи →
спокойный диалог подтверждения: что отменяем (класс, дата, время) + подтверждение, что отмена
бесплатна и без последствий (бриф, раздел 6) → подтверждение → `POST /bookings/{id}/cancel` →
список обновлён, статус записи `CANCELLED_BY_CLIENT`. Счётчик нарушений не растёт (ФТ-08, ФТ-09).

### 2.3. Поздняя отмена (< 6 часов до начала)

До начала класса **меньше 6 часов, либо класс уже начался** (D-01; бриф, раздел 7). Диалог
подтверждения показывает честное предупреждение **до** финального нажатия (Р-06):

- отмена возможна, но будет считаться поздней и засчитается как нарушение;
- текущее положение клиента: **«поздних отмен: N из 3»** (Р-06; значение — `lateCancelCount`
  из `GET /me`, `clientProfile`);
- если эта отмена станет **третьим** нарушением — предупреждение усилено конкретикой: новые записи
  станут недоступны на 7 дней, с датой, когда возможность вернётся (D-02; бриф, раздел 6).

Тон — уважительная прямота, без запугивания и канцелярита (бриф, раздел 8). Только после
подтверждения — вызов API; ответ содержит `isLateCancellation` (ФТ-08).

### 2.4. Просмотр брони, отменённой студией

Статус «отменена студией» + **причина отмены словами студии** — та же, что пришла в письме
(ФТ-04, ФТ-15; бриф, разделы 4 и 6). Нарушением клиента это не является и никак не окрашивается
негативно по отношению к нему. Действий над записью нет — статус конечный (UC-02, бриф раздел 5).

### 2.5. Просмотр неявки

Статус «неявка» + **спокойное пояснение рядом**: неявка считается нарушением и приближает паузу
в бронированиях (Р-14, в связке с Р-06 — блокировка не должна быть сюрпризом). Без обвинений —
факт и его значение. Оспорить неявку на этом экране нельзя (бриф, раздел 7 / ФТ-19; снятие —
только администратор, вне этого модуля).

### 2.6. Переход к отзыву и просмотр отправленного отзыва

- На завершённой записи (`COMPLETED`), по которой отзыва ещё нет (`hasReview = false`) — заметный
  переход «Оставить отзыв» → форма SCR-07 (условия доступности — D-05, ФТ-11; вне охвата ТЗ).
- Если отзыв оставлен (`hasReview = true`) — спокойная отметка, с которой отзыв можно открыть
  и перечитать **read-only** (Р-16, п. 6): данные — `GET /bookings/{bookingId}/review`.
  Редактирование не предусмотрено.
- Переход к отзыву не показывается на отменённых записях и неявках; «мёртвых» кнопок с ошибкой
  после нажатия быть не должно (бриф, раздел 5).

## 3. Интерфейс: экран и состояния

### 3.1. Элементы экрана (бриф SCR-06, раздел 4)

| Приоритет | Элемент | Данные (поле API) |
|---|---|---|
| 1 | Ближайшая предстоящая запись | Дата и время начала (`slot.startAt` — крупно), название программы (`slot.programTitle`), шеф (`slot.chef.name`), экипировка (`equipmentChoice`: своя / прокат с зафиксированной ценой `rentalPriceSnapshot`) |
| 2 | Остальные предстоящие записи | Тот же состав; отсортированы по времени начала; каждая — самостоятельная карточка/строка с собственным действием отмены |
| 3 | Статус записи | `status` — различим мгновенно, не только цветом; у «отменена студией» обязательно видна причина |
| 4 | Прошедшие записи | Завершённые, отменённые, неявки — история целиком (Р-16, п. 4); визуально приглушены; на завершённой без отзыва — «Оставить отзыв», с отзывом — отметка с открытием read-only |
| 5 | Путь к новым записям | Постоянный, ненавязчивый переход в расписание (SCR-03) |

Причина отмены студией: поле `slot.cancellationReason` входит в `SlotSummary` внутри `Booking`
(заполнено только при `CANCELLED_BY_STUDIO`) — список показывает причину без дополнительных
запросов; требование показывать причину на этом экране — ФТ-04/ФТ-15, бриф раздел 7.

### 3.2. Отображение статусов брони (ФТ-19)

| `Booking.status` | Отображение |
|---|---|
| `CONFIRMED` | Подтверждена; доступно действие «Отменить» (единственный статус с действием отмены — UC-02) |
| `CANCELLED_BY_CLIENT` | Отменена мной; действий нет |
| `CANCELLED_BY_STUDIO` | Отменена студией + причина; без негатива к клиенту; действий нет |
| `COMPLETED` | Завершена; переход к отзыву (`hasReview = false`) либо просмотр отзыва read-only (`hasReview = true`) |
| `NO_SHOW` | Неявка + спокойное пояснение о нарушении (Р-14); действий нет |

Статусы «отменена», «завершена», «неявка» — конечные, действий над ними нет (UC-02, бриф раздел 5).
Визуальный язык статусов (цвет, иконки, бейджи) — решает дизайнер при условии различимости не
только цветом (бриф, раздел 9).

### 3.3. Состояния экрана (бриф SCR-06, раздел 6)

| Состояние | Когда возникает | Что видит пользователь |
|---|---|---|
| По умолчанию | Есть хотя бы одна запись | Предстоящие сверху (ближайшая — акцентно), прошедшие ниже или на отдельной вкладке (компоновка — за дизайнером) |
| Загрузка | `GET /me/bookings` выполняется | Скелетон/плейсхолдер в форме будущих карточек, без прыжков вёрстки |
| Пусто | Ни одной записи (ответ — пустой массив) | «Вы пока никуда не записаны» + заметный переход в расписание; тёплое приглашение, не упрёк |
| Пусто (только история) | Предстоящих нет, прошедшие есть | Блок предстоящих с мягким приглашением записаться; история остаётся видимой |
| Ошибка | Список не загрузился | «Не получилось загрузить записи» + повторить; без технического жаргона |
| Подтверждение обычной отмены | До начала ≥ 6 ч (D-01) | Спокойный диалог: класс, дата, время + «отмена бесплатна и без последствий» |
| Подтверждение поздней отмены | До начала < 6 ч или класс начался | Предупреждение до подтверждения + «поздних отмен: N из 3» (Р-06) |
| Поздняя отмена — третье нарушение | Отмена станет 3-м нарушением | То же + «новые записи недоступны на 7 дней» с датой возврата возможности (D-02) |
| Отмена не удалась | Сбой или запись уже изменилась (409 и др., раздел 4.2) | Понятное объяснение + актуальное состояние записи после обновления; без двусмысленности «отменилось или нет» |
| Запись отменена студией | Каскадная отмена слота (UC-03) | Статус + причина словами студии |
| Запись — неявка | Шеф отметил `NO_SHOW` | Статус + спокойное пояснение (Р-14) |

### 3.4. Счётчик «поздних отмен: N из 3» (Р-06)

- **Обязательно** показывается в предупреждении о поздней отмене — до подтверждения.
- Источник значения: `GET /me` → `clientProfile.lateCancelCount` (readOnly, меняет только
  система — D-02).
- Постоянный показ счётчика где-то на экране SCR-06 — на усмотрение дизайнера (бриф, раздел 9;
  в профиле SCR-08 он показывается по Р-06, вне охвата этого ТЗ).
- Индикатор на карточке «бесплатная отмена доступна до ЧЧ:ММ» — рекомендуется, решение
  за дизайнером (бриф, раздел 9).

## 4. Контракт API

Все вызовы — с bearer-токеном (`POST /auth/code/verify`); базовые определения —
[../api/openapi.yaml](../api/openapi.yaml).

### 4.1. `GET /me/bookings` — список броней (operationId `listMyBookings`)

Ответ `200` — массив `Booking`, **от новых к старым**, вся история без обрезания (Р-16).
Группировка «предстоящие/прошедшие» и сортировка предстоящих по времени начала выполняются
на клиенте — на усмотрение разработки (API отдаёт единый список).

Поля схемы `Booking` и их использование на экране:

| Поле | Где показывается |
|---|---|
| `id` | Идентификатор для `POST /bookings/{id}/cancel` и `GET /bookings/{id}/review`; в UI не показывается |
| `slot` (`SlotSummary`) | `startAt` — дата/время (и вход в расчёт порога 6 ч); `programTitle` — название; `chef` — имя шефа; `slot.id` — переход на карточку класса SCR-04 |
| `status` (`BookingStatus`) | Статус записи (раздел 3.2) |
| `equipmentChoice` | «Своя экипировка» / «Прокат» |
| `rentalPriceSnapshot` | Зафиксированная цена проката (D-06); `null` при `OWN` — не показывается |
| `createdAt` | Показ на экране источниками не зафиксирован — на усмотрение разработки |
| `cancelledAt` | Момент отмены у отменённых записей; показ — на усмотрение разработки |
| `isLateCancellation` | Признак поздней отмены у `CANCELLED_BY_CLIENT` (readOnly, вычисляет система — D-01); показ отметки в истории — на усмотрение разработки |
| `hasReview` | Переключатель «Оставить отзыв» / «отзыв оставлен, открыть read-only» (D-05, Р-16) |

Ошибки: `401` → на вход (passwordless, ФТ-13); `403` → операция недоступна роли (для CLIENT
в штатных сценариях не ожидается; показать общее сообщение об ошибке — на усмотрение разработки).

### 4.2. `POST /bookings/{bookingId}/cancel` — отмена брони (operationId `cancelBooking`)

Тела запроса нет. Система фиксирует `cancelledAt` и вычисляет
`isLateCancellation = slot.startAt − cancelledAt < 6ч` (D-01, ФТ-08). Поздняя отмена увеличивает
`lateCancelCount`; на 3-м нарушении — блок на 7 дней (ФТ-09, D-02). Всё это делает **бэкенд**;
клиентская часть только предупреждает заранее.

**Предупреждение о поздней отмене вычисляется на клиенте** из `slot.startAt` относительно
текущего момента — **до вызова API**, при открытии диалога (порог 6 часов — D-01; граница
считается от момента отмены, поэтому текст формируется на момент открытия диалога — бриф,
раздел 7). Итоговую классификацию отмены определяет ответ сервера (`isLateCancellation`).

Реакция UI на коды ответа:

| Код | Смысл (по openapi.yaml) | Реакция UI |
|---|---|---|
| `200` | Бронь отменена (`CANCELLED_BY_CLIENT`); в ответе `Booking` с `isLateCancellation` | Закрыть диалог, обновить запись в списке. Если `isLateCancellation = true` — отмена засчитана как поздняя (клиент был предупреждён до подтверждения — Р-06); форма подтверждающего сообщения — на усмотрение разработки. Расхождение прогноза клиента с ответом сервера (пограничный момент) — ответ сервера первичен |
| `401` | Не аутентифицирован | Переход на вход (ФТ-13) |
| `403` | Бронь принадлежит другому клиенту | В штатном UI недостижимо (список — только свои брони); показать сообщение об ошибке и обновить список — на усмотрение разработки |
| `404` | Бронь не найдена | Сообщение об ошибке + обновление списка (`GET /me/bookings`) |
| `409` | Бронь не в статусе `CONFIRMED` — уже отменена (клиентом или каскадно студией) или класс завершён | Состояние «Отмена не удалась» (раздел 3.3): понятное объяснение + перезапрос списка и показ актуального статуса записи; без двусмысленности «отменилось или нет» |

### 4.3. `GET /bookings/{bookingId}/review` — просмотр отправленного отзыва (operationId `getBookingReview`)

Вызывается при открытии отзыва с записи, где `hasReview = true` (Р-16, п. 6 — read-only,
редактирование не предусмотрено). Ответ `200` — схема `Review`: `chefRating`, `programRating`,
`comment` (собственный комментарий клиента), `createdAt`. Форма показа (модальное окно /
отдельный блок) — на усмотрение разработки/дизайнера.

Ошибки: `401` → на вход; `403` (чужая бронь) — в штатном UI недостижимо; `404` — отзыв не найден:
сообщение об ошибке + обновление списка (расхождение с `hasReview`) — на усмотрение разработки.

## 5. Бизнес-правила

| ID | Правило | Проекция на модуль |
|---|---|---|
| D-01 | Отмена бесплатна при ≥ 6 ч до начала слота; иначе поздняя (`isLateCancellation`) | Диалог подтверждения обязан различать оба случая **до** финального нажатия; порог считается на клиенте из `slot.startAt` на момент открытия диалога (раздел 4.2); клиент не должен узнавать о «поздней» отмене постфактум. Отмена после начала класса — тоже поздняя |
| D-02 | 3 поздние отмены/неявки → блок новых броней на 7 дней (`blockedUntil`), затем блок и счётчик сбрасываются автоматически | Предупреждение показывает «поздних отмен: N из 3» заранее; при 3-м нарушении — прямо называет последствие и дату возврата возможности. Сам инкремент, блок и автосброс выполняет бэкенд (ФТ-09) — модуль их не дублирует |
| ФТ-08 | Клиент отменяет свою бронь; система вычисляет `isLateCancellation = slot.startAt − cancelledAt < 6ч` | Реализуется вызовом `POST /bookings/{id}/cancel`; вычисление — на сервере |
| ФТ-09 | Инкремент `lateCancelCount` при поздней отмене/неявке; при ≥ 3 — `blockedUntil = now() + 7 дней`, блок новых броней, затем сброс счётчика | Только серверная логика; модуль отражает её честными предупреждениями (Р-06) и пояснением у неявки (Р-14) |
| Р-06 | Блокировка не должна быть сюрпризом | Счётчик в предупреждении о поздней отмене — обязательно; неявка — с пояснением (Р-14); дата разблокировки — при 3-м нарушении |
| UC-02 | Отменить можно только `CONFIRMED`; остальные статусы конечные | Действие «Отменить» есть только у `CONFIRMED`; на конечных статусах действий нет |

## 6. Краевые случаи

| Случай | Поведение |
|---|---|
| Отмена, когда класс уже начался, но не завершился | Отмена возможна и считается **поздней** (D-01, бриф раздел 7: «или класс уже начался»); диалог показывает предупреждение поздней отмены |
| Отмена, когда класс уже завершился (слот автозавершён, бронь → `COMPLETED`) | Сервер отвечает `409` (бронь не `CONFIRMED`); UI — состояние «Отмена не удалась»: объяснение + перезапрос списка, запись показана как завершённая. Если список уже обновлён — кнопки отмены на `COMPLETED` нет вовсе |
| Двойной клик / повторная отправка отмены | Первый запрос — `200`, повторный — `409`. UI обязан не допускать двусмысленности «отменилось или нет» (бриф, раздел 6): блокировка кнопки подтверждения на время запроса — на усмотрение разработки; при полученном `409` после собственного `200` — просто актуализировать список |
| Бронь отменена студией, пока открыт список | Попытка отмены вернёт `409`; UI показывает объяснение и актуальное состояние — статус «отменена студией» с причиной (состояние «Отмена не удалась», бриф раздел 6). Механизм актуализации (перезапрос после ошибки; фоновое обновление) — на усмотрение разработки |
| Диалог открыт на границе 6 часов | Текст предупреждения формируется на момент **открытия** диалога (бриф, раздел 7); итоговая классификация — по `cancelledAt` на сервере (D-01). Возможное расхождение (клиент открыл диалог до порога, подтвердил после) UI разрешает по ответу сервера (`isLateCancellation`); дополнительная перепроверка порога перед отправкой — на усмотрение разработки |
| `hasReview = true`, но `GET .../review` вернул `404` | Рассинхронизация данных; сообщение об ошибке + обновление списка — на усмотрение разработки (источниками не описано) |

## 7. Критерии приёмки

**КП-1. Ранняя отмена без инкремента счётчика**
Дано: клиент с бронью `CONFIRMED`, до `slot.startAt` ≥ 6 часов, `lateCancelCount = 0`.
Когда: клиент открывает диалог отмены и подтверждает.
Тогда: диалог до подтверждения сообщает, что отмена бесплатна и без последствий; после `200`
запись в списке имеет статус «отменена мной»; в ответе `isLateCancellation = false`;
`lateCancelCount` в `GET /me` остался `0`.

**КП-2. Поздняя отмена с инкрементом**
Дано: клиент с бронью `CONFIRMED`, до `slot.startAt` < 6 часов, `lateCancelCount = 1`.
Когда: клиент открывает диалог отмены.
Тогда: **до** кнопки подтверждения диалог предупреждает, что отмена будет считаться поздней и
засчитается как нарушение, и показывает «поздних отмен: 1 из 3» (Р-06). Когда клиент подтверждает —
ответ `200` с `isLateCancellation = true`, статус записи «отменена мной», `lateCancelCount` = 2.

**КП-3. Третья поздняя отмена → блок на 7 дней**
Дано: клиент с `lateCancelCount = 2` и бронью `CONFIRMED`, до начала < 6 часов.
Когда: клиент открывает диалог отмены.
Тогда: предупреждение усилено: после этой отмены новые записи станут недоступны на 7 дней,
с датой, когда возможность вернётся (D-02). Когда клиент подтверждает — `200`,
`lateCancelCount` = 3, в `GET /me` заполнен `blockedUntil` (≈ now() + 7 дней).

**КП-4. Предупреждение показано до подтверждения**
Дано: любая бронь `CONFIRMED`, до начала < 6 часов.
Когда: клиент нажимает «Отменить».
Тогда: запрос `POST /bookings/{id}/cancel` **не отправляется** до подтверждения в диалоге;
предупреждение о поздней отмене и счётчик «N из 3» видимы в диалоге до финальной кнопки;
при закрытии диалога без подтверждения бронь остаётся `CONFIRMED`.

**КП-5. Статусы отображаются корректно**
Дано: у клиента по одной брони в каждом из статусов `CONFIRMED`, `CANCELLED_BY_CLIENT`,
`CANCELLED_BY_STUDIO`, `COMPLETED` (с отзывом и без), `NO_SHOW`.
Когда: клиент открывает «Мои брони».
Тогда: все записи присутствуют (история без обрезания — Р-16); статусы различимы мгновенно и не
только цветом; действие «Отменить» есть только у `CONFIRMED`; у «отменена студией» видна причина;
у «неявки» — спокойное пояснение о нарушении (Р-14); у завершённой без отзыва — переход
«Оставить отзыв», у завершённой с отзывом — открытие отзыва read-only без возможности
редактирования (Р-16), на отменённых и неявке перехода к отзыву нет.

**КП-6. Отмена не удалась — без двусмысленности**
Дано: список открыт; бронь тем временем каскадно отменена студией (UC-03).
Когда: клиент подтверждает отмену этой брони.
Тогда: сервер отвечает `409`; UI показывает понятное объяснение и актуальное состояние записи —
«отменена студией» с причиной; у клиента нет сомнения, отменилась запись или нет.
