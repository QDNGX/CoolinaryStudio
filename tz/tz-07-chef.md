# ТЗ-07 — Кабинет шефа: мои классы и участники

> Модуль веб-приложения «Шеф-стол» (Spring Boot + веб-фронт). Роль — `CHEF`.
> Источники: [SCR-09](../analytics/design-brief/briefs/scr-09-chef-my-classes.md),
> [SCR-10](../analytics/design-brief/briefs/scr-10-chef-class-attendance.md),
> [openapi.yaml](../api/openapi.yaml),
> [UC-06](../analytics/requirements/use-cases-cooking.md),
> [ФТ](../analytics/requirements/functional-requirements-cooking.md),
> [D-правила](../analytics/requirements/constraints-and-assumptions-cooking.md),
> [журнал решений Р-…](../analytics/design-brief/design-decisions-cooking.md).
> Всё нижесказанное имеет ID источника; места, где источники молчат, помечены **[Пробел]**.

## 1. Назначение и охват

Модуль реализует рабочее место шефа — **ровно два экрана** (интерфейс шефа зафиксирован
из двух экранов решением Р-05, других «своих» экранов у роли нет):

| Экран | Назначение | Источник |
|---|---|---|
| **SCR-09 «Мои классы»** | Расписание текущего шефа: предстоящие и прошедшие классы; история прошедших показывается **вся, без обрезания по давности** (Р-16, п. 4). Домашний экран роли после входа (из SCR-02). Единственный переход — в SCR-10 | [SCR-09](../analytics/design-brief/briefs/scr-09-chef-my-classes.md), Р-16 |
| **SCR-10 «Участники класса»** | Список участников конкретного класса с заметками об аллергии (ФТ-18) и экипировкой (ФТ-06); единственная точка проставления неявки `NO_SHOW` (ФТ-10, UC-06) | [SCR-10](../analytics/design-brief/briefs/scr-10-chef-class-attendance.md) |

Вне охвата модуля: управление расписанием (только ADMIN — ФТ-03, UC-07), редактирование
профиля/фото/био шефа (ADMIN, Р-05), снятие отметки `NO_SHOW` (ADMIN, D-11/ФТ-22 — экран
SCR-13), тексты отзывов (только ADMIN — Р-07).

Контекст использования (SCR-09, разд. 1; SCR-10, разд. 1; НФТ-01): мобильный браузер на кухне,
занятые/мокрые руки — крупные цели касания, минимум шагов; десктоп вторичен.

## 2. Пользовательские сценарии

### 2.1. Просмотр своего расписания (SCR-09, предусловие UC-06)

Шеф входит по passwordless-коду (ФТ-13, UC-05; аккаунт заведён администратором — ФТ-14, D-08,
UC-08) и попадает на SCR-09. Видит ближайший класс крупно (программа, дата/время начала,
«записано N из M»), ниже — остальные предстоящие от ближайшего к дальнему; прошедшие — в
отдельной вкладке/секции, вся история (Р-16). Класс переходит из предстоящих в прошедшие
автоматически по `startAt + 180 мин` (D-09, Р-10, ФТ-23) — никаких кнопок «завершить».
Расписание для шефа строго read-only (ФТ-03/UC-07): создание, перенос, редактирование и отмена
классов недоступны и не должны выглядеть доступными (SCR-09, разд. 5).

### 2.2. Открытие списка участников (SCR-10, ФТ-18)

Касание по любому классу (предстоящему или прошедшему) открывает SCR-10: имя каждого
участника, заметка об аллергии — **только если заполнена** (ФТ-18: необязательное поле профиля
клиента; отсутствие заметки ≠ отсутствие аллергии, экран не должен утверждать обратное —
SCR-10, разд. 7), экипировка «Своя»/«Прокат» (ФТ-06, шеф видит как данность) и сводка
«Прокатных наборов подготовить: N» (SCR-10, разд. 4).

### 2.3. Отметка неявки (UC-06, ФТ-10, Р-13)

Действие доступно **с момента `slot.startAt` и без крайнего срока** (Р-13); **до начала класса
недоступно** — кнопки видимы, но выключены, с подсказкой «Отметить неявку можно после начала
класса» (SCR-10, разд. 5–6). Никаких таймеров «успейте до…» (SCR-10, разд. 7). Действие
двухшаговое всегда: касание «Не пришёл» → диалог подтверждения с именем участника и честным
предупреждением о последствиях для клиента (влияет на возможность бронировать — D-02) →
явное подтверждение; кнопки диалога разнесены (SCR-10, разд. 5). Массовой отметки «отметить
всех» и одношаговых свайпов нет (SCR-10, разд. 7 — ФТ-09/D-02). Действия «пришёл» нет: по
умолчанию все считаются пришедшими, брони пришедших завершаются автоматически вместе со слотом
(Р-10, D-09, UC-10).

### 2.4. Ошибочная отметка → снятие только администратором (D-11, Р-02)

У шефа **обратного действия нет** — снять `NO_SHOW` может только администратор (D-11, ФТ-22,
UC-06/A1: бронь → `COMPLETED`, счётчик −1, блокировка пересчитывается). Это надо **явно показать
в UI**: после подтверждения строка меняется «навсегда» (никакого вида переключателя, который
тянет нажать ещё раз), действие исчезает, уместна тихая подсказка «ошиблись — обратитесь к
администратору» (SCR-10, разд. 5–6, Р-02). Вся защита от ошибки сосредоточена **до** нажатия
(диалог подтверждения), после — только путь через администратора.

## 3. Интерфейс: экраны и состояния

### 3.1. SCR-09 «Мои классы» — элементы

По брифу SCR-09, разд. 4 (приоритет = порядок важности):

| # | Элемент | Содержимое | Правила |
|---|---|---|---|
| 1 | Ближайший класс | Программа, дата/время начала, «записано N из M» | Самый крупный элемент; класс «сегодня» помечается явно |
| 2 | Список предстоящих | Те же данные по каждому классу | Сортировка от ближайшего; вся строка — цель касания |
| 3 | Счётчик записанных | «N из M» на каждом классе | Главная рабочая цифра шефа; полный класс можно акцентировать |
| 4 | Прошедшие классы | Тот же список завершённых; **вся история** (Р-16) | Отдельная вкладка/секция; попадание — автоматически через 3 ч после начала (Р-10/D-09) |
| 5 | Пометка «отменён студией» | Класс остаётся виден с явной пометкой | Приглушить, но не прятать (ФТ-19) |
| 6 | Мой рейтинг | Средняя оценка и число отзывов, напр. «4,8 · 26 отзывов» — **только агрегат** (ФТ-12, Р-07) | См. 3.3 |
| 7 | Служебное | Имя шефа, выход из аккаунта | Минимально |

Состояния (SCR-09, разд. 6): по умолчанию / загрузка (скелетон) / пусто-предстоящие («Пока нет
назначенных классов» + пояснение, что расписание формирует студия, без кнопок-обманок) /
пусто-прошедшие («Здесь появится история ваших классов») / ошибка (сообщение без жаргона +
крупная «Повторить») / класс сегодня (акцент «Сегодня») / класс отменён студией (карточка
приглушена, переход внутрь возможен — посмотреть, кто был записан, — но без рабочих действий) /
нет отзывов (блок рейтинга не показывает «0,0» — скрыт или «Отзывов пока нет»).

### 3.2. SCR-10 «Участники класса» — элементы

По брифу SCR-10, разд. 4:

| # | Элемент | Содержимое | Правила |
|---|---|---|---|
| 1 | Список участников | Имя каждого записанного | Имя — самое крупное в строке; одна строка = один человек |
| 2 | Заметка об аллергии | Текст заметки, если указана (ФТ-18) | Бросается в глаза, не за иконкой; если нет — **ничего не показывать** (не «Аллергий нет») |
| 3 | Экипировка | «Своя»/«Прокат» (ФТ-06) | Сводка сверху: «Прокатных наборов подготовить: N» |
| 4 | Действие «Не пришёл» | Кнопка у каждого участника, доступного для отметки | Крупная, с обязательным подтверждением; не как безобидный чекбокс |
| 5 | Статус участника | «Неявка», «Бронь отменена» | Отменённые — приглушённо, без кнопки; неявка — явно и визуально необратимо |
| 6 | Шапка класса | Программа, дата/время, N из M | Контекст: шеф уверен, что открыл тот самый класс |
| 7 | Подсказка о доступности | Почему отметка пока недоступна (класс не начался — Р-13) | Избавляет от ощущения «сломано» |

Состояния (SCR-10, разд. 6):

| Состояние | Когда | Что видно |
|---|---|---|
| До начала класса | Открыт предстоящий класс | Полный список (режим «подготовка»); кнопки неявки видимы, но **неактивны**, подсказка «Отметить неявку можно после начала класса» (Р-13) |
| Рабочее | `now() ≥ startAt` (класс идёт или прошёл) | Кнопки «Не пришёл» активны у всех с действующей бронью; крайнего срока нет (Р-13) |
| Подтверждение | Нажато «Не пришёл» | Диалог: имя, предупреждение о последствиях (D-02), разнесённые кнопки; мимо диалога отметку не поставить |
| Участник отмечен | Неявка подтверждена | Статус «Неявка», действие исчезает, вид окончательный; подсказка «ошиблись — обратитесь к администратору» (Р-02/D-11) |
| Бронь отменена | Отмена клиентом/студией до класса | Строка приглушена, «Бронь отменена», без кнопки (ФТ-19); отменённых уместно уводить вниз |
| Класс отменён студией | Открыт отменённый слот | Баннер «Класс отменён студией», список только для справки, все действия недоступны |
| Пусто | Никто не записан / все брони отменены | «На этот класс пока никто не записан» |
| Загрузка / ошибка загрузки | Данные не пришли / сеть | Скелетон / сообщение + «Повторить» |
| Ошибка действия | Отметка не сохранилась | Строка возвращается в исходное состояние + «Не получилось сохранить, попробуйте ещё раз»; шеф никогда не гадает, засчиталась ли отметка |

### 3.3. Видимость данных: отзывы и рейтинг (Р-07, ФТ-12)

Шеф **не видит тексты отзывов и оценки по отдельным классам** — их читает только администратор
(Р-07). В интерфейсе шефа показывается **только агрегат** — средний рейтинг + число отзывов
(ФТ-12) — компактным фоновым блоком в шапке/профиле SCR-09, не конкурирующим с расписанием и
**не намекающим, что его можно «развернуть» до отдельных отзывов** (SCR-09, разд. 4 и 9).
API это гарантирует: в `ParticipantBooking` и ответах кабинета шефа полей отзывов нет
(openapi.yaml: «Тексты отзывов сюда не входят (Р-07)» в описании `GET /slots/{slotId}/bookings`).

## 4. Контракт API

Все операции — по [../api/openapi.yaml](../api/openapi.yaml), авторизация bearer (ФТ-13).
Тело ошибок — схема `Error` (`code` машиночитаемый, `message` для UI, `details`).

### 4.1. `GET /chef/slots` — мои классы (operationId `listChefSlots`)

- Параметр `period` (query): `UPCOMING` | `PAST`; **по умолчанию — все**.
- **200** — массив `SlotSummary`, отсортирован по времени начала → рендер SCR-09; счётчик
  «записано N из M» UI вычисляет как `capacityTotal − freeSpots` из `M = capacityTotal`
  **[Пробел]** — выделенного поля «число записанных» в `SlotSummary` нет, формула — вывод
  из схемы, требует подтверждения (в т.ч. учитываются ли в `freeSpots` возвращённые места
  отменённых броней — по D-03 `freeSpots` пересчитывается системой, так что да, но явной
  фразы нет).
- **401** → экран входа (сессия истекла).
- **403** → «нет доступа» (роль не CHEF); в норме недостижимо из UI шефа.

Реакция UI: загрузка — скелетон; сетевая ошибка — сообщение + «Повторить» (SCR-09, разд. 6).

### 4.2. `GET /slots/{slotId}/bookings` — участники класса (operationId `listSlotBookings`)

Доступ: CHEF **своего** слота, ADMIN. Тексты отзывов не входят (Р-07).

- **200** — массив `ParticipantBooking`: `id`, `clientName`, `allergyNote` (nullable, ФТ-18),
  `status` (`CONFIRMED`/`CANCELLED_BY_CLIENT`/`CANCELLED_BY_STUDIO`/`COMPLETED`/`NO_SHOW` —
  ФТ-19), `equipmentChoice` (`OWN`/`RENTAL`, ФТ-06) → рендер SCR-10; пустой массив →
  состояние «Пусто».
- **401** → экран входа.
- **403** — «роль CLIENT либо **шеф чужого слота**» → UI показывает «нет доступа» и возвращает
  на SCR-09; из штатной навигации (только свои классы из `GET /chef/slots`) недостижимо —
  защита от подмены `slotId` в адресе.
- **404** — слот не существует → сообщение + возврат на SCR-09.

### 4.3. `POST /bookings/{bookingId}/no-show` — отметить неявку (operationId `markNoShow`)

Доступно только шефу, ведущему класс, и только после `slot.startAt` (Р-13, крайнего срока нет).
Эффект: `Booking.status → NO_SHOW`, `lateCancelCount` клиента +1; при достижении 3 — блок на
7 дней (D-02). Обратного действия у шефа нет (D-11). Вызов — только после подтверждения в диалоге.

| Код | Смысл (по спеке) | Реакция UI |
|---|---|---|
| **200** | Неявка зафиксирована; в ответе обновлённый `ParticipantBooking` | Строка сразу переходит в «Неявка», кнопка исчезает, вид окончательный; шеф остаётся на SCR-10 (неявившихся может быть несколько — SCR-10, разд. 3); подсказка про администратора (Р-02) |
| **401** | Сессия истекла | На экран входа; после входа — обратно к классу |
| **403** | **Не шеф этого класса** | «Нет доступа к этому классу», возврат на SCR-09; строка не меняется |
| **404** | Бронь не найдена | «Данные устарели» + перезагрузка списка (`GET /slots/{slotId}/bookings`) |
| **409** | **Класс ещё не начался (Р-13)** либо **бронь не в подходящем статусе** | Строка возвращается в исходное состояние; показать `message` из `Error`; перезагрузить список — статус мог измениться (бронь отменили, отметку уже поставили). Для «класс не начался» кнопки и так неактивны — код страхует от рассинхрона часов клиента и сервера |
| Обрыв сети | Ответ не получен | Состояние «Ошибка действия»: строка в исходное + «Не получилось сохранить, попробуйте ещё раз» (SCR-10, разд. 6) |

**[Пробел]** Конкретные значения `Error.code` для двух причин 409 (`класс не начался` vs
`бронь не в подходящем статусе`) в openapi.yaml не заданы (в примерах перечня кодов их нет) —
UI сможет различать причины только по коду; нужно зафиксировать коды при реализации бэкенда.

Смежная операция вне этого модуля: `DELETE /bookings/{bookingId}/no-show` (`revokeNoShow`) —
только ADMIN (ФТ-22, D-11, SCR-13); в интерфейсе шефа не вызывается и не показывается.

## 5. Бизнес-правила

| ID | Правило | Следствие для модуля |
|---|---|---|
| **ФТ-10** (Must) | `NO_SHOW` проставляет **только шеф** через свой интерфейс | SCR-10 — единственная точка входа действия (UC-06: у клиента и администратора кнопки нет); действие должно быть безусловно находимым |
| **D-02** / ФТ-09 (Must) | Неявка = **+1 к `lateCancelCount` клиента**; при достижении 3 — `blockedUntil = now() + 7 дней`, новые брони блокируются, затем блок и счётчик сбрасываются автоматически | Именно поэтому подтверждение обязательное, с именем и честным предупреждением о последствиях; без массовых отметок и свайпов (SCR-10, разд. 7) |
| **D-11** / ФТ-22 (Should) / Р-02 | Снятие ошибочной `NO_SHOW` — **только администратор**: бронь → `COMPLETED`, счётчик −1, блокировка пересчитывается. **У шефа обратного действия нет** | UI изображает отметку необратимой и подсказывает путь исправления через администратора |
| **D-09** / ФТ-23 (Must) / Р-10 | Слот и его подтверждённые брони автозавершаются по `startAt + 180 мин`; отметки `NO_SHOW` сохраняются (UC-10). **Ручного «завершить» нет ни у шефа, ни у администратора** | В кабинете шефа нет действия «завершить»/«сдать смену»; класс сам уходит в прошедшие (SCR-09), брони пришедших сами становятся `COMPLETED` |
| **Р-13** | Окно отметки: **с момента `startAt`, без крайнего срока; до начала — недоступно** | До начала кнопки неактивны с пояснением; после начала — активны бессрочно; никаких таймеров |
| ФТ-19 (Must) | Жизненные циклы `Slot`/`Booking`; отметка только по действующей брони | Отменённые и уже отмеченные брони недоступны для действия — интерфейс не допускает даже попытку (SCR-10, разд. 7) |
| ФТ-18 (Could) | Заметка об аллергии — необязательное поле | Показ только при заполнении; лимит ≤ 500 символов (Р-16, п. 3) |
| Р-07 / ФТ-12 | Шеф видит только агрегат рейтинга | См. 3.3 |
| ФТ-03/UC-07, ФТ-14/D-08 | Расписание и аккаунты — только ADMIN | Кабинет шефа read-only по расписанию, без онбординга и редактирования профиля |

## 6. Краевые случаи

1. **Отметка во время идущего класса** (`startAt ≤ now() < startAt + 180 мин`): бронь ещё
   `CONFIRMED`, переход `CONFIRMED → NO_SHOW` — штатный основной поток UC-06 (предусловие
   «класс начался», Р-13). Допустим.
2. **Отметка после автозавершения** (`now() ≥ startAt + 180 мин`): по UC-10/D-09 брони
   `CONFIRMED` уже переведены в `COMPLETED`. Р-13 прямо разрешает отметку «и позже, спокойно,
   после смены» (SCR-10, разд. 6), а D-11 описывает обратный переход `NO_SHOW → COMPLETED` при
   снятии администратором. Отсюда следует, что переход **`COMPLETED → NO_SHOW` должен быть
   допустим** — иначе Р-13 («без крайнего срока») невыполнимо после `startAt + 180 мин`.
   **[Пробел / открытый вопрос]** Ни один источник (UC-06, D-09/D-11, Р-13, openapi.yaml —
   формулировка 409 «бронь не в подходящем статусе» без перечня подходящих статусов) не
   фиксирует этот переход явно. До подтверждения принимаем: для `markNoShow` подходящие
   исходные статусы — `CONFIRMED` и `COMPLETED`; `CANCELLED_BY_CLIENT`, `CANCELLED_BY_STUDIO`
   и повторный `NO_SHOW` → 409. Требуется подтвердить у аналитика и зафиксировать в
   openapi.yaml.
3. **Слот отменён студией** (ФТ-04, UC-03): брони каскадно переходят в `CANCELLED_BY_STUDIO`
   (UC-03, шаг 2 — помечен в источнике как логический вывод). На SCR-09 карточка остаётся с
   пометкой «Отменён студией», приглушена, переход внутрь возможен; на SCR-10 — баннер, список
   только для справки, **все действия недоступны** (SCR-09/SCR-10, разд. 6): по ФТ-19 отметка
   возможна только по действующей брони, а действующих здесь нет. Показывать ли отменённый
   класс среди прошедших после его даты — на усмотрение дизайнера (SCR-09, разд. 9).
4. **Пустой список участников** (никто не записан или все брони отменены): `GET
   /slots/{slotId}/bookings` → 200 с пустым массивом; состояние «На этот класс пока никто не
   записан» — спокойно, без тревоги (SCR-10, разд. 6). Отличать «пусто» от «ошибка загрузки».
5. **Гонка «класс вот-вот начнётся»**: кнопка на клиенте активировалась по локальным часам, но
   сервер ещё считает `now() < startAt` → 409 (Р-13); UI показывает `message` и не помечает
   строку. Обратная гонка (кнопка неактивна, а класс уже начался) лечится действием «обновить
   список» (SCR-10, разд. 5). **[Пробел]** Способ синхронизации времени клиента с сервером
   (пуш/поллинг/серверный флаг «отметка доступна» в ответе) источниками не задан — решение за
   разработкой.
6. **Отметка не сохранилась** (обрыв сети): строка возвращается в исходное состояние, явное
   сообщение; повтор — заново через диалог (SCR-10, разд. 6, состояние «Ошибка действия»).
7. **Параллельное снятие отметки администратором** (D-11) во время открытого у шефа экрана:
   шеф увидит актуальный статус после обновления списка; специальных push-уведомлений
   источники не требуют.

## 7. Критерии приёмки

Формат «Дано / Когда / Тогда». КП-1…КП-4 — обязательный минимум.

**КП-1. Недоступность отметки до начала класса (Р-13, UC-06 предусловие)**
- *Дано:* шеф открыл SCR-10 своего слота в статусе `SCHEDULED`, `now() < slot.startAt`;
  у участника бронь `CONFIRMED`.
- *Когда:* шеф смотрит на строку участника.
- *Тогда:* кнопка «Не пришёл» видима, но неактивна, рядом пояснение «Отметить неявку можно
  после начала класса»; вызов `POST /bookings/{id}/no-show` из UI невозможен, а прямой вызов
  API возвращает **409** и статус брони не меняется.

**КП-2. Успешная отметка с инкрементом счётчика (UC-06, ФТ-10, D-02)**
- *Дано:* `now() ≥ slot.startAt`; шеф — ведущий этого класса; бронь участника в допустимом
  статусе (`CONFIRMED`; для `COMPLETED` — см. открытый вопрос в разд. 6, п. 2);
  `lateCancelCount` клиента = k (k < 2).
- *Когда:* шеф нажимает «Не пришёл», в диалоге видит имя участника и предупреждение о
  последствиях и подтверждает.
- *Тогда:* API возвращает **200** с `ParticipantBooking.status = NO_SHOW`; строка сразу
  показывает «Неявка», кнопка действия исчезает; `lateCancelCount` клиента = k + 1; шеф
  остаётся на SCR-10. *Дополнительно (D-02):* если k = 2, после отметки
  `ClientProfile.blockedUntil = now() + 7 дней` и новые брони клиента отклоняются до этой даты.

**КП-3. 403 на чужой слот (`GET /slots/{slotId}/bookings`)**
- *Дано:* аутентифицированный шеф A; существует слот шефа B.
- *Когда:* от имени A запрашивается `GET /slots/{slotId_B}/bookings` (например, подменой id в
  адресе).
- *Тогда:* API возвращает **403**; список участников не раскрывается; UI показывает «нет
  доступа» и возвращает на SCR-09. Аналогично `POST /bookings/{id}/no-show` по брони чужого
  класса → **403**, статус брони не меняется.

**КП-4. Отсутствие обратного действия у шефа (D-11, Р-02)**
- *Дано:* участник отмечен как `NO_SHOW` на SCR-10.
- *Когда:* шеф просматривает строку участника (в т.ч. после перезагрузки списка).
- *Тогда:* в интерфейсе шефа нет никакого элемента снятия/переключения отметки; строка выглядит
  окончательной (не переключатель); присутствует подсказка «ошиблись — обратитесь к
  администратору»; операция `DELETE /bookings/{id}/no-show` от имени роли CHEF возвращает
  **403** (по openapi.yaml она только для ADMIN).

**КП-5. Подтверждение обязательно (SCR-10, разд. 5–6)**
- *Дано:* рабочее состояние SCR-10, активная кнопка «Не пришёл».
- *Когда:* шеф нажимает кнопку и в диалоге выбирает «отменить» (или закрывает диалог).
- *Тогда:* запрос к API не отправляется, статус участника не меняется; поставить отметку мимо
  диалога невозможно.

**КП-6. Полная история и разделение периодов (Р-16, SCR-09)**
- *Дано:* у шефа есть завершённые классы старше любого «разумного» горизонта (например, > 1 года).
- *Когда:* шеф открывает вкладку/секцию «Прошедшие» (`GET /chef/slots?period=PAST`).
- *Тогда:* показаны **все** прошедшие классы без обрезания по давности; предстоящие и прошедшие
  не смешаны.

**КП-7. Отменённый студией класс (ФТ-19, SCR-09/SCR-10)**
- *Дано:* слот шефа в статусе `CANCELLED_BY_STUDIO`.
- *Когда:* шеф видит его на SCR-09 и открывает SCR-10.
- *Тогда:* на SCR-09 карточка с пометкой «Отменён студией», приглушена; на SCR-10 — баннер,
  список для справки, ни у одной строки нет активного действия «Не пришёл».

**КП-8. Аллергия показывается только при наличии (ФТ-18)**
- *Дано:* в классе участник X с заполненной заметкой об аллергии и участник Y без заметки.
- *Когда:* шеф открывает SCR-10.
- *Тогда:* у X заметка видна заметно (не спрятана за иконкой); у Y на месте заметки нет ничего —
  в частности, нет текста вида «Аллергий нет».

### Реестр пробелов (все помечены **[Пробел]** по тексту)

1. **Переход `COMPLETED → NO_SHOW`** после автозавершения слота явно не зафиксирован ни в
   UC-06/D-09/D-11/Р-13, ни в openapi.yaml (409: «бронь не в подходящем статусе» — без перечня

   зафиксировать при реализации (разд. 4.3).
3. **Число записанных «N из M»** в `SlotSummary` — выводится как `capacityTotal − freeSpots`,
   выделенного поля нет; формулу подтвердить (разд. 4.1).
4. **Синхронизация «класс начался»** между клиентом и сервером (когда активировать кнопки) —
   механизм источниками не задан, на усмотрение разработки (разд. 6, п. 5).
