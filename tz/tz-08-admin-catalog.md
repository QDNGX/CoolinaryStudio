# ТЗ-08 — Админ: каталог программ

> Модуль административного ведения каталога программ веб-приложения «Шеф-стол».
> Роли системы: CLIENT / CHEF / ADMIN; бэкенд — Spring Boot, фронт — веб (адаптив).
>
> Источники (всё в этом ТЗ прослеживается до ID; собственных решений документ не вводит):
> - Брифы экранов: [SCR-11](../analytics/design-brief/briefs/scr-11-admin-programs-list.md),
>   [SCR-12](../analytics/design-brief/briefs/scr-12-admin-program-form.md)
> - Контракт API: [openapi.yaml](../api/openapi.yaml) — `listPrograms`, `createProgram`,
>   `getProgram`, `updateProgram`; схемы `Program`, `ProgramInput`, `Error`
> - Требования: [ФТ-01, ФТ-02](../analytics/requirements/functional-requirements-cooking.md),
>   [UC-07](../analytics/requirements/use-cases-cooking.md),
>   [US-08](../analytics/requirements/user-stories-cooking.md)
> - Решения заказчика: [Р-04, Р-15, Р-16](../analytics/design-brief/design-decisions-cooking.md)

---

## 1. Назначение и охват

Модуль покрывает два экрана административной части:

| Экран | Название | Назначение |
|---|---|---|
| **SCR-11** | Каталог программ | Список всех программ студии; точка входа в создание и редактирование. Единственная точка входа в обе операции (ФТ-01, бриф SCR-11 §7) |
| **SCR-12** | Программа: форма | Одна форма на два режима — создание (пустая) и редактирование (с данными); режимы различаются только заполненностью и заголовком (бриф SCR-12 §7, UC-07/US-08) |

Доступ на запись — только роль ADMIN (ФТ-01). Каталог небольшой: единицы — максимум пара
десятков программ; интерфейс не проектируется под сотни строк (бриф SCR-11 §1, §4).

**Вне охвата** (по брифам SCR-11 §5, SCR-12 §5):
- удаление и архивация программ — операций нет в системе (Р-15);
- вместимость, шеф и время конкретного класса — это уровень слота (SCR-14), не программы;
- инлайн-редактирование полей прямо в списке SCR-11 — правки только через форму SCR-12.

## 2. Пользовательские сценарии

Актёр всех сценариев — Администратор (UC-07, US-08).

### 2.1. Создание программы

1. Админ на SCR-11 нажимает «Создать программу» (кнопка доступна и при пустом каталоге — бриф SCR-11 §4).
2. Открывается пустая форма SCR-12; значения по умолчанию: сложность — «новичок» (BEGINNER),
   признак сложного оборудования — выключен; подпись про лимит группы видна сразу (бриф SCR-12 §6).
3. Админ заполняет карточку: название, описание, тип кухни, сложность, признак оборудования,
   блюда, фото (состав фиксирован — ФТ-01).
4. «Сохранить» → `POST /programs` → возврат на SCR-11, где видна созданная программа
   (бриф SCR-12 §3; UC-07 постусловие: программа в каталоге).

### 2.2. Редактирование программы

1. Админ на SCR-11 кликает по названию программы (название — точка входа в редактирование, бриф SCR-11 §4).
2. Открывается та же форма SCR-12 с данными программы; заголовок ясно отличает правку от создания (бриф SCR-12 §6).
3. Админ правит поля (опечатка, новое фото, добавленное блюдо — бриф SCR-12 §1).
4. «Сохранить» → `PUT /programs/{programId}` → возврат на SCR-11 с обновлённой программой.
5. «Отмена» или уход с формы при несохранённых правках → предупреждение о потере правок
   с выбором «остаться / уйти без сохранения» (бриф SCR-12 §3, §6).

### 2.3. Включение «сложного оборудования» у программы с существующими слотами

Ключевой сценарий модуля («момент истины экрана» — бриф SCR-12 §6).

1. Админ открывает на редактирование программу, у которой уже созданы слоты (например, на 12 мест).
2. Включает переключатель «сложное оборудование».
3. **UI обязан немедленно, заметно и немодально показать два факта** (Р-04, бриф SCR-12 §4, §6, §7):
   - лимит группы для **будущих** классов меняется: 12 → 8;
   - **уже созданные слоты сохраняют свою вместимость** — задним числом ничего не пересчитывается
     и не ломается.
   Запрещено: «тихая галочка» (молча запомнить) и блокирующее модальное окно.
4. Админ сохраняет → `PUT /programs/{programId}`. Сервер не трогает `capacityTotal`
   существующих слотов (описание операции `updateProgram` в openapi.yaml; поле
   `SlotSummary.capacityTotal`: «зафиксирована при создании слота (Р-04)»).

Цель явного сообщения: Артём не должен бояться трогать признак у «живой» программы (бриф SCR-12 §7).

## 3. Интерфейс: экраны и состояния

### 3.1. SCR-11 — Каталог программ

Элементы, по приоритету (бриф SCR-11 §4):

| # | Элемент | Содержимое | Примечания |
|---|---|---|---|
| 1 | Список программ | Название (главное, точка входа в редактирование), тип кухни, уровень сложности («новичок» / «продвинутый») | Короткий список, не «таблица-простыня» |
| 2 | Признак «сложное оборудование» | Заметная, но спокойная пометка; желательно с намёком на следствие («группа до 8») | Обязательная видимость: Артём планирует расписание, глядя на каталог (ФТ-02, бриф SCR-11 §7) |
| 3 | Кнопка «Создать программу» | Вход в пустую SCR-12 | Одна, заметная, стабильное место; доступна и при пустом каталоге |
| 4 | Фото программы (превью) | Первое фото, если загружено | При отсутствии — аккуратный плейсхолдер, не «сломанная картинка» |
| 5 | Счётчик/заголовок раздела | Число программ в каталоге | Служебное |

Состояния (бриф SCR-11 §6):

| Состояние | Когда | Что видно |
|---|---|---|
| По умолчанию | Есть программы | Список + кнопка создания |
| Загрузка | Данные едут | Скелетон/плейсхолдер списка; кнопка создания может появиться сразу |
| Пусто | Каталог не наполнялся | Дружелюбный empty state: программы — основа расписания; кнопка «Создать программу» — единственный акцент |
| Ошибка | Список не загрузился | «Не получилось загрузить каталог» + «повторить»; без технических кодов |
| Программа со сложным оборудованием | Признак включён | Пометка у строки/карточки, коротко («группа до 8») |

На усмотрение дизайнера (бриф SCR-11 §9): карточки или строки; сортировка по умолчанию;
внешний вид пометки оборудования; мобильное поведение; микрокопирайт; поиск/фильтр
(требованиями не зафиксированы, при размере каталога вероятно избыточны).

### 3.2. SCR-12 — Форма программы

Элементы, по приоритету (бриф SCR-12 §4; состав карточки фиксирован ФТ-01, форма одноуровневая — ничего не прятать в «дополнительные настройки»):

| # | Элемент | Требования |
|---|---|---|
| 1 | Название | Обязательное; крупно, первым; в режиме редактирования говорит «где я» |
| 2 | Признак «сложное оборудование» + пояснение | Переключатель и **постоянная** подпись рядом (не по наведению): обычная программа — группа до 12, со сложным оборудованием — до 8 (ФТ-02). При переключении — явная реакция: смена лимита 12 → 8 (или 8 → 12) и «уже созданных классов это не коснётся» (Р-04) |
| 3 | Уровень сложности | Ровно два значения — «новичок» (`BEGINNER`) / «продвинутый» (`ADVANCED`); переключатель/сегмент, не выпадающий список |
| 4 | Тип кухни | Короткое поле. Формат — свободный текст с подсказками из уже использованных значений (чтобы не плодить дубли «итальянская»/«Итальянская») — рекомендация из design-decisions, раздел «Оставлены на усмотрение дизайнера/разработки», не требование |
| 5 | Список блюд | Несколько позиций; добавление/удаление/порядок; лёгкое редактирование без модального окна на каждое блюдо |
| 6 | Фото | Загрузка нескольких фото, **не более 10** (Р-16 п.3); превью, удаление; показать, какое фото — «лицо» программы. Ограничение на размер файла — **уточнит разработка** (бриф SCR-12 §4, §9 — в источниках не задано) |
| 7 | Описание | Свободный текст, многострочное; жёсткий лимит в источниках не задан («подсказать разумный объём» — на усмотрение дизайнера) |
| 8 | «Сохранить» / «Отмена» | Сохранение — единственный акцент внизу формы |

Состояния (бриф SCR-12 §6):

| Состояние | Когда | Что видно |
|---|---|---|
| По умолчанию (создание) | Вход через «Создать программу» | Пустая форма; дефолты: «новичок», признак выключен; подпись про лимит видна сразу |
| По умолчанию (редактирование) | Клик по программе в SCR-11 | Та же форма с данными; заголовок отличает правку от создания |
| Загрузка | Едут данные (редактирование) или идёт сохранение | Скелетон формы / заблокированная кнопка с индикатором; **двойной клик не создаёт дубль** |
| Пусто (внутри формы) | Списки блюд/фото пусты | «Блюда ещё не добавлены», «фото ещё нет» + очевидный способ добавить |
| Ошибка (валидация) | Не заполнено обязательное (например, название) | Подсветка конкретного поля с человеческим текстом; форма не очищается |
| Ошибка (сохранение) | Сеть/сервер не приняли | Введённое остаётся на экране; «не получилось сохранить» + «повторить» |
| Переключение признака оборудования | Вкл/выкл признака | Заметная немодальная реакция: лимит будущих классов 12 → 8 (или 8 → 12), существующие классы сохраняют вместимость |
| Несохранённые правки при уходе | Уход с формы с изменениями | Предупреждение: остаться / уйти без сохранения |

На усмотрение дизайнера (бриф SCR-12 §9): компоновка; паттерн редактирования списка блюд;
механика загрузки фото; форма реакции на переключение признака (инлайн-подпись, плашка — лишь бы
заметно и немодально); все тексты вне фиксированного смысла «12 → 8».

## 4. Контракт API

Все операции — из [openapi.yaml](../api/openapi.yaml), тег `catalog`. Тело ошибок — схема
`Error` (`code` — машиночитаемый код, `message` — человекочитаемое сообщение для UI,
`details` — дополнительный контекст).

### 4.1. Схемы

**`Program`** (ответы; required: все поля):

| Поле | Тип | Примечание |
|---|---|---|
| `id` | string (uuid) | |
| `title` | string | |
| `description` | string | |
| `cuisineType` | string | Свободный текст; подсказки из использованных значений — рекомендация design-decisions |
| `difficultyLevel` | enum `BEGINNER` \| `ADVANCED` | Схема `DifficultyLevel` |
| `requiresComplexEquipment` | boolean | true — лимит группы 8 вместо 12 (ФТ-02); действует только на новые слоты (Р-04) |
| `dishes` | string[] | |
| `photos` | string[] (URL), `maxItems: 10` | До 10 фото (Р-16 п.3) |

**`ProgramInput`** (тела POST/PUT; required: `title`, `description`, `cuisineType`,
`difficultyLevel`, `requiresComplexEquipment`): те же поля без `id`; `dishes` и `photos`
по контракту **необязательны** (см. §6.3). `photos` — `maxItems: 10`.

> **Пробел источников:** отдельная операция загрузки файла фото в openapi.yaml отсутствует —
> `photos` передаются строками-URL. Механизм получения URL (эндпоинт загрузки, хранилище) и
> лимит размера файла — **на усмотрение/уточнение разработки** (бриф SCR-12 §9).

### 4.2. `GET /programs` — `listPrograms`

- **Доступ:** без аутентификации (`security: []`) — чтение доступно всем ролям и гостю (Р-16 п.2).
- **Ответ 200:** массив `Program` — все программы каталога. Пагинации нет (каталог только растёт,
  архивации нет — Р-15; размер — единицы–десятки).
- **Реакция UI (SCR-11):** 200 с непустым массивом → состояние «по умолчанию»; 200 с `[]` →
  empty state; сетевая/серверная ошибка → состояние «Ошибка» с кнопкой «повторить» (§3.1).

### 4.3. `POST /programs` — `createProgram` (ADMIN)

- **Тело:** `ProgramInput` (required).
- **Ответ 201:** созданная `Program`. UI: возврат на SCR-11, программа видна в списке.
- **400 `ValidationError`:** некорректный запрос — отсутствие required-полей; нарушение
  `photos.maxItems: 10` (Р-16 п.3). UI: подсветка конкретного поля с человеческим текстом
  (`Error.message`), форма не очищается (SCR-12 §6 «Ошибка (валидация)»).
- **401 `Unauthorized`:** не аутентифицирован (ФТ-13). UI: направить на вход.
- **403 `Forbidden`:** роль не ADMIN (ФТ-01, матрица доступа). UI: спокойное сообщение
  «операция недоступна»; экраны SCR-11/12 в норме недостижимы для других ролей.
- **Реакция на сеть/5xx:** введённое сохраняется на экране, «не получилось сохранить» + повторить;
  защита от дубля по двойному клику — блокировка кнопки на время запроса (SCR-12 §6).

### 4.4. `GET /programs/{programId}` — `getProgram`

- **Доступ:** без аутентификации (`security: []`).
- **Ответ 200:** `Program`. UI: заполнение формы SCR-12 в режиме редактирования
  (состояние «Загрузка» — скелетон, пока данные едут).
- **404 `NotFound`:** ресурс не найден. UI: точная форма сообщения в источниках не задана —
  **на усмотрение разработки**; логично сообщить об ошибке и вернуть в каталог SCR-11.

### 4.5. `PUT /programs/{programId}` — `updateProgram` (ADMIN)

- **Тело:** `ProgramInput` (required).
- **Семантика:** смена `requiresComplexEquipment` **не трогает** вместимость уже созданных
  слотов (Р-04 — описание операции в openapi.yaml).
- **Ответ 200:** обновлённая `Program`. UI: возврат на SCR-11.
- **400 / 401 / 403:** как в §4.3 (включая лимит 10 фото).
- **404 `NotFound`:** программа не найдена — как в §4.4.

## 5. Бизнес-правила

| ID | Правило | Проявление в модуле |
|---|---|---|
| **ФТ-01** | Только администратор создаёт и редактирует программы; состав карточки: название, описание, тип кухни, уровень сложности, признак сложного оборудования, блюда, фото | POST/PUT — только ADMIN (401/403); SCR-11 — единственная точка входа в обе операции; форма SCR-12 одноуровневая, состав полей фиксирован |
| **ФТ-02** | Вместимость класса: 12 по умолчанию, 8 при `requiresComplexEquipment = true` | Постоянная подпись у переключателя на SCR-12; видимая пометка признака в списке SCR-11 (желательно с «группа до 8») — Артём не вспоминает лимит по памяти |
| **Р-04** (и примечание к ФТ-02) | Смена признака действует **только на новые слоты**; существующие сохраняют вместимость (`capacityTotal` фиксируется при создании слота) | UI обязан сообщить оба факта в момент переключения (§2.3); сервер не пересчитывает существующие слоты при PUT |
| **Р-15** | Архивации/удаления программ нет — каталог в MVP только растёт (осознанное ограничение) | Ни на SCR-11, ни на SCR-12 нет кнопок удаления/архивации («не рисовать кнопку-обещание»); в API нет DELETE |
| **Р-16 п.3** | Фото программы ≤ 10 | Лимит в UI загрузки (SCR-12) и в контракте (`maxItems: 10`, иначе 400) |
| **Р-16 п.2** | Витрина видна без входа | `GET /programs`, `GET /programs/{id}` — `security: []` (чтение каталога публично) |
| **UC-07 / US-08** | Ведение каталога — часть цикла «каталог → расписание» | После сохранения/отмены возврат на SCR-11, контекст не теряется |

Смежное, но вне модуля: вместимость конкретного слота задаётся на форме слота SCR-14 (ФТ-03);
программа лишь определяет потолок через признак оборудования (бриф SCR-12 §5).

## 6. Краевые случаи

### 6.1. Снятие признака сложного оборудования

Симметрично включению (бриф SCR-12 §6: «включает/выключает»): при выключении UI показывает
обратную смену лимита **8 → 12** для новых слотов; уже созданные слоты (на 8 мест) сохраняют
свою вместимость (Р-04 действует в обе стороны: «признак действует только на новые слоты»).

### 6.2. Пустой каталог

Старт студии в системе: `GET /programs` → `[]`. SCR-11 показывает empty state с объяснением,
что программы — основа расписания, и кнопкой «Создать программу» как единственным акцентом
(бриф SCR-11 §6). Кнопка создания работает при пустом каталоге (бриф SCR-11 §4).

### 6.3. Программа без фото и/или без блюд

- По контракту `ProgramInput` поля `dishes` и `photos` **не входят в required** — программа без
  фото и без блюд допустима на уровне API.
- В брифах обязательным явно названо только название (SCR-12 §6 «Ошибка (валидация)»); для
  фото/блюд предусмотрены пустые состояния внутри формы («блюда ещё не добавлены», «фото ещё нет»).
- SCR-11 при отсутствии фото показывает аккуратный плейсхолдер (бриф SCR-11 §4).
- **Пробел источников:** требование минимального числа блюд/фото нигде не задано — блокировать ли
  сохранение без блюд/фото на фронте — **на усмотрение разработки** (рекомендуемая трактовка —
  не блокировать, следуя контракту API).
- Примечание: `description` и `cuisineType` входят в required `ProgramInput`, при этом брифы их
  обязательность не акцентируют; расхождение трактуется в пользу контракта API (400 при отсутствии).

### 6.4. Прочее

- **Двойной клик по «Сохранить»** — не создаёт дубль: кнопка блокируется на время запроса (бриф SCR-12 §6).
- **Уход с несохранёнными правками** — предупреждение с выбором (бриф SCR-12 §3, §6).
- **Дубли типа кухни** («итальянская»/«Итальянская») — смягчаются подсказками из использованных
  значений; это рекомендация, жёсткой нормализации в источниках нет (design-decisions,
  «Оставлены на усмотрение…»).
- **Ошибка загрузки списка** — состояние «Ошибка» SCR-11 с повтором, без технических кодов.

## 7. Критерии приёмки

Формат: Дано / Когда / Тогда.

**КП-1. Создание программы (UC-07, ФТ-01)**
Дано: админ аутентифицирован, открыта пустая форма SCR-12 (дефолты: BEGINNER, признак выключен).
Когда: заполнены `title`, `description`, `cuisineType` и нажато «Сохранить».
Тогда: выполнен `POST /programs`, получен 201 с `Program`; админ возвращён на SCR-11, где видна новая программа.

**КП-2. Редактирование программы (ФТ-01)**
Дано: в каталоге есть программа; админ кликнул по её названию на SCR-11.
Когда: изменено любое поле и нажато «Сохранить».
Тогда: выполнен `PUT /programs/{programId}`, получен 200; на SCR-11 отображены обновлённые данные.

**КП-3. Подпись о лимите видна до переключения (ФТ-02)**
Дано: открыта форма SCR-12 (создание или редактирование).
Когда: админ ещё ничего не трогал.
Тогда: рядом с переключателем «сложное оборудование» постоянно видна подпись «обычная — группа до 12, со сложным оборудованием — до 8» (не по наведению).

**КП-4. Реакция на включение признака (ФТ-02, Р-04)**
Дано: форма редактирования программы, признак выключен.
Когда: админ включает признак.
Тогда: без модального окна и до сохранения показаны два факта: лимит будущих классов 12 → 8 и «уже созданные классы сохраняют вместимость».

**КП-5. Смена `requiresComplexEquipment` не меняет `capacityTotal` существующих слотов (Р-04)**
Дано: программа с `requiresComplexEquipment = false` и созданным слотом с `capacityTotal = 12`.
Когда: админ включает признак и сохраняет (`PUT /programs/{programId}` → 200).
Тогда: у существующего слота `capacityTotal` остаётся 12; слот, созданный по программе после сохранения, получает `capacityTotal = 8`.

**КП-6. Обратное снятие признака (Р-04)**
Дано: программа с `requiresComplexEquipment = true` и слотом с `capacityTotal = 8`.
Когда: админ выключает признак и сохраняет.
Тогда: UI при переключении показал смену 8 → 12 для новых классов; существующий слот сохраняет `capacityTotal = 8`; новые слоты создаются с лимитом 12.

**КП-7. Лимит фото (Р-16 п.3)**
Дано: у программы в форме 10 фото.
Когда: админ пытается добавить одиннадцатое.
Тогда: UI не даёт превысить лимит (и/или сервер отвечает 400 `ValidationError` при `photos > 10`); данные формы не теряются.

**КП-8. Валидация обязательного поля**
Дано: форма создания с пустым названием.
Когда: нажато «Сохранить».
Тогда: показана подсветка поля с человеческим текстом; форма не очищена; при серверной проверке — 400 `ValidationError` с `message` для UI.

**КП-9. Доступ только для админа (ФТ-01)**
Дано: запрос `POST /programs` или `PUT /programs/{programId}` без аутентификации / с ролью CLIENT или CHEF.
Когда: запрос выполнен.
Тогда: ответ 401 `Unauthorized` / 403 `Forbidden` соответственно; программа не создана/не изменена. При этом `GET /programs` и `GET /programs/{id}` доступны без входа (Р-16 п.2).

**КП-10. Пустой каталог**
Дано: `GET /programs` возвращает `[]`.
Когда: админ открывает SCR-11.
Тогда: показан empty state с кнопкой «Создать программу»; кнопка ведёт в пустую SCR-12.

**КП-11. Нет удаления и архивации (Р-15)**
Дано: SCR-11 и SCR-12 отрисованы для любой программы.
Когда: админ осматривает все действия экранов.
Тогда: операции удаления/архивации отсутствуют как элементы UI.

**КП-12. Защита от дубля и потеря правок**
Дано: форма с заполненными данными, сохранение в процессе.
Когда: админ повторно кликает «Сохранить».
Тогда: второй запрос не отправляется (кнопка заблокирована), дубль не создаётся.
Дано: форма с несохранёнными правками. Когда: админ уходит с формы. Тогда: показано предупреждение с выбором «остаться / уйти без сохранения».

**КП-13. Ошибка сохранения не теряет ввод**
Дано: заполненная форма; сервер/сеть недоступны.
Когда: нажато «Сохранить» и запрос упал.
Тогда: введённые данные остаются на экране; показано «не получилось сохранить» с возможностью повторить.

**КП-14. Программа не найдена**
Дано: переход к редактированию по несуществующему `programId`.
Когда: `GET /programs/{programId}` вернул 404.
Тогда: UI сообщает об ошибке (точная форма — на усмотрение разработки, см. §4.4) и не показывает пустую форму как «создание».
