## 1. Назначение и охват

Модуль реализует рабочий стол администратора студии по расписанию: обзор сетки слотов,
создание слота, отмену слота с уведомлением гостей и снятие ошибочной отметки неявки.

| Экран | Название | Роль | Что делает в модуле |
|---|---|---|---|
| SCR-13 | Расписание слотов (админ) | ADMIN | Сетка слотов по датам; вход в создание (SCR-14) и отмену (SCR-15); вход в список броней завершённого слота для снятия ошибочной отметки неявки (Р-02) |
| SCR-14 | Слот: форма создания | ADMIN | Создание слота из программы, шефа, даты/времени, вместимости и проката (ФТ-03, UC-07) |
| SCR-15 | Отмена слота (диалог) | ADMIN | Отмена слота с обязательной причиной, каскадной отменой броней и письмами гостям (ФТ-04, UC-03) |

**Платформа:** адаптивный веб — десктоп + мобильный браузер (НФТ-01; брифы SCR-13/14/15).
Отдельно бриф SCR-15 требует полноценной работы диалога отмены с телефона (НФТ-01, бриф SCR-15 §7).

**Что модуль осознанно НЕ делает:**

- **Редактирование слота** — операции нет вовсе: исправление = отмена (с письмами) + создание заново (Р-03, бриф SCR-13 §5, бриф SCR-14 — вход только на создание).
- **Ручное завершение слота** — кнопки «завершить» не существует: слот завершается автоматически по времени `startAt + 180 мин` (D-09, Р-10, ФТ-23, UC-10); интерфейс не должен её изображать (бриф SCR-13 §7).
- **Проставление неявки** — это делает шеф (ФТ-10, UC-06); админу доступно только снятие ошибочной отметки (ФТ-22, D-11, Р-02).
- **Отмена «в один клик» из сетки** — отмена всегда идёт через отдельный диалог SCR-15 с обязательной причиной (бриф SCR-13 §5, ФТ-04).

## 2. Пользовательские сценарии

1. **Планирование недели: создание слота.** Админ из сетки SCR-13 (кнопка «Создать слот», возможно «из дня» с предзаполненной датой — бриф SCR-13 §4, SCR-14 §3) открывает форму SCR-14, выбирает программу и шефа, задаёт дату/время, вместимость и прокат, создаёт слот. **Лимит вместимости определяет программа:** 12 по умолчанию, 8 при `requiresComplexEquipment = true` — форма подставляет и ограничивает лимит сама в момент выбора программы, с видимым объяснением; проверка «после отправки» недопустима (ФТ-02, бриф SCR-14 §4 п.5, §7). Созданный слот немедленно доступен клиентам — черновиков и отложенной публикации нет (UC-07, US-09, бриф SCR-14 §7).
2. **Конфликт пересечения слотов шефа.** Выбранный шеф уже занят в это время другим классом (`SCHEDULED`, с учётом фиксированных 180 минут) — сохранение отклоняется, ошибка обязана **назвать конфликтующий класс** (программа, дата и время), а не ограничиваться «время занято»; данные формы целы (D-10, ФТ-24, UC-07/A1, Р-08, бриф SCR-14 §6, §7). API возвращает 409 с `details.conflictingSlotId` — UI по нему подгружает и показывает конфликтующий слот (см. §4.1).
3. **Отмена слота (форс-мажор).** Админ находит класс в сетке, открывает диалог SCR-15, видит что именно отменяет и сколько гостей затронуто, вводит обязательную причину (≤ 300 символов — Р-16 п.3) и подтверждает. Слот → `CANCELLED_BY_STUDIO`, все подтверждённые брони каскадно → `CANCELLED_BY_STUDIO`, каждому записавшемуся уходит email с причиной (UC-03, ФТ-04, ФТ-15, D-07). Отправка писем — best-effort: сбой почты не откатывает отмену (НФТ-03); отчёта о доставке нет (Р-16 п.5). Отмена необратима, слот закрыт для броней навсегда (ФТ-05, D-04).
4. **Исправление ошибки в слоте.** Не туда поставил время / не того шефа — редактирования нет (Р-03): админ отменяет слот через SCR-15 (гости получают письма) и создаёт новый через SCR-14 (UC-07, примечание).
5. **Снятие ошибочного `NO_SHOW`.** С завершённого слота в сетке админ открывает список броней (единственное место в админке для этого действия — Р-02, бриф SCR-13 §3, §5) и снимает ошибочную отметку неявки, поставленную шефом: бронь → `COMPLETED`, `lateCancelCount` клиента −1, блокировка снимается, если условие перестаёт выполняться (ФТ-22, D-11, UC-06/A1). Действие осознанное, с подтверждением; интерфейс подтверждает результат («отметка снята») — бриф SCR-13 §6. Для доступа к прошлым классам навигация по датам позволяет уйти в прошлое на любую глубину (Р-16 п.4, бриф SCR-13 §4).

## 3. Интерфейс: экраны и состояния

### 3.1 SCR-13 — Расписание слотов

Элементы (приоритеты — бриф SCR-13 §4):

| # | Элемент | Содержимое | Источник данных |
|---|---|---|---|
| 1 | Сетка слотов по датам | Классы, сгруппированные по дням; на карточке: программа, шеф, время начала. Горизонт «текущая + следующая неделя» — на расстоянии одного взгляда/жеста | `GET /slots` (массив `SlotSummary`): `programTitle`, `chef`, `startAt` |
| 2 | Занятость слота | Брони против вместимости («7 из 12»); «пустой / заполняется / полный» различимы с одного взгляда, не только цифрами | `capacityTotal`, `freeSpots` (занято = `capacityTotal − freeSpots`) |
| 3 | Статус слота | `SCHEDULED` / `CANCELLED_BY_STUDIO` / `COMPLETED` (ФТ-19); отменённые не прятать, но и не давать спорить за внимание с живыми | `SlotSummary.status` |
| 4 | Кнопка «Создать слот» | Вход в SCR-14; главный акцент экрана; желательно создание «в контексте дня» (предзаполнение даты — на усмотрение дизайнера, бриф §9) | — |
| 5 | Прокат на слоте | Сколько наборов заложено и почём — кратко, без детализации | `rentalSetsAvailable`, `rentalPricePerSet` |
| 6 | Навигация по датам | Недели/периоды, включая прошлые без ограничения глубины (Р-16 п.4) — прошлые периоды нужны в т.ч. для снятия ошибочной неявки | Параметры `dateFrom`/`dateTo` |

`COMPLETED` выставляет **только система** по `startAt + 180 мин` (D-09, Р-10, ФТ-23, UC-10):
никакой кнопки «завершить» на экране нет и не изображается (бриф SCR-13 §5, §7). Также на
карточке слота нет «карандашей» редактирования (Р-03) и быстрых «крестиков» отмены (бриф SCR-13 §5, §7).

Состояния (бриф SCR-13 §6):

| Состояние | Когда | Что видит админ |
|---|---|---|
| По умолчанию | Есть слоты в периоде | Сетка по дням со статусами и заполненностью, кнопка создания |
| Загрузка | Данные периода запрашиваются | Скелетон сетки; навигация по датам не скачет |
| Пусто | В периоде нет слотов (типично — следующая неделя не спланирована) | Не «ошибка», а приглашение: «на эту неделю классов пока нет» + кнопка создать слот из пустого состояния |
| Ошибка | Расписание не загрузилось | Спокойное сообщение + «повторить», без технических деталей |
| Слот отменён студией | `status = CANCELLED_BY_STUDIO` | Карточка на месте, приглушена/маркирована «отменён»; отмена из неё недоступна; никаких намёков на «восстановить» (ФТ-05) |
| Слот завершён | `status = COMPLETED` (автоматически по времени — Р-10) | «Архивная» подача; единственное действие — открыть список броней для снятия ошибочной неявки (Р-02) |
| Снятие отметки неявки | Админ открыл брони завершённого класса | Список гостей класса с отметками (`GET /slots/{slotId}/bookings`); снятие — осознанное действие с подтверждением; после — «отметка снята», счётчик и блокировка пересчитаны |
| Слот полон | `freeSpots = 0` | «Полный» читается с одного взгляда — позитивная подача |
| Смешанный день | В одном дне живые, полные и отменённые классы | Статусы различимы, не только цветом (проверка на дальтонизм) |

Форма сетки (календарь/лента/список), визуальный код статусов, подача списка броней
завершённого класса (панель/отдельный экран) — решает дизайнер (бриф SCR-13 §9).

### 3.2 SCR-14 — Слот: форма создания

Элементы (приоритеты — бриф SCR-14 §4):

| # | Элемент | Правило | Поле API |
|---|---|---|---|
| 1 | Выбор программы | Из каталога; в выборе показывается признак сложного оборудования — лимит понятен до выбора | `programId` |
| 2 | Выбор шефа | Список шефов студии небольшой; выбор «по лицам» приятнее списка фамилий | `chefId` |
| 3 | Дата и время начала | Ближайшие даты — на расстоянии одного действия; возможно предзаполнение «из дня» | `startAt` |
| 4 | Длительность — 180 минут | **Справка, не поле ввода** (ФТ-03); полезно показывать время окончания («18:00–21:00») | не передаётся |
| 5 | Вместимость | 12 по умолчанию; при программе со сложным оборудованием форма сама ставит 8 и не даёт поднять выше, с коротким объяснением (ФТ-02) | `capacityTotal` |
| 6 | Прокатные наборы | Число от нуля; 0 — легальный случай «прокат не предусмотрен» | `rentalSetsAvailable` |
| 7 | Цена проката за набор | Справочная цена для клиента; никакого чекаута за ней не стоит (БТ-08); обязательна при наборах > 0 | `rentalPricePerSet` |
| 8 | «Создать» / «Отмена» | Одна акцентная кнопка создания; отмена — возврат в SCR-13 без сохранения | — |

Состояния (бриф SCR-14 §6):

| Состояние | Когда | Что видит админ |
|---|---|---|
| По умолчанию | Чистая форма | Пустые выборы программы/шефа, вместимость 12, длительность 180 как справка; возможно предзаполненная дата |
| Загрузка | Едут списки программ/шефов или идёт сохранение | Скелетоны / заблокированная кнопка с индикатором; повторный клик не создаёт дубль слота |
| Пусто (нет программ) | Каталог программ пуст | «Сначала нужна программа» + переход в SCR-12; не пустой выпадающий список |
| Пусто (нет шефов) | Нет ни одного шефа | Подсказка завести шефа (экран «Шефы») вместо тупика |
| Выбрана программа со сложным оборудованием | `requiresComplexEquipment = true` | Вместимость сама становится 8 (максимум 8) + объяснение «у программы сложное оборудование — до 8 мест»; если вписано больше — корректируется явно, с той же подсказкой |
| Смена программы обратно на обычную | Выбрана программа без сложного оборудования | Потолок возвращается к 12; значение не «прыгает» молча |
| Ошибка (валидация) | Не выбрана программа/шеф, не задано время, вместимость вне пределов | Подсветка конкретного поля, человеческий текст; введённое не теряется |
| Пересечение слотов шефа | Шеф занят в это время (API 409, D-10) | Форма не сохраняет: понятная ошибка **с указанием конфликтующего класса** (программа, дата, время); данные формы целы |
| Ошибка (сохранение) | Сеть/сервер не приняли слот | «Не получилось создать» + повторить; данные формы целы |
| Дата в прошлом | Выбрано уже прошедшее время | Форма мягко не даёт создать класс «во вчера» (бриф SCR-14 §6; серверное правило — см. §6) |

Лимит вместимости применяется **по текущему признаку программы на момент создания слота**;
существующих слотов смена признака не касается (Р-04, примечание к ФТ-02).

**Оставлено на усмотрение разработки** (журнал решений, раздел «Оставлены на усмотрение», бриф SCR-14 §9) —
пробелы источников, требований нет: шаг сетки времени начала (любое время или кратно 30 минутам),
нижняя граница вместимости (в API — `minimum: 1`), ограничение прокатного фонда вместимостью слота
(наборов не больше, чем мест).

### 3.3 SCR-15 — Отмена слота (диалог)

Единственный вход — действие «отменить» у конкретного класса в SCR-13 (бриф SCR-15 §3).

Элементы (приоритеты — бриф SCR-15 §4):

| # | Элемент | Правило |
|---|---|---|
| 1 | Что именно отменяется | Название программы, дата и время, шеф — убедиться, что это тот самый класс |
| 2 | Сколько гостей затронуто | Число активных записей («Записаны 9 гостей — все получат письмо об отмене»); заметная цифра, не сноска; если записей нет — прямо «на класс пока никто не записан» |
| 3 | Поле причины отмены | Обязательный текст, уходит гостям в письме; максимум 300 символов (Р-16 п.3); у поля пояснение «эту причину увидят гости»; счётчик символов — на усмотрение дизайнера |
| 4 | Кнопка подтверждения | Визуально деструктивная; текст про суть («Отменить класс и уведомить гостей»), не «ОК» |
| 5 | Путь к отступлению | Закрыть диалог без последствий; заметен, но не конкурирует с подтверждением |
| 6 | Пояснение необратимости | Класс нельзя «включить обратно», запись закроется навсегда (ФТ-05, D-04); одна спокойная строка |

Что сделать нельзя (бриф SCR-15 §5): отменить без причины (кнопка не срабатывает при пустом
поле — ФТ-04); перенести класс (переноса в системе нет — Р-03); вернуть класс после
подтверждения (необратимо — ФТ-19); выбрать, кого уведомлять (письма уходят всем автоматически,
варианта «отменить втихую» не существует намеренно — ФТ-15/D-07).

Состояния (бриф SCR-15 §6):

| Состояние | Когда | Что видит админ |
|---|---|---|
| По умолчанию | Диалог открыт для класса с записями | Сводка класса, число затронутых гостей, пустое поле причины, подтверждение неактивно до ввода причины |
| Без записей | На класс никто не записан | Тот же диалог легче по тону: «никто не записан, письма отправлять некому»; **причина всё равно обязательна** — останется в истории |
| Причина не заполнена | Попытка подтвердить с пустым полем | Мягкая подсказка у поля; кнопка не срабатывает |
| Загрузка | Операция выполняется | Кнопка в ожидании, повторное нажатие невозможно, диалог не закрыть случайно |
| Успех | Класс отменён (API 200) | «Класс отменён, гостям отправляются письма»; возврат в SCR-13, класс виден отменённым. Статус доставки писем не показывается и не обещается — best-effort, отчёта о доставке в MVP нет (НФТ-03, Р-16 п.5) |
| Ошибка операции | Сеть/сбой | «Класс НЕ отменён» — явно; причина сохранена в поле; «Попробовать ещё раз» |
| Причина длиннее 300 символов | Текст сверх лимита | Мягкое ограничение у поля; текст не обрезается молча (Р-16 п.3) |
| Класс уже нельзя отменить | Слот уже отменён или завершён (API 409; напр., устаревшая вкладка) | Диалог объясняет текущий статус и не предлагает подтверждение; выход в расписание |

## 4. Контракт API

Все операции модуля требуют роль ADMIN (кроме `GET /slots`, доступного и без входа, и
`GET /slots/{slotId}/bookings`, доступного также шефу своего слота). Схема ошибки везде —
`Error { code, message, details? }`.

### 4.1 POST /slots — создать слот (SCR-14)

Запрос `SlotCreateRequest` (ФТ-03; **длительность фиксированная 180 минут и в запросе не передаётся**):

| Поле | Тип | Обязательность | Правило |
|---|---|---|---|
| `programId` | uuid | обязательное | Программа из каталога |
| `chefId` | uuid | обязательное | Шеф студии |
| `startAt` | date-time | обязательное | Время начала |
| `capacityTotal` | integer ≥ 1 | обязательное | Не выше лимита программы: 12, либо 8 при `requiresComplexEquipment` (ФТ-02) |
| `rentalSetsAvailable` | integer ≥ 0 | нет (default 0) | 0 — «прокат не предусмотрен» |
| `rentalPricePerSet` | number ≥ 0 | **обязательна при `rentalSetsAvailable > 0`** | Справочная цена проката (БТ-08, D-06) |

Ответы и реакция UI:

| Код | Когда | Реакция UI |
|---|---|---|
| `201` | Слот создан, тело — `SlotDetails` | Возврат в SCR-13, новый слот виден на своём месте (бриф SCR-14 §3); слот сразу доступен клиентам (US-09) |
| `400 ValidationError` | Невалидные поля (вместимость выше лимита, нет цены при наборах > 0 и т.п.) | Подсветка конкретного поля, человеческий текст; введённое не теряется (бриф SCR-14 §6). Клиентская валидация обязана отработать раньше — ФТ-02 требует ограничения «в момент выбора программы», не постфактум |
| `401 Unauthorized` | Нет/истекла сессия | Переход на вход (SCR-01/02) |
| `403 Forbidden` | Роль не ADMIN | Отказ в доступе к разделу администратора |
| `409` | Пересечение по времени со слотом того же шефа в `SCHEDULED` (D-10, ФТ-24); код `CHEF_SLOT_OVERLAP`, в `details.conflictingSlotId` — конфликтующий слот | Состояние «Пересечение слотов шефа»: по `conflictingSlotId` показать конфликтующий класс — программу, дату и время (бриф SCR-14 §6, §7); данные формы целы |

Повторный клик по «Создать» во время сохранения не создаёт дубль (бриф SCR-14 §6, состояние «Загрузка»).

### 4.2 POST /slots/{slotId}/cancel — отменить слот (SCR-15)

Запрос `SlotCancelRequest`:

| Поле | Тип | Обязательность | Правило |
|---|---|---|---|
| `reason` | string, 1–300 символов | обязательное | Причина отмены (ФТ-04); лимит 300 — Р-16 п.3; текст попадает в email клиентам (ФТ-15) |

Ответы и реакция UI:

| Код | Когда | Реакция UI |
|---|---|---|
| `200` | Слот отменён, тело — `SlotDetails`. Слот → `CANCELLED_BY_STUDIO` с `cancellationReason`; связанные подтверждённые брони **каскадно** → `CANCELLED_BY_STUDIO`; email с причиной поставлены в отправку (UC-03, ФТ-15, D-07). Отправка best-effort: сбой почты не влияет на исход операции (НФТ-03); отчёта о недоставке нет (Р-16 п.5) | Состояние «Успех»: «класс отменён, гостям отправляются письма» — только факт отправки, без обещания доставки; возврат в SCR-13, слот виден отменённым |
| `400 ValidationError` | Причина пуста или длиннее 300 символов | Подсказка у поля; кнопка и так неактивна при пустой причине (клиентская валидация — бриф SCR-15 §6) |
| `401 Unauthorized` | Нет/истекла сессия | Переход на вход |
| `403 Forbidden` | Роль не ADMIN | Отказ в доступе |
| `404 NotFound` | Слот не существует | Сообщение об ошибке + возврат в расписание |
| `409` | Слот уже не в `SCHEDULED` — отменён ранее или завершён автозавершением (D-09) | Состояние «Класс уже нельзя отменить»: объяснить текущий статус, подтверждение не предлагать, выход в расписание (бриф SCR-15 §6) |

### 4.3 GET /slots — сетка расписания (SCR-13)

Тот же метод, что у клиентской витрины; для админа — с параметром `status` (описание операции
`listSlots` в openapi.yaml: «ADMIN использует тот же метод для сетки SCR-13, включая отменённые
и завершённые — параметр `status`»).

| Параметр | Тип | Правило |
|---|---|---|
| `dateFrom` / `dateTo` | date | Период сетки; по умолчанию — текущий момент / +7 дней. Навигация в прошлое без ограничения глубины (Р-16 п.4) |
| `status` | `SlotStatus` | Фильтр по статусу; правило «только `SCHEDULED` по умолчанию» в описании API отнесено к гостю и клиенту — админской сетке нужны все три статуса |

Ответ `200` — массив `SlotSummary`, отсортирован по времени начала. `400 ValidationError` —
некорректные параметры дат → состояние «Ошибка» сетки.

> **Пробел источника:** параметр `status` принимает одно значение, а сетке SCR-13 нужны все три
> статуса разом; явного описания выдачи для ADMIN без параметра (все статусы или только
> `SCHEDULED`) в openapi.yaml нет. Рекомендация: для роли ADMIN запрос без `status` возвращает
> слоты всех статусов — это согласуется с формулировкой «гостю и клиенту по умолчанию отдаются
> только `SCHEDULED`». Требует подтверждения при реализации.

### 4.4 GET /slots/{slotId}/bookings — брони слота (вход из SCR-13)

Админ использует список для снятия ошибочной отметки `NO_SHOW` (ФТ-22, описание операции
`listSlotBookings`); тот же метод служит шефу для его класса (SCR-10 — вне этого модуля).
Тексты отзывов в ответ не входят (Р-07).

| Код | Когда | Реакция UI |
|---|---|---|
| `200` | Массив `ParticipantBooking { id, clientName, allergyNote?, status, equipmentChoice }` | Список гостей класса с отметками (состояние «Снятие отметки неявки», бриф SCR-13 §6); действие «снять неявку» доступно только у броней `status = NO_SHOW` |
| `401 Unauthorized` | Нет сессии | Переход на вход |
| `403` | Роль CLIENT либо шеф чужого слота | Отказ в доступе |
| `404 NotFound` | Слот не существует | Сообщение + возврат в сетку |

### 4.5 DELETE /bookings/{bookingId}/no-show — снять ошибочную отметку неявки (ADMIN)

Операция `revokeNoShow` (ФТ-22, D-11, UC-06/A1). Тела запроса нет.

| Код | Когда | Реакция UI |

## 6. Краевые случаи

| Случай | Поведение | Источник / статус |
|---|---|---|
| Отмена слота с 0 броней | Диалог тот же, легче по тону («никто не записан, письма отправлять некому»); **причина всё равно обязательна** — останется в истории; каскад и письма вырождаются в пустые множества | Бриф SCR-15 §6 («Без записей»); ФТ-04 |
| Отмена за минуты до начала | Разрешена: единственное предусловие отмены — слот в `SCHEDULED` (UC-03); минимального срока до начала источники не задают. После автозавершения (`startAt + 180 мин`) — уже 409 | UC-03, D-09; ограничения «не позже чем за N часов» в источниках **нет** — считается осознанным отсутствием |
| Отмена наперегонки с автозавершением / из устаревшей вкладки | API 409 (слот не в `SCHEDULED`); диалог объясняет текущий статус и не предлагает подтверждение | openapi.yaml (`cancelSlot` 409, D-09); бриф SCR-15 §6 |
| Создание слота в прошлом | UI: форма «мягко не даёт создать класс во вчера» (бриф SCR-14 §6). **Пробел источника:** серверное правило (отклонять ли `startAt` в прошлом и каким кодом) не зафиксировано ни в ФТ/D, ни в openapi.yaml. Рекомендация: запретить и на сервере — 400 `ValidationError`; требует подтверждения | Бриф SCR-14 §6 (только UI); серверная часть — **открытый вопрос** |
| Пересечение ровно впритык (конец одного = начало другого, напр. 15:00–18:00 и 18:00–21:00) | **Пробел источника:** D-10 запрещает «пересекающиеся по времени» слоты, но случай касания границ не определён. Рекомендация: считать интервалы полуоткрытыми `[startAt, startAt + 180 мин)` — касание границ пересечением **не** является, слоты подряд разрешены; требует подтверждения | D-10, ФТ-24 — формулировка не покрывает случай; **открытый вопрос** |
| Дубль отправки формы создания | Повторный клик при сохранении не создаёт второй слот | Бриф SCR-14 §6 («Загрузка») |
| Снятие `NO_SHOW` с брони не в `NO_SHOW` (двойное снятие, гонка вкладок) | API 409; UI сообщает, что отметки уже нет, и перезагружает список | openapi.yaml (`revokeNoShow` 409) |
| Снятие `NO_SHOW`, когда у клиента активна блокировка | Блокировка снимается, если после декремента условие (`lateCancelCount ≥ 3`) перестаёт выполняться — пересчёт на стороне системы, UI ничего не вычисляет | D-11, UC-06/A1 |
| Отменённый слот в сетке | Не прячется, приглушён; действий нет (ни отмены, ни «восстановить») | Бриф SCR-13 §6; ФТ-05 |
| Нет программ / нет шефов при открытии SCR-14 | Не тупик: подсказка сначала завести программу (SCR-12) / шефа | Бриф SCR-14 §6 |

## 7. Критерии приёмки

**КП-1. Лимит 8 при сложном оборудовании (ФТ-02, Р-04)**
Дано: программа с `requiresComplexEquipment = true`; админ в форме SCR-14 уже ввёл вместимость 12.
Когда: он выбирает эту программу.
Тогда: поле вместимости само становится 8 (максимум 8) с видимым объяснением «у программы сложное оборудование — до 8 мест»; поднять выше 8 форма не позволяет; при отправке `POST /slots` с `capacityTotal ≤ 8` сервер отвечает 201; ранее созданные слоты этой программы вместимость не меняют.

**КП-2. 409 при пересечении слотов шефа (ФТ-24, D-10)**
Дано: у шефа X существует слот `SCHEDULED` на 18:00–21:00 (180 мин).
Когда: админ создаёт слот для шефа X с `startAt = 19:00` того же дня (`POST /slots`).
Тогда: сервер отвечает 409 с кодом `CHEF_SLOT_OVERLAP` и `details.conflictingSlotId` существующего слота; форма показывает конфликтующий класс — программу, дату и время (не абстрактное «время занято»); введённые данные формы сохранены.

**КП-3. Отмена с каскадом и письмами (UC-03, ФТ-04, ФТ-15, НФТ-03)**
Дано: слот `SCHEDULED` с тремя бронями `CONFIRMED`; диалог SCR-15 показывает «записаны 3 гостя».
Когда: админ вводит причину (1–300 символов) и подтверждает (`POST /slots/{id}/cancel`).
Тогда: ответ 200; слот — `CANCELLED_BY_STUDIO` с сохранённой `cancellationReason`; все три брони — `CANCELLED_BY_STUDIO`; каждому из трёх клиентов поставлен в отправку email с текстом причины; сбой почтового провайдера не меняет ни один из этих исходов; UI сообщает факт отправки, не обещая доставку; в сетке SCR-13 слот виден отменённым.

**КП-4. Недоступность брони после отмены (ФТ-05, D-04)**
Дано: слот отменён студией (КП-3).
Когда: клиент отправляет `POST /bookings` с `slotId` этого слота.
Тогда: сервер отвечает 410 с `details.cancellationReason`; бронь не создаётся; никакой операции «восстановить слот» не существует ни в API, ни в UI.

**КП-5. Снятие NO_SHOW с пересчётом блокировки (ФТ-22, D-11)**
Дано: завершённый слот; у брони клиента статус `NO_SHOW`, его `lateCancelCount = 3` и выставлен `blockedUntil` в будущем.
Когда: админ из сетки SCR-13 открывает брони слота (`GET /slots/{id}/bookings`), подтверждает снятие отметки и выполняется `DELETE /bookings/{id}/no-show`.
Тогда: ответ 200; статус брони — `COMPLETED`; `lateCancelCount = 2`; блокировка снята (условие `≥ 3` перестало выполняться) — клиент снова может бронировать; UI подтверждает «отметка снята»; повторный `DELETE` на ту же бронь возвращает 409.

**КП-6. Причина отмены обязательна и ограничена (ФТ-04, Р-16 п.3)**
Дано: открыт диалог SCR-15 для слота `SCHEDULED`.
Когда: поле причины пусто — либо содержит 301+ символов.
Тогда: подтверждение не срабатывает (пустое поле) / у поля видимое ограничение без молчаливого обрезания (превышение); запрос с пустой или сверхлимитной причиной сервер отклоняет 400.

**КП-7. Слот нельзя отменить повторно или после завершения (UC-03, D-09)**
Дано: слот в статусе `CANCELLED_BY_STUDIO` или `COMPLETED`; диалог SCR-15 открыт из устаревшей вкладки.
Когда: админ подтверждает отмену (`POST /slots/{id}/cancel`).
Тогда: сервер отвечает 409; диалог объясняет текущий статус класса и не предлагает подтверждение; никаких изменений слота и броней не происходит.

**КП-8. В сетке нет запрещённых действий (Р-03, Р-10, ФТ-23)**
Дано: сетка SCR-13 с живым, отменённым и завершённым слотами.
Когда: админ рассматривает карточки слотов.
Тогда: ни на одной карточке нет редактирования («карандаша»), кнопки «завершить» и отмены «в один клик»; отмена доступна только переходом в SCR-15 у слота `SCHEDULED`; у завершённого слота единственное действие — открыть список броней.
