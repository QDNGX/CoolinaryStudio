## 1. Назначение и охват

Модуль «Ядро бэкенда» реализует серверную часть веб-приложения «Шеф-стол» (БТ-01: единое
веб-приложение, фронт + бэкенд на **Spring Boot**, без стороннего бэкенда-«чёрного ящика»;
НФТ-01). Три роли — CLIENT / CHEF / ADMIN — обслуживаются одним приложением и одним
passwordless-механизмом входа (ФТ-13, БТ-06).

В охвате модуля:

- **доменная модель и схема БД** — сущности User, ClientProfile, ChefProfile, StudioSettings,
  Program, Slot, Booking, Review + TTL-хранилище кодов подтверждения (раздел 2);
- **матрица доступа** «кто читает / кто меняет» и производные поля (раздел 3);
- **инварианты D-01…D-11** и серверная логика операций API (раздел 4);
- **фоновые процессы** — автозавершение слотов (UC-10) и email-напоминания (UC-09) (раздел 5);
- **email-уведомления** EML-01…EML-04 (раздел 6);
- **безопасность и сессии** — bearer-токен, роли, лимиты кода (раздел 7).

Вне охвата: все экраны и клиентская логика (tz-01…tz-10); онлайн-оплата, программа лояльности,
web push (БТ-08 — вне скоупа MVP).

Выбор конкретных технологий сверх зафиксированного стека (Spring Boot — БТ-01/НФТ-01) —
СУБД, планировщик, SMTP-провайдер, механизм токена — этим документом **не навязывается**:
там, где источники не отвечают, стоит пометка «на усмотрение разработки» с рекомендацией.

## 2. Модель данных

Схема — по ER-модели ([er-model-cooking.md](../analytics/design/er-model-cooking.md)); типы —
по доменной модели; лимиты и форматы сверены с [openapi.yaml](../api/openapi.yaml).
Все первичные ключи — UUID. Именование таблиц/колонок — на усмотрение разработки.

### 2.1 User

Базовая учётная запись всех трёх ролей. Аккаунт клиента создаётся молчаливо при первом входе
(ФТ-21, Р-01); аккаунт шефа заводит только администратор (D-08, ФТ-14); паролей нет ни у кого
(БТ-06) — поля `passwordHash` в модели не существует.

| Поле | Тип | Ограничения | Комментарий |
|---|---|---|---|
| `id` | UUID | PK | |
| `role` | enum `CLIENT` \| `CHEF` \| `ADMIN` | not null | ФТ-19 (роли), матрица доступа |
| `name` | string ≤ 100 | nullable | у нового клиента пусто до ответа «Как к вам обращаться?» (Р-12, ФТ-21); лимит 100 — из `UpdateMeRequest`/`ChefCreateRequest` openapi.yaml |
| `email` | string, формат email | **unique**, not null | логин (ФТ-13) и канал уведомлений (D-07); в MVP не редактируется никем (Р-16 п. 1) |
| `enabled` | boolean | not null | |
| `createdAt` | datetime | not null | |

Слот расписания — самая «горячая» сущность: на ней живут оптимистичная блокировка и кэш мест
(D-03). Редактирования слота нет — исправление = отмена + создание заново (Р-03).

|---|---|---|---|

| `startAt` | datetime | not null | |
| `durationMinutes` | int | not null, **фиксированно 180** | не передаётся в API, не настраивается (ФТ-03) |

| `status` | enum `SCHEDULED` \| `CANCELLED_BY_STUDIO` \| `COMPLETED` | not null | ФТ-19; переходы — см. §4 (D-04, D-09) и ФТ-04 |

| `rentalPricePerSet` | decimal | ≥ 0; обязательна при `rentalSetsAvailable > 0` (openapi `SlotCreateRequest`) | текущая цена; в брони фиксируется снапшотом (D-06) |

других переходов нет.

|---|---|---|---|
| `id` | UUID | PK | |

### 2.9 Код подтверждения входа (TTL-хранилище)

**Сознательно не доменная сущность** (ER-модель, НФТ-05): короткоживущая запись с TTL.
Конкретный механизм хранения — таблица с очисткой по расписанию **или** кэш — **на усмотрение
разработки** (НФТ-05 явно оставляет выбор реализации).

Хранимые данные должны обеспечивать проверки НФТ-04/Р-09:

| Данные | Зачем |
|---|---|
| email, на который выдан код | сопоставление при `POST /auth/code/verify` |
| код (шестизначный, ФТ-13) | проверка соответствия; хранение в хэшированном виде — рекомендация, на усмотрение разработки |
| момент выдачи / срок действия | TTL 10 минут (НФТ-04) |
| счётчик попыток ввода | ≤ 5 попыток (НФТ-04); неверный ввод инкрементирует счётчик |
| момент последней отправки | повторная отправка не чаще 1 раза в 60 секунд (НФТ-04) → 429 + `Retry-After` |

Рекомендация (не требование): таблица в основной БД с индексом по email и периодической
очисткой просроченных записей тем же планировщиком, что и фоновые процессы §5, — без внешнего
кэша, в духе минимизации зависимостей (НФТ-06).

## 3. Матрица доступа: кто читает / кто меняет

Перенесено из ER-модели ([er-model-cooking.md, «Матрица доступа»](../analytics/design/er-model-cooking.md#матрица-доступа-кто-читает--кто-меняет));
бэкенд обязан проверять эти права на каждой операции (см. §7). Обозначения: **C** — создаёт,
**U** — изменяет, **R** — читает, **—** — не имеет доступа. «Система» — автоматические переходы
без действия пользователя (транзакции, планировщик).

| Сущность | Клиент | Шеф | Админ | Система | Комментарий |
|---|---|---|---|---|---|
| User (свой) | R; U только `name` при первом входе | R | C (для шефа, D-08) / R | C (для клиента, ФТ-21) | email никем не редактируется (Р-16) |
| ClientProfile | R; U только `allergyNote` | R (`allergyNote` участников) | R | U (`lateCancelCount`, `blockedUntil` — D-02, D-11) | дисциплинарные поля — только система |
| ChefProfile | R (карточка класса) | R (свой) | C/U (`photo`, `bio` — Р-05) | U (`averageRating`, `reviewsCount`) | агрегаты не меняет никто из ролей |
| StudioSettings | R | R | U (ФТ-17) | — | синглтон |
| Program | R | R | C/U (ФТ-01) | — | для клиента и шефа — только чтение |
| Slot | R | R (свои) | C (ФТ-03); U только отмена (ФТ-04) | U (`freeSpots`/`version` в транзакции брони — D-03; `status → COMPLETED` — D-09) | редактирования нет (Р-03) |
| Booking | C (бронь) / U (отмена своей) / R (свои) | R (брони своих классов); U только `NO_SHOW` (ФТ-10) | R; U только снятие `NO_SHOW` (D-11) и каскад отмены слота | U (`COMPLETED` — D-09; каскад `CANCELLED_BY_STUDIO`) | каждый актор меняет строго свой кусок жизненного цикла |
| Review | C (D-05) / R (свой, read-only) | R только агрегата на профиле | R (включая `comment` — Р-07) | — (агрегат пересчитывает при создании) | текст комментария — только админу |

**Производные поля меняет только система** — ни одна роль напрямую, ни через какой эндпоинт
(в openapi.yaml они помечены `readOnly` и отсутствуют во входных схемах):

| Поле | Инвариант / правило |
|---|---|
| `Slot.freeSpots`, `Slot.version` | D-03 — транзакция брони/отмены |
| `ClientProfile.lateCancelCount`, `ClientProfile.blockedUntil` | D-02, D-11 |
| `ChefProfile.averageRating`, `ChefProfile.reviewsCount` | ФТ-12 — пересчёт при создании отзыва |
| `Booking.isLateCancellation`, `Booking.rentalPriceSnapshot` | D-01, D-06 |
| статусы `COMPLETED` (Slot и Booking) | D-09 |

Бэкенд обязан игнорировать/отклонять попытки записи этих полей во входных запросах.

## 4. Инварианты и серверная логика

По каждому инварианту: формулировка (из [constraints-and-assumptions-cooking.md](../analytics/requirements/constraints-and-assumptions-cooking.md)),
где срабатывает (операция API из [openapi.yaml](../api/openapi.yaml) или процесс §5),
как реализуется.

### D-01 — порог бесплатной отмены

- **Формулировка:** отмена бесплатна, если `slot.startAt − now() ≥ 6ч`; иначе (включая отмену
  после старта) — `isLateCancellation = true`.
- **Где:** `POST /bookings/{bookingId}/cancel` (UC-02, ФТ-08).
- **Как:** система фиксирует `cancelledAt = now()`, вычисляет
  `isLateCancellation = slot.startAt − cancelledAt < 6ч`, переводит `status → CANCELLED_BY_CLIENT`,
  возвращает `isLateCancellation` в ответе. Клиент значение не передаёт (derived, §3).

### D-02 — лимит поздних отмен, с периодическим сбросом

- **Формулировка:** `lateCancelCount` растёт на 1 при поздней отмене или `NO_SHOW`; при
  `lateCancelCount ≥ 3` — `blockedUntil = now() + 7 дней`, новые брони блокируются до этой
  даты; по наступлении `blockedUntil` блок снимается и счётчик обнуляется до 0 (ФТ-09).
- **Где срабатывает:**
  - инкремент — `POST /bookings/{bookingId}/cancel` при `isLateCancellation = true` (UC-02 шаг 5)
    и `POST /bookings/{bookingId}/no-show` (UC-06 шаг 2);
  - выставление блока — там же, сразу после инкремента, если счётчик достиг 3;
  - проверка блока — `POST /bookings` (предусловие UC-01): при `now() < blockedUntil` —
    **403** с датой разблокировки (`BlockedError.blockedUntil`);
  - сброс — по наступлении `blockedUntil` (см. ниже).
- **Как:** инкремент и выставление блока — в той же транзакции, что и смена статуса брони
  (обе величины — производные, меняет только система). Клиент видит счётчик и дату разблокировки
  в `GET /me` (Р-06 — блокировка не должна быть сюрпризом), но не меняет их.
- **Момент сброса — открытый вопрос.** Источники (D-02, ФТ-09, UC-02) фиксируют *что*
  происходит («по наступлении `blockedUntil` блок снимается и счётчик обнуляется»), но не
  *каким механизмом* — лениво при проверке или планировщиком. **Рекомендация:** ленивый сброс —
  при каждой проверке блокировки (создание брони, выдача `GET /me`) если `now() ≥ blockedUntil`,
  то в той же транзакции `blockedUntil := null`, `lateCancelCount := 0`. Это не требует
  отдельного планировщика и не создаёт окна, где клиент видит устаревший блок. Допустима и
  комбинация с периодическим планировщиком §5 — на усмотрение разработки. Наблюдаемое
  поведение в любом варианте обязано соответствовать КП-5 (§9).

### D-03 — запрет двойной брони

- **Формулировка:** 0 двойных броней; `@Version` на `Slot` (оптимистичная блокировка) +
  пересчёт `freeSpots` внутри той же транзакции, что и вставка `Booking`; при конфликте
  версий — retry или явный отказ (НФТ-02, БТ-02, ФТ-07).
- **Где:** `POST /bookings` (`createBooking`, UC-01) — и симметрично возврат места при отменах
  (отмена клиентом/каскад отмены слота увеличивают `freeSpots` в той же транзакции — «кэш,
  пересчитывается транзакционно при брони/отмене», доменная модель).
- **Как** — строго по [sequence-диаграмме](../analytics/design/sequence-create-booking-cooking.md):
  1. Транзакция: чтение слота (`status`, `freeSpots`, `rentalSetsAvailable`,
     `rentalPricePerSet`, `version`).
  2. `status = CANCELLED_BY_STUDIO` → ROLLBACK, **410** (D-04, см. ниже).
  3. `freeSpots = 0` (или `RENTAL` при `rentalSetsAvailable = 0`) → ROLLBACK, **409**
     «места закончились».
  4. Иначе: при `RENTAL` — `rentalPriceSnapshot := slot.rentalPricePerSet` (D-06),
     `rentalSetsAvailable − 1`; `INSERT booking (CONFIRMED)`;
     `UPDATE slot SET freeSpots = freeSpots − 1, version = version + 1 WHERE id = :slotId AND version = :readVersion`.
  5. 1 строка обновлена → COMMIT, **201**. 0 строк (конфликт версии — параллельная бронь
     успела раньше) → ROLLBACK и **ровно один retry**: перечитать слот и повторить; если мест
     уже нет или версия снова ушла — **409** «место только что заняли». Бесконечных циклов и
     «тихих» ошибок нет (НФТ-02).
- Письма-напоминания в транзакцию не входят (D-07) — планируются процессом §5.2 и на исход
  `createBooking` не влияют.

### D-04 — запрет записи на отменённый слот

- **Формулировка:** слот `CANCELLED_BY_STUDIO` недоступен для новой брони (ФТ-05).
- **Где:** `POST /bookings`.
- **Как:** проверка статуса внутри транзакции брони (шаг 2 D-03) → **410 Gone** с причиной
  отмены в `details.cancellationReason`. 410, а не 409: состояние безвозвратно, ретраить
  бессмысленно (sequence-диаграмма, «Почему 409 и 410 — разные коды»).

### D-05 — момент доступности отзыва

- **Формулировка:** `Review` создаётся только при `slot.status = COMPLETED` и неотменённой
  брони; один отзыв на бронь (ФТ-11).
- **Где:** `POST /bookings/{bookingId}/review` (UC-04).
- **Как:** проверки — бронь принадлежит клиенту (иначе 403); `slot.status = COMPLETED` и бронь
  не отменена (иначе **422**, UC-04/A1); отзыва ещё нет (иначе **409**, UC-04/A2). Уникальность
  на уровне БД — unique-индекс по `Review.bookingId` (§2.8) как последняя линия защиты от
  гонки. После создания в той же операции пересчитываются `averageRating`/`reviewsCount` шефа
  (ФТ-12, UC-04 шаг 4). Отзыв не редактируется (Р-16 п. 6) — операций UPDATE/DELETE нет.

### D-06 — снапшот цены проката

- **Формулировка:** `rentalPriceSnapshot` фиксируется при создании брони и не пересчитывается
  задним числом (НФТ-07, БТ-07, ФТ-06).
- **Где:** `POST /bookings` при `equipmentChoice = RENTAL` (шаг 4 транзакции D-03).
- **Как:** копирование `slot.rentalPricePerSet` в момент вставки брони; ни одна последующая
  операция (в т.ч. смена цены на слоте — которой в MVP и нет, Р-03) поле не трогает; в API
  поле `readOnly`.

### D-07 — уведомления по email

- **Формулировка:** отмена студией и напоминания (за 24ч и за 2ч до `startAt`) уходят на
  `User.email`; сбой отправки — best-effort, не блокирует и не откатывает транзакции и не
  влияет на другие письма (НФТ-03).
- **Где и как:** раздел 6 (триггеры, получатели, содержимое) и §5.2 (процесс напоминаний).

### D-08 — заведение аккаунтов Chef

- **Формулировка:** шеф не регистрируется сам — только Admin создаёт `User(role=CHEF)` по
  email (ФТ-14, UC-08).
- **Где:** `POST /chefs` (только ADMIN).
- **Как:** создаётся `User(role=CHEF)` + `ChefProfile`; пароль не создаётся — первый вход
  через тот же passwordless-код (UC-05). Email уже занят → **409** (кейс «email занят
  клиентом» в MVP решается вручную другим email — Р-16). Молчаливое создание аккаунта при
  входе (ФТ-21) действует **только** для роли CLIENT: неизвестный email на входе — всегда
  новый клиент; шеф/админ, не заведённый заранее, доступ к своей роли не получает (UC-05).

### D-09 — автозавершение слота

- **Формулировка:** по наступлении `startAt + durationMinutes` (180 мин) слот автоматически
  переходит в `COMPLETED`; его брони в `CONFIRMED` — тоже в `COMPLETED`; отметки `NO_SHOW`
  сохраняются; ручного «завершить» у Admin нет (Р-10, ФТ-23).
- **Где и как:** фоновый процесс §5.1 (UC-10).

### D-10 — запрет пересечения слотов шефа

- **Формулировка:** нельзя создать слот, пересекающийся по времени с другим слотом того же
  шефа в статусе `SCHEDULED`; при попытке — явный отказ с указанием конфликтующего слота
  (Р-08, ФТ-24).
- **Где:** `POST /slots` (только ADMIN, UC-07/A1).
- **Как:** при создании слота проверяется пересечение интервала `[startAt, startAt + 180 мин)`
  нового слота с интервалами `[startAt, startAt + 180 мин)` всех слотов того же `chefId` в
  статусе `SCHEDULED` (отменённые и завершённые не учитываются). Пересечение найдено →
  **409** с кодом `CHEF_SLOT_OVERLAP` и `details.conflictingSlotId` (openapi.yaml).
  Защита от гонки двух параллельных созданий слотов одного шефа источниками не зафиксирована —
  **на усмотрение разработки** (рекомендация: проверка внутри транзакции создания).

### D-11 — снятие ошибочной неявки

- **Формулировка:** Admin может снять `NO_SHOW` (бронь → `COMPLETED`); `lateCancelCount − 1`;
  если условие блокировки перестаёт выполняться — `blockedUntil` снимается (Р-02, ФТ-22).
  У шефа обратного действия нет.
- **Где:** `DELETE /bookings/{bookingId}/no-show` (только ADMIN).
- **Как:** бронь не в `NO_SHOW` → **409**. Иначе в одной транзакции: `status → COMPLETED`,
  `lateCancelCount − 1`; если после декремента `lateCancelCount < 3` — `blockedUntil := null`.

## 5. Фоновые процессы

Оба процесса — актор «Система» матрицы доступа (§3): выполняются планировщиком без действия
пользователя. Конкретный механизм планировщика — на усмотрение разработки
(рекомендация: стандартный `@Scheduled` Spring Boot — без внешних зависимостей, в духе НФТ-06).

### 5.1 Автозавершение слотов (UC-10, D-09, ФТ-23)

- **Условие:** слот в статусе `SCHEDULED`, момент `startAt + durationMinutes` (= `startAt + 180 мин`)
  наступил.
- **Действия (в одной транзакции на слот):**
  1. `Slot.status → COMPLETED`.
  2. Все брони слота в статусе `CONFIRMED` → `COMPLETED`. Брони в `NO_SHOW` **не трогаются** —
     отметка сохраняется (D-09); отменённые брони тоже не трогаются.
- **Следствие:** клиентам с бронями `COMPLETED` открывается возможность оставить отзыв
  (D-05, UC-04); у Admin нет ручного действия «завершить» (Р-10) и нет операции API для этого
  перехода.
- **Идемпотентность:** процесс отбирает только слоты в `SCHEDULED` — повторный прогон по уже
  завершённому слоту ничего не делает.
- **Периодичность запуска** источниками не зафиксирована — **на усмотрение разработки**.
  Рекомендация: раз в 1–5 минут; задержка перехода на величину интервала допустима — точность
  до секунды ни одним требованием не требуется.

### 5.2 Email-напоминания (UC-09, ФТ-16, D-07)

- **Условие:** существует `Booking` в статусе **`CONFIRMED`** (только — UC-09), для которой
  наступил один из двух моментов: за 24 часа или за 2 часа до `slot.startAt`.
- **Действия:** отправить клиенту на `User.email` письмо EML-03 (24ч) или EML-04 (2ч) — §6.
- **Только CONFIRMED на момент отправки:** отменённые (клиентом или студией) и уже завершённые
  брони напоминаний не получают; при отмене слота студией напоминания по его броням
  автоматически перестают подходить под условие (статус уже не `CONFIRMED`).
- **Идемпотентность — письмо не отправляется дважды:** по каждой паре (бронь, тип напоминания
  24ч/2ч) письмо отправляется максимум один раз. Механизм фиксации факта отправки источниками
  не зафиксирован — **на усмотрение разработки**; рекомендация: отметки на брони
  (например, `reminder24hSentAt` / `reminder2hSentAt`) или отдельная таблица отправленных
  напоминаний, проставляемые до/при постановке письма в отправку.
- **Best-effort (D-07, НФТ-03):** сбой отправки одного письма не блокирует и не откатывает
  никакую транзакцию и не влияет на отправку другого напоминания (в т.ч. второго напоминания
  той же брони). Гарантированная доставка не обещана (UC-09, постусловие).
- **Периодичность запуска** — **на усмотрение разработки**; рекомендация: раз в 1–5 минут
  (реже — растёт запаздывание относительно моментов «за 24ч»/«за 2ч»).
- **Пробел источников:** отправлять ли «пропущенное» напоминание за 24ч броне, созданной
  менее чем за 24 часа до начала (аналогично — за 2ч), источники не фиксируют —
  **на усмотрение разработки**. Рекомендация: не отправлять задним числом — клиент только что
  сам создал бронь и видел подтверждение (sequence: экран успеха упоминает оба напоминания;
  формального требования нет).

### 5.3 Очистка TTL-хранилища кодов (НФТ-05)

Если выбран вариант «таблица с очисткой по расписанию» (§2.9) — периодическое удаление
просроченных кодов. Периодичность — на усмотрение разработки; на корректность проверок
НФТ-04 не влияет (срок действия проверяется по данным записи, а не фактом её удаления).

## 6. Email-уведомления

Реестр писем — [screen-registry-cooking.md, раздел EML](../analytics/design-brief/screen-registry-cooking.md#объекты-дизайна-вне-экранов-email-шаблоны).
Вёрстка и тон писем — отдельный дизайн-пакет (там же); здесь — серверные триггеры и
обязательное содержимое.

| ID | Письмо | Триггер (серверный) | Получатель | Обязательное содержимое | Закрывает |
|---|---|---|---|---|---|
| EML-01 | Одноразовый код входа | `POST /auth/code/request` | введённый email | шестизначный код (ФТ-13); срок действия 10 минут (НФТ-04) | ФТ-13, UC-05 |
| EML-02 | Отмена класса студией | `POST /slots/{slotId}/cancel` — каскадная отмена броней (UC-03 шаг 3) | каждый клиент с бронью слота, попавшей под каскад `CONFIRMED → CANCELLED_BY_STUDIO` | **причина отмены** (`cancellationReason` — ФТ-15) + какой класс отменён | ФТ-15, UC-03, D-07 |
| EML-03 | Напоминание за 24 часа | процесс §5.2, момент `startAt − 24ч` | клиент брони `CONFIRMED`, на `User.email` | напоминание о предстоящей записи (ФТ-16): класс и время начала | ФТ-16, UC-09, D-07 |
| EML-04 | Напоминание за 2 часа | процесс §5.2, момент `startAt − 2ч` | клиент брони `CONFIRMED`, на `User.email` | то же | ФТ-16, UC-09, D-07 |

Правила для всех писем:

- **Канал — только email** (`User.email`): он есть всегда, так как это же поле — логин (D-07);
  web push и SMS вне скоупа (БТ-08, НФТ-06).
- **Best-effort (D-07, НФТ-03):** сбой отправки не блокирует и не откатывает породившую
  транзакцию (отмена слота завершается успешно при недоступном SMTP; бронь создаётся
  независимо от судьбы напоминаний) и не влияет на другие письма (сбой EML-03 не отменяет
  EML-04, сбой письма одному клиенту при отмене слота не мешает письмам остальным).
  Формулировка НФТ-03 покрывает EML-02/03/04; для EML-01 API фиксирует асинхронную семантику
  ответом 202 «код отправлен (или будет отправлен)» — сбой отправки кода не раскрывается в
  ответе (§7), обработка сбоя — на усмотрение разработки.
- **Отчёта о недоставке в MVP нет** (Р-16 п. 5): никакого UI/лога недоставленных писем для
  администратора не строится; статус доставки нигде не показывается.
- Рекомендация (не требование): отправку выполнять после коммита транзакции (или из отдельной
  очереди/асинхронно) — это самый простой способ гарантировать «не блокирует и не откатывает».

## 7. Безопасность и сессии

- **Схема аутентификации** — bearer-токен (`securitySchemes.bearerAuth`, openapi.yaml:
  `type: http, scheme: bearer, bearerFormat: JWT`), выдаваемый `POST /auth/code/verify`
  (UC-05). Паролей нет ни у одной роли (БТ-06). Конкретный формат/срок токена: `bearerFormat:
  JWT` — из контракта; длительность сессии — открытый вопрос уровня интерфейса
  (openapi `/auth/logout`, design-decisions), на серверный контракт не влияет;
  **на усмотрение разработки**. `POST /auth/logout` инвалидирует текущий токен (204).
- **Проверка ролей** — на каждой операции по матрице доступа §3; несоответствие роли → 403
  (`Forbidden`), отсутствие/невалидность токена → 401. Дополнительно проверяется
  принадлежность ресурса: клиент отменяет/читает только **свои** брони, шеф видит участников
  и ставит `NO_SHOW` только по **своему** слоту (403 «не шеф этого класса»).
- **Гостевой доступ (Р-16 п. 2):** без токена доступны только `GET /slots`,
  `GET /slots/{slotId}`, `GET /programs`, `GET /programs/{programId}`, `GET /studio-settings`
  и оба `/auth/code/*` (`security: []` в openapi.yaml). Всё остальное — только с токеном.
- **Лимиты кода подтверждения (НФТ-04/Р-09):** срок действия **10 минут**; не более
  **5 попыток** ввода (неверный код → 400 `INVALID_CODE`, попытка учтена; исчерпание →
  400 `ATTEMPTS_EXCEEDED`; истечение → 400 `CODE_EXPIRED` — во всех случаях выход один:
  запросить новый код, UC-05/A1); повторная отправка — не чаще **1 раза в 60 секунд**
  (нарушение → **429** с заголовком `Retry-After`).
- **Не раскрывать существование аккаунта:** ответ `POST /auth/code/request` — всегда **202**
  и не отличается для существующего и несуществующего email (openapi: «неизвестный email не
  является ошибкой — он обрабатывается как вход нового клиента, поэтому ответ не раскрывает,
  существует ли аккаунт»; ФТ-21, Р-01). Различие ролей проявляется только после успешной
  верификации кода: неизвестный email → молчаливое создание `User(role=CLIENT)` +
  `ClientProfile`, `isNewUser: true` (ФТ-21); для CHEF/ADMIN самосоздания нет (D-08).
- **Производные поля** не принимаются во входных данных ни на одном эндпоинте (§3) — попытка
  их передать не должна иметь эффекта.

## 9. Критерии приёмки

Уровень — система (API + БД + процессы), без UI.

**КП-1. Конкурентная бронь — одно место, два запроса (D-03, НФТ-02)**
Дано: слот `SCHEDULED` с `freeSpots = 1`; два аутентифицированных клиента без блокировок.
Когда: оба одновременно отправляют `POST /bookings` на этот слот.
Тогда: ровно один запрос получает **201** и создаёт бронь `CONFIRMED`; второй получает **409**
(«место только что заняли» — после не более чем одного внутреннего retry); в БД ровно одна
бронь `CONFIRMED` на слот, `freeSpots = 0`, `version` увеличен; двойной брони нет.

**КП-2. Автозавершение переводит статусы (D-09, ФТ-23, UC-10)**
Дано: слот `SCHEDULED` с `startAt + 180 мин` в прошлом; у слота брони: одна `CONFIRMED`,
одна `NO_SHOW`, одна `CANCELLED_BY_CLIENT`.
Когда: отрабатывает процесс автозавершения (§5.1).
Тогда: слот — `COMPLETED`; бронь `CONFIRMED` — `COMPLETED`; бронь `NO_SHOW` осталась `NO_SHOW`;
отменённая не изменилась; повторный прогон процесса ничего не меняет; по завершённой брони
теперь можно создать отзыв (D-05).

**КП-3. Напоминание не дублируется (UC-09, ФТ-16)**
Дано: бронь `CONFIRMED`, момент «за 24ч до `startAt`» наступил; процесс напоминаний уже
отправил EML-03 по этой брони.
Когда: процесс §5.2 запускается повторно (в т.ч. многократно).
Тогда: второе письмо EML-03 по этой паре (бронь, 24ч) не отправляется; при наступлении
момента «за 2ч» отправляется ровно одно EML-04.

**КП-4. Сбой SMTP не откатывает операцию (D-07, НФТ-03)**
Дано: SMTP-сервер недоступен; слот `SCHEDULED` с двумя бронями `CONFIRMED`.
Когда: ADMIN выполняет `POST /slots/{slotId}/cancel` с причиной.
Тогда: ответ — **200**; слот — `CANCELLED_BY_STUDIO` с причиной; обе брони —
`CANCELLED_BY_STUDIO`; неотправленные EML-02 не приводят ни к откату, ни к ошибке операции;
аналогично сбой отправки напоминания не влияет на бронь и на другое напоминание.

**КП-5. Блок снимается и счётчик обнуляется (D-02, ФТ-09)**
Дано: клиент с `lateCancelCount = 3` и `blockedUntil` в прошлом (7 дней истекли).
Когда: клиент выполняет `POST /bookings` на доступный слот (или запрашивает `GET /me`).
Тогда: отказа 403 нет — бронь создаётся (201); `blockedUntil` снят (null),
`lateCancelCount = 0`. До истечения `blockedUntil` тот же запрос брони давал **403** с датой
разблокировки в ответе.

**КП-6. Снапшот цены неизменен (D-06, НФТ-07)**
Дано: бронь с `equipmentChoice = RENTAL` создана при `rentalPricePerSet = X`;
`rentalPriceSnapshot = X`.
Когда: цена проката на слоте становится иной, чем X (в MVP слот не редактируется — Р-03;
проверка допустима прямым изменением данных на стенде), либо проходит любое время и любые
операции чтения.
Тогда: `rentalPriceSnapshot` брони по-прежнему равен X — во всех ответах API и в БД; ни одна
операция API не принимает это поле на вход.

**КП-7. Пересечение слотов шефа (D-10, ФТ-24)**
Дано: у шефа есть слот `SCHEDULED` на `[T, T + 180 мин)`.
Когда: ADMIN создаёт второй слот того же шефа со `startAt` внутри `(T − 180 мин, T + 180 мин)`.
Тогда: **409** с `details.conflictingSlotId` существующего слота; слот не создан. Если
существующий слот в `CANCELLED_BY_STUDIO` или `COMPLETED` — создание проходит (201).

**КП-8. Ответ не раскрывает существование аккаунта (§7, Р-01/ФТ-21)**
Дано: email A существует в системе, email B — нет.
Когда: гость выполняет `POST /auth/code/request` для A и для B.
Тогда: оба ответа — **202** и неотличимы по телу; повторный запрос раньше 60 секунд — **429**
с `Retry-After` в обоих случаях.
